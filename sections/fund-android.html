1. Lifecycles (Activity, Fragment, Process, Application & App Startup) Lifecycles

Concepts & Explanations

  • Activity & Fragment Lifecycles: Activity transitions manage UI setups and teardowns. Fragments have a separate view lifecycle; use viewLifecycleOwner for observers to prevent leaks after views are destroyed.
  • Process & Application Lifecycles: The OS manages process lifecycles dynamically, terminating background applications using out-of-memory scores (OOM ADJ) under memory pressure.
  • App Startup Library: Initializes components at application startup using a shared ContentProvider, reducing launch overhead.

Code Example

import android.os.Bundle
import android.view.View
import androidx.fragment.app.Fragment
import androidx.lifecycle.*
import androidx.startup.Initializer

// Fragment view lifecycle observer safety
class TravelFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        super.onViewCreated(view, savedInstanceState)
        
        // Always observe using viewLifecycleOwner, NOT 'this'
        viewModel.data.observe(viewLifecycleOwner) { data ->
            println("Received: $data")
        }
    }
}

// App Startup Initializer
class AnalyticsInitializer : Initializer<Unit> {
    override fun create(context: android.content.Context) {
        println("Initializing SDK at startup")
    }
    override fun dependencies(): List<Class<out Initializer<*>>> = emptyList()
}
Why is using 'this' as the lifecycle owner inside Fragment onViewCreated risky?
Answer
Fragments survive view recreation (e.g. when replaced on the backstack). If you pass this (the fragment instance), the observer holds a reference to the old view layout even after onDestroyView() executes, leaking memory. viewLifecycleOwner guarantees cleanup immediately when the view is destroyed.

2. Looper, MessageQueue, and Handler Loop Threading

Concepts & Explanations

  • Looper: Runs an infinite loop on a thread, fetching tasks from a MessageQueue. Sleeps using Linux epoll when empty.
  • MessageQueue: Priority queue holding scheduled tasks (messages and runnables).
  • Handler: Posts runnables and schedules messages to execute on a target thread's Looper context.

Code Example

import android.os.Handler
import android.os.Looper
import android.os.Message

class CustomWorkerThread : Thread() {
    lateinit var handler: Handler

    override fun run() {
        // Initialize Looper context on background thread
        Looper.prepare()
        
        handler = object : Handler(Looper.myLooper()!!) {
            override fun handleMessage(msg: Message) {
                println("Handling background message ID: ${msg.what} on ${Thread.currentThread().name}")
            }
        }
        
        // Start infinite thread message loop
        Looper.loop()
    }
}
How does Looper sleep without consuming CPU cycles?
Answer
The Looper blocks inside the native code of MessageQueue.next(). It performs a Linux epoll_wait system call, placing the thread to sleep until a write event occurs on the event file descriptor, indicating a new message has been posted.

3. ART Runtime, Garbage Collection & Memory Leaks Memory

Concepts & Explanations

  • ART Runtime: Android Runtime. Uses JIT compilation for quick starts and AOT compilation during device charging cycles to optimize performance.
  • Garbage Collection: Uses a concurrent copying collector. It copies alive objects to another heap segment while compacting, minimizing thread pauses.
  • Memory Leaks: Occur when objects (like Activities) are retained by long-lived GC root references (like static fields or active background threads).

Code Example

import android.content.Context
import java.lang.ref.WeakReference

class LeakFreeHelper(context: Context) {
    // Avoid storing context directly to prevent memory leaks
    private val contextRef: WeakReference<Context> = WeakReference(context)

    fun doWork() {
        val context = contextRef.get()
        if (context != null) {
            println("Context is still alive, performing work")
        }
    }
}
How do non-static inner classes cause memory leaks in Android?
Answer
In Java and Kotlin (unless declared as inner), non-static nested classes hold an implicit reference to the outer class. If a non-static inner class (such as an anonymous Handler runnable) executes a long-running task in the background, it retains the outer Activity reference, preventing it from being garbage collected and causing a memory leak.

4. Binder IPC, PendingIntent Security & Notifications IPC & Security

Concepts & Explanations

  • Binder IPC: Lightweight process-to-process communication. Uses memory-mapped buffers to share data, restricted to a strict 1MB transaction buffer.
  • PendingIntent Security: Grants external apps authorization to execute intents on your app's behalf. Must declare FLAG_IMMUTABLE (API 31+) to prevent execution exploits.

Code Example

import android.app.PendingIntent
import android.content.Context
import android.content.Intent

fun createSecurePendingIntent(context: Context): PendingIntent {
    val intent = Intent(context, TargetActivity::class.java).apply {
        action = "ACTION_VIEW_DETAILS"
    }
    
    // Explicitly declare mutability flag (FLAG_IMMUTABLE) for security
    return PendingIntent.getActivity(
        context,
        1001,
        intent,
        PendingIntent.FLAG_IMMUTABLE or PendingIntent.FLAG_UPDATE_CURRENT
    )
}

class TargetActivity : android.app.Activity()
Why did Android 12 make declaring PendingIntent mutability flags mandatory?
Answer
If a PendingIntent is mutable by default, a malicious application that receives the token can modify the underlying intent (e.g. changing the target class or intent data) to access private components or execute operations with your application's privileges. FLAG_IMMUTABLE blocks modification.

5. Background Processing (WorkManager, Services vs WorkManager, Foreground Services) Background

Concepts & Explanations

  • WorkManager Internals: The standard for persistent, deferrable background work. Saves task state in a local SQLite database and schedules tasks using JobScheduler when constraints are met.
  • Services: Framework components that execute on the main thread by default. Kept alive relative to the host process priority.
  • Foreground Services: Tasks that require continuous execution. They must display a persistent notification and declare service types on Android 14+.

Code Example

import android.content.Context
import androidx.work.*

class DatabaseSyncWorker(context: Context, params: WorkerParameters) : CoroutineWorker(context, params) {
    override suspend fun doWork(): Result {
        println("Syncing database in background...")
        return Result.success()
    }
}

fun scheduleDatabaseSync(context: Context) {
    val constraints = Constraints.Builder()
        .setRequiredNetworkType(NetworkType.UNMETERED)
        .setRequiresCharging(true)
        .build()

    val request = OneTimeWorkRequestBuilder<DatabaseSyncWorker>()
        .setConstraints(constraints)
        .build()

    WorkManager.getInstance(context).enqueue(request)
}
When should you use WorkManager instead of a Foreground Service?
Answer
Use WorkManager for persistent tasks that require execution guarantees (like database syncs or analytics uploads) even if the app or device restarts. Use a Foreground Service for immediate, user-noticeable, non-deferrable tasks (like music playback or active route navigation).
