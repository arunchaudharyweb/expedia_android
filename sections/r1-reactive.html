1. Flow vs LiveData Comparison Matrix Comparison

Quick Cheat Sheet

Feature LiveData StateFlow / Flow
Lifecycle Awareness Yes (bound to Android Lifecycle) No (requires lifecycle-aware helpers like `repeatOnLifecycle`)
Backpressure No (values overwritten immediately) Yes (supported via suspension, buffer, and conflation)
Threading Main thread bound for observers Flexible (configured using dispatchers and `flowOn`)
Operators Limited (Transformations map/switchMap) Extensive (standard functional operators)
Compare LiveData and StateFlow in terms of lifecycle awareness.
Answer
LiveData is bound to Android's UI lifecycle and automatically stops delivering updates when the view is inactive. StateFlow is a pure Kotlin API and does not know about the Android lifecycle. To collect values safely, use the repeatOnLifecycle helper in the UI layer, which pauses and resumes the collection coroutine automatically.

Common Mistake
Using LiveData in the data or domain layers of an application, coupling business logic to Android platform components.

2. Flow vs RxJava Threading Models Threading

Quick Cheat Sheet

  • RxJava subscribeOn vs observeOn: subscribeOn sets the thread context for the entire chain. observeOn overrides threading for all downstream operators.
  • Kotlin Flow flowOn: The flowOn operator changes the execution dispatcher for **upstream** operators only. The collector runs on the dispatcher of the calling scope.
How does flowOn differ from RxJava's subscribeOn/observeOn?
Answer
In RxJava, subscribeOn configures execution for the entire chain, while observeOn changes the thread for downstream operators. In Kotlin Flow, the flowOn operator only changes the dispatcher for *upstream* operators. The collection always runs on the context of the calling scope.
Senior-Level Follow-up
What is the performance overhead of using multiple flowOn calls in a single pipeline? (It creates intermediate channel buffers, introducing context switching overhead).
Common Mistake
Placing flowOn at the end of a Flow chain and expecting it to configure the collection thread.

3. Backpressure Mitigation (Buffer vs Conflate) Execution

Quick Cheat Sheet

  • Suspension: By default, Flow suspends the emitter coroutine if the collector is busy, naturally matching emission rate to consumption.
  • buffer(): Queues emitted values in a buffer, allowing the emitter to continue executing while the collector catches up.
  • conflate(): Skips intermediate values, delivering only the latest value to the collector when it is ready.
Explain the difference between buffer() and conflate() in Flow.
Answer
buffer() caches emitted values in a queue, allowing the emitter to run ahead of a slow collector. conflate() drops intermediate values, ensuring the collector only receives the latest value when it is ready. Use conflate() when intermediate state updates do not matter (e.g. high-frequency scroll offsets).
Senior-Level Follow-up
What happens under the BufferOverflow.DROP_OLDEST policy?
Common Mistake
Using a large buffer size for real-time events, which can cause high memory usage and delay processing of the latest events.

4. State vs Event Containers (SingleLiveEvent problem) Architecture

Quick Cheat Sheet

  • StateFlow (State): Retains and replays the current state value. Not suited for transient actions (like showing a Toast or navigation) because configuration changes will replay the state, triggering duplicate actions.
  • SharedFlow (Event): Does not retain state. Broadcasts events to active collectors, making it ideal for transient actions.
  • State-Driven Event Pattern: Model navigation or messages as state property (e.g. navigationDestination: Screen?). Clear the state (set to null) immediately after the action is executed.
Why should you avoid using StateFlow for one-time UI events?
Answer
StateFlow retains its state value. When a new collector subscribes (e.g. during screen rotation), it replays the last value. If that value represents a one-time event (like navigation or a dialog), it will be re-triggered, causing duplicate actions. Use SharedFlow or state-driven events instead.
Senior-Level Follow-up
Why is the EventWrapper pattern considered an anti-pattern? (It introduces mutable state inside immutable state models).
Common Mistake
Setting replay = 1 on a SharedFlow that handles navigation events, causing duplicate navigation triggers on rotation.
