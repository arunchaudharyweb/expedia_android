1. Flow vs LiveData Comparison Comparison

Compare the design, lifecycle behavior, and features of LiveData and Kotlin Flow:

Feature LiveData Kotlin Flow
Lifecycle Awareness Yes (bound to Android Lifecycle) No (requires lifecycle-aware collection helpers)
Backpressure Support No (value is overwritten immediately) Yes (supported via buffer and conflate operators)
Threading Model Main thread bound for observers Flexible (configured via dispatchers using flowOn)
Operators Limited (Transformations.map/switchMap) Extensive (standard functional operators)
Interview Question
Compare LiveData and StateFlow in terms of thread execution models and backpressure handling.
Answer
LiveData is bound to the main thread; updates posted from background threads are dispatched to the main thread. It does not support backpressure, so rapid updates overwrite previous values. StateFlow is built on coroutines, allowing you to configure the execution thread using dispatchers. It handles backpressure by suspending the emitter or using buffering/conflation strategies.
Senior Follow-Up
How does repeatOnLifecycle prevent background resource waste when collecting Flows in Android UI components?
Common Candidate Mistake
Collecting Flows in an Activity using lifecycleScope.launch instead of repeatOnLifecycle, leaving the stream active in the background when the app is stopped.

2. Flow vs RxJava Comparison & Migration Comparison

Compare Kotlin Flow and RxJava:

Metric RxJava Kotlin Flow
Cold Stream Type Observable, Flowable Flow
Threading operator subscribeOn, observeOn flowOn (applies only upstream)
Backpressure Flowable (buffer, drop, latest) Built into suspending functions natively
API footprint Large (hundreds of operator methods) Small (uses standard extension functions)
Interview Question
What is the threading model difference between RxJava's subscribeOn/observeOn and Flow's flowOn?
Answer
In RxJava, subscribeOn configures execution for the entire chain, while observeOn overrides threading for downstream operators. In Kotlin Flow, the flowOn operator changes the dispatcher for *upstream* operators only. The collector runs on the dispatcher of the calling scope.
Senior Follow-Up
How do you bridge RxJava Observables to Kotlin Flows in a legacy codebase?
Common Candidate Mistake
Assuming that placing flowOn at the end of a Flow chain configures the collection thread (it only configures upstream operators).

3. When to Still Use LiveData Legacy

While Flow is preferred for new code, LiveData is still useful in specific scenarios:

  • Legacy XML Data Binding: XML layout bindings support LiveData natively, updating views automatically.
  • Simple UI State: For simple screens without complex transformations or concurrency, LiveData provides a simple, main-thread-safe API.
  • Java Interoperability: If parts of your application are still written in Java, LiveData is easier to observe than Kotlin coroutine flows.
Interview Question
In which scenarios is LiveData still acceptable in modern Android architecture?
Answer
LiveData is still useful when integration with legacy XML Data Binding is required, or when interoperating with Java code. For simple UI states that require main-thread safety without complex transformations, LiveData remains a viable option.
Senior Follow-Up
How does liveData {} coroutine builder bridge asynchronous coroutine code with LiveData consumers?
Common Candidate Mistake
Using LiveData in the data or domain layers of an application, which couples business logic to the Android platform libraries.

4. Cold vs Hot Streams Streams

Understand the differences between stream categories:

  • Cold Streams: The producer block starts executing only when a collector subscribes (e.g. flow { emit(1) }). Each subscriber triggers an independent execution of the stream.
  • Hot Streams: The stream executes independent of subscribers (e.g. StateFlow, SharedFlow). Values are broadcast to all active subscribers, and new subscribers receive either the latest value (for StateFlow) or the buffered replay window.
Interview Question
Define hot and cold streams and provide an example of each in Android.
Answer
Cold streams (such as standard flow { ... } or Room query streams) run their producer blocks on-demand for each subscriber. Hot streams (such as StateFlow or SharedFlow) broadcast values to all active collectors from an active execution source, remaining independent of subscriber lifecycles.
Senior Follow-Up
How does a hot stream handle resource cleanup when there are no active collectors?
Common Candidate Mistake
Assuming that database queries wrapped in a standard cold Flow run once and share results among multiple collectors.

5. Backpressure Mitigation (Buffer, Conflate) Execution

Backpressure occurs when a producer emits values faster than the consumer can process them.

Mitigation:

  • Suspension: By default, a Flow suspends the producer coroutine when the collector is busy, matching the emission rate to the consumption rate.
  • Buffer: Queues emitted values in a buffer, allowing the producer to continue executing while the collector catches up.
  • Conflate: Skips intermediate values, delivering only the latest value to the collector when it is ready.
val searchFlow = rawQueryFlow
    .conflate() // Skips intermediate queries if processing search results is slow
    .collect { query ->
        updateSearchUI(query)
    }
Interview Question
How does Flow handle backpressure, and what is the difference between buffer() and conflate()?
Answer
Flow uses coroutine suspension by default to pause emission until the collector is ready. buffer() queues values in a buffer so the producer can emit without suspending immediately. conflate() drops intermediate values, ensuring the collector only receives the latest value when it is ready.
Senior Follow-Up
What happens when a buffer overflows under the BufferOverflow.DROP_OLDEST policy?
Common Candidate Mistake
Using a large buffer size for real-time events, which can cause high memory usage and delay processing of the latest events.

6. State vs Event Containers: MutableStateFlow vs MutableSharedFlow Architecture

Architectures must separate persistent state from transient events:

  • MutableStateFlow (State Container): Tracks the current state of the UI (e.g. loading, success, or error states). It retains the latest value and replays it to new subscribers, surviving configuration changes.
  • MutableSharedFlow (Event Channel): Broadcasts transient actions (e.g. navigation triggers, snackbar messages, or error dialogs). It does not retain state, ensuring events are processed only once by active listeners.
Interview Question
Why should you avoid using MutableStateFlow to dispatch one-time UI events?
Answer
MutableStateFlow retains its state. When a new collector subscribes (e.g., during screen rotation), it replays the last emitted state. If that state represents a one-time event (like navigation or a dialog), it will be re-triggered, causing duplicate actions. Use MutableSharedFlow or state-driven events instead.
Senior Follow-Up
How do you configure MutableSharedFlow to emulate a single-live-event channel?
Common Candidate Mistake
Setting replay = 1 on a SharedFlow that handles navigation events, which causes duplicate navigation triggers on rotation.

7. Why Channel is not Recommended for UI Events Architecture

While Channel supports point-to-point communication, it is no longer recommended for UI events:

  • Single Consumer Limit: Channels are designed for single-consumer queues. If multiple collectors subscribe, events are distributed among them randomly rather than broadcast to all.
  • Buffer Loss: If no collectors are active (e.g., during configuration changes), sent events are buffered. When the view reconnects, these events can be lost or delivered in an incorrect state.
  • Lifecycle Management: Channels do not integrate with lifecycle-aware flow collection, making it difficult to pause and resume emissions safely.
Interview Question
Explain why Channel is discouraged for managing UI events in modern Android architectures.
Answer
Channels are designed for single-consumer queues. If a screen has multiple observers, events are split between them. Additionally, managing buffer state during lifecycle transitions is difficult, which can lead to lost events or memory leaks. MutableSharedFlow is preferred for broadcasting events.
Senior Follow-Up
What is the difference between a rendezvous channel and a conflated channel?
Common Candidate Mistake
Using a Channel to dispatch events to both an Activity and a Fragment, resulting in events being delivered to only one of them.

8. Once-Only Event Problem: EventWrapper vs CurrentScreen Architecture

Handling one-time events (like navigation or alerts) requires careful state design:

  • EventWrapper Anti-Pattern: Wraps events in an object that tracks whether the content has been handled (e.g. getContentIfNotHandled()). This approach is discouraged because it introduces mutable state inside UI event models and couples state modeling to consumer consumption flags.
  • CurrentScreen (State-driven) Approach: Models navigation explicitly as a state property (e.g., a nullable destination field inside the screen state). When the navigation is executed, the UI sends an intent back to the ViewModel to reset the destination property to null, ensuring the event is processed only once.
// State-driven approach
data class SearchUiState(
    val hotels: List = emptyList(),
    val navigationDestination: Hotel? = null // Destination is part of the state
)

// UI Collection
viewModel.uiState.collect { state ->
    state.navigationDestination?.let { hotel ->
        navigator.navigateToDetails(hotel)
        viewModel.onNavigationHandled() // Reset destination to null in state
    }
}
Interview Question
Why is the EventWrapper pattern considered an anti-pattern, and how does the State-driven approach resolve it?
Answer
The EventWrapper pattern introduces mutable state inside immutable state models. The state-driven approach treats the navigation destination as a state property. Once the navigation is executed, the UI triggers an event to clear the destination property in the ViewModel, keeping the state immutable and clean.
Senior Follow-Up
How does this state-driven navigation handle complex backstack recovery scenarios?
Common Candidate Mistake
Failing to clear the navigation state property in the ViewModel after executing the navigation, triggering duplicate navigation events on configuration changes.
