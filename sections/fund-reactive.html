Reactive Programming Paradigms

⏱ 10 min read ✓ Part 0 Fundamentals

Deep dive into reactive stream patterns, cold and hot emissions, stream processing operators, backpressure resolution strategies, and platform lifecycle boundaries.

1. Flow (Cold Streams) vs StateFlow/SharedFlow (Hot Streams) Streams

Concepts & Explanations

  • Cold Streams (Flow): Independent data streams. The producer block runs on-demand for each subscriber when collection begins. If there are no active collectors, the stream does not execute or emit.
  • Hot Streams (StateFlow & SharedFlow): Active broadcast streams. They broadcast emissions immediately to all active collectors, regardless of whether collection has started, sharing a single source.

Code Example

import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*

fun runColdAndHotStreams() = runBlocking {
    // 1. Cold Flow: executes code block for each collector
    val coldFlow = flow {
        println("Cold Flow execution started")
        emit("Data A")
    }

    // 2. Hot StateFlow: executes immediately and shares state
    val hotState = MutableStateFlow("Initial")
    
    println("Pre-collection status")
    
    // Collecting cold flow
    coldFlow.collect { println("Collector 1: $it") }
    coldFlow.collect { println("Collector 2: $it") } // triggers execution again

    // Collecting hot flow
    launch {
        hotState.collect { println("Hot Collector: $it") }
    }
    
    hotState.value = "Updated" // updates state, broadcasting to collectors
}
How can you convert a cold Flow into a hot Flow?
Answer
Use the shareIn or stateIn operators. These operators launch a coroutine in a specified scope to collect the cold flow upstream and broadcast its emissions hot to downstream collectors. For example, stateIn(scope, SharingStarted.WhileSubscribed(5000), initialValue).

2. StateFlow & SharedFlow State & Event

Concepts & Explanations

  • StateFlow: A state holder. It requires an initial value, exposes its state via the value property, filters duplicate values, and replays the latest value to new collectors.
  • SharedFlow: An event broadcaster. It has no initial value, does not filter duplicate values, and does not replay events to new collectors (unless configured with a replay cache > 0).

Code Example

import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*

class ScreenViewModel {
    // StateFlow exposes state to UI
    private val _uiState = MutableStateFlow("Loading")
    val uiState: StateFlow = _uiState.asStateFlow()

    // SharedFlow broadcasts one-time navigation events
    private val _eventFlow = MutableSharedFlow()
    val eventFlow: SharedFlow = _eventFlow.asSharedFlow()

    fun loadData() {
        _uiState.value = "ContentLoaded" // Updates StateFlow
    }

    suspend fun triggerNavigation() {
        _eventFlow.emit("NavigateToHome") // Broadcasts SharedFlow event
    }
}
Why should you avoid exposing MutableStateFlow directly to views?
Answer
Exposing MutableStateFlow violates the encapsulation principle. It allows views to directly modify the state, bypassing the ViewModel's business logic. Always expose it as a read-only StateFlow using asStateFlow().

3. Flow Operators (map, filter, flatMapLatest, flowOn) Operators

Concepts & Explanations

  • map & filter: Perform transformations and conditions on stream elements.
  • flatMapLatest: When a new item is emitted, it cancels all previous active flows, making it ideal for search queries.
  • flowOn: Configures the dispatcher context of the **upstream** stream source, leaving downstream collector Dispatchers unaffected.

Code Example

import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*

@OptIn(ExperimentalCoroutinesApi::class)
fun runOperators() = runBlocking {
    val searchFlow = flowOf("Hot", "Hote", "Hotel")
    
    searchFlow
        .filter { it.length > 3 } // Upstream operation: runs on IO
        .map { it.uppercase() }  // Upstream operation: runs on IO
        .flowOn(Dispatchers.IO) // Configures upstream context to IO
        .flatMapLatest { query ->
            // Cancels the previous search if a new query is received
            flow {
                emit("Results for $query")
            }
        }
        .collect {
            println("Collected on main thread: $it")
        }
}
What is the difference between flowOn and withContext in a flow?
Answer
In a flow builder, calling withContext directly to change dispatchers violates the flow preservation rule and throws a runtime exception. You must use the flowOn(Dispatcher) operator, which manages context switching safely using channel-based buffering under the hood.

4. Backpressure Concepts (Buffer vs Conflate vs Suspension) Flow Control

Concepts & Explanations

  • Suspension: Flow's natural backpressure mechanism. Emitters suspend until collectors finish.
  • buffer(): Buffers emitted items in a channel, allowing the producer to emit ahead of slow collectors.
  • conflate(): Discards intermediate values, sending only the latest state to a slow collector.

Code Example

import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*

fun testBackpressure() = runBlocking {
    val fastProducer = flow {
        for (i in 1..3) {
            println("Emitting: $i")
            emit(i)
            delay(50) // Fast emission
        }
    }

    // 1. Conflate Demo: discards intermediate emissions
    println("--- Conflate ---")
    fastProducer.conflate().collect { value ->
        delay(200) // Slow collector
        println("Collected: $value")
    }

    // 2. Buffer Demo: buffers emissions
    println("--- Buffer ---")
    fastProducer.buffer(2).collect { value ->
        delay(200) // Slow collector
        println("Collected: $value")
    }
}
How does Flow resolve backpressure by default?
Answer
Flow resolves backpressure using coroutine suspension. If a collector is busy, the emitter suspends inside the emit() call, pausing emission until the collector finishes processing the current item and resumes.

5. Flow vs LiveData Comparison

Concepts & Explanations

  • LiveData: An Android platform component. It runs on the main UI thread and stops delivering updates when the observer's lifecycle is inactive, avoiding memory leaks.
  • StateFlow: A pure Kotlin API. It is lifecycle-agnostic and can run on any thread. To collect values safely in the UI, use repeatOnLifecycle to automatically pause and resume collection.

Code Example

// Android UI Layer safely collecting StateFlow vs LiveData
import androidx.lifecycle.*
import kotlinx.coroutines.launch

class MyActivity : androidx.activity.ComponentActivity() {
    private val viewModel = ScreenViewModel()

    fun observeLiveData(liveData: LiveData) {
        // LiveData: handles lifecycles automatically
        liveData.observe(this) { data ->
            println("LiveData received: $data")
        }
    }

    fun observeStateFlow() {
        // StateFlow: needs repeatOnLifecycle helper to prevent resource leaks
        lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                viewModel.uiState.collect { data ->
                    println("StateFlow received: $data")
                }
            }
        }
    }
}
Why is StateFlow preferred over LiveData in Clean Architecture?
Answer
LiveData is an Android platform dependency, which couples the domain and data layers to Android framework classes, breaking clean architecture. StateFlow is a pure Kotlin API, making it portable and testable in pure JVM modules without mocking Android dependencies.
