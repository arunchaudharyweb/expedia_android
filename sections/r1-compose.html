Section 4: Jetpack Compose

⏱ 25 min read ✓ Part 1 Core

Declarative UI runtime architecture, state tracking, composition lifecycles, recomposition scopes, snapshot system internals, stability rules, and performance rendering optimization.

1. Composable Functions Runtime

Internal Working

Composable functions are annotated with @Composable. The Compose compiler plugin transforms these functions during compilation, adding an implicit parameter representing the Composer instance and inserting start/end group calls (like composer.startRestartGroup). These groups map UI nodes to a persistent data structure called the Slot Table, representing the composition tree.

Recomposition Triggers

A composable is executed during the initial Composition. It is re-executed (recomposed) when the state it reads changes, or when the arguments passed to it by its parent change.

Performance Impact

Composable functions do not return UI elements; they emit layout descriptions to the Composer. The runtime compares descriptions to minimize actual tree modifications (Layout & Draw phases), ensuring high-speed rendering.

Interview Questions

What does the Compose compiler add to @Composable functions during compilation?
Answer
The compiler adds an implicit $composer parameter to the function signature and injects calls to $composer.startRestartGroup(...) at the beginning and $composer.endRestartGroup() at the end of the function body. This allows the runtime to track the function's identity, input arguments, and state reads in the Slot Table.
Senior-Level Follow-up
Explain how the Composer uses the key parameter in startRestartGroup to resolve recomposition targets.
Common Mistake
Thinking that Composable functions return views or construct static class hierarchies similar to standard Java objects.

2. State & MutableState State

Internal Working

State<T> is an observable value container integrated with the Compose runtime. Under the hood, it uses the Snapshot system. When a Composable function reads a State's value during execution, the Compose runtime records this read in the active RecomposeScope. When the State is mutated, the runtime marks that scope as dirty, scheduling it for recomposition.

Recomposition Triggers

Writing a new value to a MutableState container triggers recomposition of all scopes that read that State's value.

Performance Impact

Reads are optimized through a lock-free snapshot system, minimizing thread overhead. However, mutating state inside high-frequency loops triggers continuous recompositions (recomposition thrashing).

Interview Questions

How does the Compose runtime track state reads?
Answer
The Compose runtime utilizes thread-local Snapshot contexts. During the Composition phase, the current RecomposeScope is registered as a listener on the thread. When a State.value getter is invoked, it triggers a read callback that links the State object instance to the active RecomposeScope, saving this dependency in the Slot Table.
Senior-Level Follow-up
Explain the internal lock-free concurrency model used by the Compose Snapshot system when updating state across threads.
Common Mistake
Mutating a standard Kotlin variable and expecting the Composable to update (it must be wrapped in `MutableState` to be tracked).

3. remember API State

Internal Working

The remember block stores a value in the Slot Table at the current call site. During initial Composition, the block is executed, and its result is written to the table. On subsequent recompositions, the runtime returns the cached value directly instead of re-executing the block, unless the key arguments passed to remember change.

Recomposition Triggers

remember itself does not trigger recomposition. It caches values across recompositions. If it caches a MutableState, updates to that state's value will trigger recomposition.

Performance Impact

Prevents expensive recalculations (e.g., parsing dates or formatting text) on every frame render. Forgetting to remember heavy calculations degrades performance.

Interview Questions

What is the difference between remembering a value with keys vs without keys?
Answer
remember { calculate() } executes the block once during initial composition and returns the cached value forever (or until the composable leaves the tree). remember(key1) { calculate(key1) } compares the current key1 instance with the value stored in the slot table. If they are not equal, the block is re-evaluated, caching and returning the new result.
Senior-Level Follow-up
How does the Slot Table store and align keys and values of remember calls during structural tree changes?
Common Mistake
Assuming that `remember` preserves state across Activity destruction or process recreation (it only survives recompositions within the active window).

4. rememberSaveable API State

Internal Working

rememberSaveable behaves like remember, but also serializes the stored value into Android's Bundle system. It automatically saves values during configuration changes (like rotation) and process recreation, restoring them during re-composition.

Recomposition Triggers

Restores values during recreation, which initializes the composable with its previous state, triggering initial rendering with historical values.

Performance Impact

Slight overhead due to serialization/deserialization. Storing massive data structures in rememberSaveable can trigger TransactionTooLargeException.

Interview Questions

How does rememberSaveable serialize custom data structures?
Answer
It supports types that can be put inside a Bundle by default. For custom objects, you must provide a custom Saver implementation defining how to convert the object to a saveable format (like list, map, or parcelable) and how to reconstruct it.
Senior-Level Follow-up
How does the SaveableStateRegistry coordinate with the Android system lifecycle to persist state?
Common Mistake
Using rememberSaveable for non-parcelable types without declaring a custom Saver, resulting in runtime crashes during state saving.

5. State Hoisting Design

Internal Working

State Hoisting is a pattern of moving state to a composable's caller to make the composable stateless. It replaces internal state updates with parameters: a value parameter (e.g. value: T) and an event callback (e.g. onValueChange: (T) -> Unit).

Recomposition Triggers

Recomposition is triggered when the hoisted state parameter updates at the parent level, propagating down to the stateless child.

Performance Impact

Improves testability, reusability, and stability. However, hoisting state too high up the tree can cause unrelated sibling composables to recompose unless they are configured as skippable.

Interview Questions

What are the architectural benefits of State Hoisting in Jetpack Compose?
Answer
State hoisting decoupled UI rendering from state ownership. It creates a single source of truth, makes composables stateless (which makes them easier to preview and test), and guarantees a Unidirectional Data Flow (UDF) where state flows down and events flow up.
Senior-Level Follow-up
Explain the trade-offs between hoisting state to a parent Composable vs hoisting it to a ViewModel.
Common Mistake
Creating duplicate local states in child composables and trying to sync them with hoisted states manually.

6. Recomposition Phase Runtime

Internal Working

Recomposition is the phase where the Compose runtime re-executes Composable functions that read updated state. The runtime identifies changes in the Slot Table, performs structural updates, and generates a new description tree. It then transitions to the Layout and Draw phases to update the screen.

Recomposition Triggers

Triggered by mutations in Compose MutableState read inside the scope, or changes to non-equal parameters passed to child composables.

Performance Impact

Recomposition is designed to be incremental. The compiler targets the smallest possible scope (usually the nearest enclosing Composable block) to avoid re-running the entire screen UI tree.

Interview Questions

What is a RecomposeScope and how does it optimize UI updates?
Answer
A RecomposeScope is an executable block representing a section of the composition tree that can be run independently. The runtime maps state read dependencies to these scopes. When a state change occurs, the runtime invalidates only the specific scopes reading that state, re-running only those functions instead of the entire composable hierarchy.
Senior-Level Follow-up
Explain the internal workings of the Recomposer class and its coordination with the Choreographer VSYNC signals.
Common Mistake
Performing side-effects or heavy computations directly in the body of a Composable function, causing them to execute on every recomposition.

7. Side Effects Lifecycle

Internal Working

A side effect is a change to the state of the app that happens outside the control of a Composable function. Composable functions must be side-effect free. To interact with the outside world safely, Compose provides side-effect APIs that bind operations to the Composable's lifecycle.

Recomposition Triggers

Side-effect APIs coordinate with recompositions to ensure actions (like starting an animation or listening to a flow) occur only when appropriate keys change, avoiding redundant triggers.

Performance Impact

Using correct side-effect builders prevents resource leaks (like unclosed sockets or active listeners) and keeps the UI responsive.

Interview Questions

Why is performing direct side effects in a Composable body dangerous?
Answer
Composable functions can run in any order, can be executed concurrently across threads, and can be discarded before completing if a new recomposition is scheduled. Executing side effects directly in the body can lead to duplicate executions, thread safety issues, and memory leaks. Side-effect blocks ensure operations execute in a controlled manner.
Senior-Level Follow-up
How does the SideEffect API differ from LaunchedEffect in terms of execution phase?
Common Mistake
Triggering analytical network requests directly inside the composable code block instead of wrapping them in a `LaunchedEffect`.

8. LaunchedEffect API Lifecycle

Internal Working

LaunchedEffect is a Composable function that runs a suspending block in a coroutine scope managed by the Compose runtime. It starts the coroutine when it enters the Composition, and cancels it when it leaves the Composition. If any of the key parameters change, the current coroutine is cancelled and a new one is launched.

Recomposition Triggers

Recomposition does not execute the suspending block again unless one of the keys passed to LaunchedEffect has changed.

Performance Impact

Provides a safe way to perform asynchronous tasks (e.g. fetching network data, triggering animations) that are tied to the UI lifecycle.

Interview Questions

What happens to the coroutine launched by LaunchedEffect(key) when the key changes?
Answer
The Compose runtime immediately cancels the active coroutine job associated with the effect, executing its cancellation handlers. It then creates a new coroutine using the new key, launching the suspending block under the scope.
Senior-Level Follow-up
How does `rememberCoroutineScope()` compare to `LaunchedEffect` for launching coroutines in response to user clicks?
Common Mistake
Passing `Unit` as a key for an effect that needs to respond to dynamic parameter changes (the effect will only run once and ignore updates).

9. DisposableEffect API Lifecycle

Internal Working

DisposableEffect is a side-effect API designed for operations that require cleanup when the Composable leaves the Composition or when keys change. The block executes when entering composition, and it must end with an onDispose block containing the cleanup logic.

Recomposition Triggers

Executes the cleanup logic and re-runs the setup logic when the passed keys change, preventing resource leaks.

Performance Impact

Crucial for managing listeners, broadcast receivers, and observers. Forgetting to call `onDispose` leaks the registered resources.

Interview Questions

Provide an example of when DisposableEffect is mandatory over LaunchedEffect.
Answer
DisposableEffect is mandatory when registering callbacks or observers that need explicit unregistration (e.g., listening to location updates or system broadcast receivers). LaunchedEffect is for suspending coroutine actions, whereas DisposableEffect is for synchronous setup/cleanup. 
DisposableEffect(manager) {
    val listener = LocationListener { ... }
    manager.register(listener)
    onDispose {
        manager.unregister(listener)
    }
}
Senior-Level Follow-up
How does the compiler enforce the presence of onDispose inside a DisposableEffect block?
Common Mistake
Leaving the onDispose block empty when wrapping a listener, leaking the listener object reference.

10. produceState API State

Internal Working

produceState converts non-Compose state sources (like cold Flows, RxJava streams, or callback listener streams) into Compose State. It launches a coroutine block that can emit values using value = newValue, automatically cleaning up resources when the producer leaves the composition.

Recomposition Triggers

Mutations to the underlying value property inside produceState trigger recompositions for all views reading that state.

Performance Impact

Simplifies state bridge management, ensuring that flow collections and callbacks are cancelled when the UI component is no longer active.

Interview Questions

How does produceState simplify Flow observation compared to standard remember and LaunchedEffect?
Answer
produceState wraps the remember { mutableStateOf(initial) } and LaunchedEffect pattern into a single cleaner API. It handles the scope lifecycle, runs the collection coroutine, updates the state value, and manages cancellation automatically under the hood.
Senior-Level Follow-up
Decompile a produceState implementation and explain the returned State wrapper.
Common Mistake
Forgetting to clean up listener registrations within the `awaitDispose` block inside `produceState` adapters.

11. derivedStateOf API State

Internal Working

derivedStateOf creates a new State based on calculations derived from other State objects. It buffers updates, recalculating the derived value only when the dependencies change, and emits updates only when the calculated result changes.

Recomposition Triggers

Triggers recomposition only when the *result* of the calculation changes, filtering out intermediate state changes (conflating notifications).

Performance Impact

Crucial for high-frequency state updates (e.g. scroll offsets or list index tracking). Without it, reading raw scroll positions directly triggers recomposition on every single pixel scrolled.

Interview Questions

When should you use derivedStateOf vs standard remember(key)?
Answer
Use remember(key) when the input key changes infrequently and you want to re-execute a calculation on key change. Use derivedStateOf when the input states change frequently (like scroll positions), but you only want to trigger recomposition when the *result* of the calculation changes (e.g. `scrollOffset > 100`).
Senior-Level Follow-up
Describe the dependency tracking mechanism of derivedStateOf inside the Compose Snapshot system.
Common Mistake
Wrapping a simple state mapping (e.g. transforming `x` to `x + 1`) in `derivedStateOf`, which introduces unnecessary allocation overhead (simple computations should use `remember(x)` instead).

12. Snapshot System Runtime

Internal Working

The Snapshot system is the database engine under Compose state. It tracks values at a specific point in time, allowing isolated reads and writes across threads (Multi-Version Concurrency Control). Changes made in a snapshot are not visible to other threads until the snapshot is committed.

Recomposition Triggers

Committing a write transaction in the global snapshot notifies the recomposer, which identifies matching scopes and schedules recomposition.

Performance Impact

Enables parallel composition: Compose can calculate UI trees in parallel on background thread snapshots, merging them to the main thread snapshot on VSYNC.

Interview Questions

What is a Snapshot in Compose, and how does it support thread-safe operations?
Answer
A Snapshot represents the state of all Compose State variables at a given time. It uses a copy-on-write model. When a thread takes a snapshot, it reads a consistent state version. Writes occur in a localized frame, and are committed atomically. This isolates the main thread rendering from concurrent background calculations.
Senior-Level Follow-up
What happens if a conflict occurs when two concurrent snapshots attempt to modify the same state value?
Common Mistake
Thinking that Compose state variables are simple volatile variables with standard JVM synchronized locks.

13. Navigation Compose Architecture

Internal Working

Navigation Compose binds navigation graphs to composables. The NavHost reads a NavController backstack. When a navigation action occurs, the controller updates the backstack, triggering the NavHost to recompose and swap the active composables on screen.

Recomposition Triggers

Backstack state shifts trigger recomposition, rendering the new active screen destination and destroying old screen composition blocks.

Performance Impact

Avoids fragment transaction overheads. However, passing complex objects as parameters using JSON strings in route arguments introduces serialization delays (typesafe arguments or IDs should be used instead).

Interview Questions

What is the recommended way to pass arguments using Jetpack Navigation Compose?
Answer
You should pass simple arguments (like IDs or strings) using type-safe routes (or URL query syntax in older versions). Avoid passing complex serialized JSON objects over the route. Instead, pass the ID and let the destination screen's ViewModel fetch the full object data from the repository layer.
Senior-Level Follow-up
How does Navigation Compose handle ViewModel lifecycles for destinations inside nested graphs?
Common Mistake
Passing full domain object models in navigation route paths, causing route parsing failures and memory footprint inflation.

14. Performance Optimization Performance

Internal Working

Compose optimization targets three phases: Composition, Layout, and Draw. Deferring reads is the key optimization technique. Reading state during layout or draw phases instead of the composition phase avoids re-running Composable functions altogether, directly performing only measure/draw passes.

Recomposition Triggers

Minimize recompositions by using lambda versions of modifiers (e.g. Modifier.offset { IntOffset(0, state.value) }) to defer state reads to the layout/draw phase.

Performance Impact

Using lambda-based modifiers reduces CPU load, preventing frame drops (jank) during animations or high-speed scrolls.

Interview Questions

Why does using Modifier.offset(x, y) cause more jank than Modifier.offset { IntOffset(x, y) } during animations?
Answer
Modifier.offset(x, y) takes raw values. When these values change, the state read occurs during the Composition phase, triggering full recomposition of the Composable and its children. The lambda version Modifier.offset { IntOffset(x, y) } wraps the state read inside a lambda. The read is deferred to the Layout phase, bypasses Composition entirely, and only re-measures/positions the layout node, optimizing performance.
Senior-Level Follow-up
How does using ReusableGraphicsLayerScope inside Compose optimize GPU hardware layers?
Common Mistake
Evaluating list aggregations (like `.filter` or `.map`) inside the layout body without wrapping them in a `remember` block.

15. Compose Stability (Stable and Immutable) Performance

Internal Working

The Compose compiler evaluates every class to determine its stability:

  • Stable: Class properties are mutable, but the Compose runtime will be notified when mutations occur (e.g., standard Compose state).
  • Immutable: Class properties cannot change (e.g., val primitives).
  • Unstable: Class contains mutable fields that Compose cannot track (e.g. standard JVM collections like List, which can be cast to MutableList).

Recomposition Triggers

If a composable's parameters are all Stable/Immutable, Compose can skip recomposing it if the parameters compare equal. If a parameter is Unstable, Compose *always* recomposes the function, even if the values are identical.

Performance Impact

Unstable models in lists propagate recomposition to all items in the list on any item mutation, degrading scroll performance.

Interview Questions

Why is a standard java.util.List considered Unstable by the Compose compiler, and how do you fix it?
Answer
The compiler cannot guarantee that a List interface will not be backed by a mutable list implementation (like ArrayList). Since it cannot track mutations of standard lists, it marks them as unstable. You can fix this by: 1. Wrapping the model class with the `@Stable` or `@Immutable` annotation. 2. Using Kotlinx Immutable Collections (like PersistentList). 3. Annotating parameter models directly to force compile-time stability.
Senior-Level Follow-up
Explain how the @Immutable annotation overrides compiler checks, and what bug occurs if you mutate an annotated class.
Common Mistake
Assuming all data classes are automatically stable, even if they contain mutable var variables or generic lists.

16. Skippability Performance

Internal Working

Skippability is a compiler optimization. If a Composable's parameters are stable and compare equal to their previous values, the runtime skips executing the function body, returning the cached Slot Table result directly.

Recomposition Triggers

A skippable function is only recomposed if its parameters change (according to structural equality `equals()`) or if it reads a modified internal state.

Performance Impact

Maximizes UI performance. Ensuring that custom leaf nodes and list rows are skippable minimizes CPU work, resulting in smooth 60fps/120fps animations.

Interview Questions

How do you verify if a Composable function is skippable at build time?
Answer
You can enable Compose Compiler Metrics by adding compiler arguments in your `build.gradle` (using `reportsDestination` and `metricsDestination`). This generates text files (such as `composables.txt`) specifying whether each function is restartable and skippable, and identifying which parameters are unstable.
Senior-Level Follow-up
What is the difference between a Composable being restartable vs skippable?
Common Mistake
Passing lambda parameters (like `onClick = { model.action() }`) that capture unstable closures, which invalidates skippability unless wrapped in a `remember` block or compiler lambda optimization.
