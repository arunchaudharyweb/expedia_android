Section 4: Jetpack Compose

⏱ 8 min read ✓ Part 1 Core

Streamlined Jetpack Compose rendering runtime, state tracking, optimization constraints, and stability rules.

1. @Composable compiler transformation (Slot Table mapping) Runtime

Quick Cheat Sheet

  • Compiler Plugin: The Compose compiler transforms functions annotated with @Composable, injecting an implicit parameter for the Composer and inserting start/end group calls.
  • Slot Table: A flat array-based data structure representing the composition tree. It stores the position, keys, and values of all UI nodes, states, and cached values.
What happens to a @Composable function during compilation?
Answer
The compiler adds an implicit $composer parameter to the function signature and injects calls to $composer.startRestartGroup(...) and $composer.endRestartGroup(). This allows the runtime to track the function's identity, input arguments, and state reads in the Slot Table.
Senior-Level Follow-up
How does the Slot Table store and align keys and values of remember calls during structural tree changes?
Common Mistake
Thinking that Composable functions return views or construct static class hierarchies similar to standard Java objects.

2. State & MutableState (Snapshot system) State

Quick Cheat Sheet

  • State Tracking: During initial Composition, when a composable reads a State.value, the Compose runtime records this read in the active RecomposeScope.
  • Snapshot System: The database engine of Compose state. It tracks values at a specific point in time, allowing isolated reads and writes across threads (Multi-Version Concurrency Control). Commit actions schedule recomposition.
How does the Compose runtime track state reads?
Answer
The Compose runtime utilizes thread-local Snapshot contexts. During the Composition phase, the current RecomposeScope is registered as a listener on the thread. When a State.value getter is invoked, it triggers a read callback that links the State object instance to the active RecomposeScope, saving this dependency in the Slot Table. When the state changes, the runtime invalidates only the registered scopes.
Senior-Level Follow-up
Explain the internal lock-free concurrency model used by the Compose Snapshot system when updating state across threads.
Common Mistake
Mutating standard Kotlin variables directly and expecting the UI to update. (Updates must be wrapped in `MutableState` to be tracked).

3. remember vs rememberSaveable State

Quick Cheat Sheet

  • remember: Caches the value in the Slot Table, returning it on subsequent recompositions. It survives recompositions, but is cleared if the composable leaves the tree or the Activity is recreated.
  • rememberSaveable: Caches the value like remember, but also serializes it into Android's Bundle system. This preserves state across screen rotations and system process death.
What is the difference between remember and rememberSaveable?
Answer
remember caches values in memory within the Slot Table; these are lost during configuration changes (like rotation) or process death. rememberSaveable automatically serializes the value into a Bundle, allowing it to survive configuration changes and system-initiated process death.
Senior-Level Follow-up
How do you implement a custom Saver to serialize complex objects with rememberSaveable?
Common Mistake
Using rememberSaveable for non-parcelable types without declaring a custom Saver, resulting in runtime crashes.

4. Recomposition Phase & Scope Optimization Runtime

Quick Cheat Sheet

  • Recomposition Scope: The smallest executable block (usually a Composable function) that can be run independently to update the UI.
  • Incremental Recomposition: The runtime uses the Slot Table to skip execution of composables whose arguments have not changed, updating only invalid scopes.
What is a RecomposeScope and how does it optimize UI updates?
Answer
A RecomposeScope is an executable block representing a section of the composition tree. The runtime maps state read dependencies to these scopes. When a state change occurs, the runtime invalidates only the specific scopes reading that state, re-running only those functions instead of the entire composable hierarchy.
Senior-Level Follow-up
Describe how the Recomposer coordinates composition updates with the platform Choreographer VSYNC signals.
Common Mistake
Performing side-effects or heavy computations directly in the body of a Composable function, causing them to execute on every recomposition.

5. LaunchedEffect vs DisposableEffect (Side Effects) Lifecycle

Quick Cheat Sheet

  • LaunchedEffect(key): Launches a coroutine when entering the composition. If the key changes, it cancels the active coroutine and launches a new one. It automatically cancels when leaving the composition.
  • DisposableEffect(key): Designed for setup/cleanup actions. Must end with an onDispose block containing the cleanup logic (like unregistering listeners). It re-runs when keys change.
When should you use DisposableEffect instead of LaunchedEffect?
Answer
Use DisposableEffect when you need to register a callback or observer that requires explicit cleanup (like a broadcast receiver or location listener). It provides the onDispose callback to unregister resources. Use LaunchedEffect when you need to perform asynchronous, suspending coroutine actions (like showing a snackbar or fetching network data).
Senior-Level Follow-up
How does `rememberCoroutineScope()` compare to `LaunchedEffect`? (LaunchedEffect runs on composition; rememberCoroutineScope provides a scope to launch coroutines in response to user events like clicks).
Common Mistake
Leaving the onDispose block empty inside `DisposableEffect`, leaking the registered listener.

6. derivedStateOf vs remember(key) State

Quick Cheat Sheet

  • remember(key): Re-calculates the value only when the input key changes, caching the result.
  • derivedStateOf: Creates a new state based on inputs that change frequently (like scroll offsets), but filters updates so it only triggers recomposition when the *result* of the calculation changes (e.g. `scrollOffset > 100`).
When should you use derivedStateOf vs standard remember(key)?
Answer
Use remember(key) when the input key changes infrequently and you want to re-execute a calculation on key change. Use derivedStateOf when the input states change frequently (like scroll positions), but you only want to trigger recomposition when the *result* of the calculation changes (e.g. `scrollOffset > 100`). This prevents excessive recomposition.
Senior-Level Follow-up
Describe the dependency tracking mechanism of derivedStateOf inside the Compose Snapshot system.
Common Mistake
Wrapping a simple state mapping (e.g. transforming `x` to `x + 1`) in `derivedStateOf`, which introduces unnecessary allocation overhead (simple computations should use `remember(x)` instead).

7. Compose Stability and Skippability Performance

Quick Cheat Sheet

  • Stable/Immutable: Properties cannot change, or Compose is notified when they change. The compiler can skip recomposing the function if the stable arguments are equal to the previous values.
  • Unstable: Properties can change without notifying Compose (e.g. standard Java collections like List). If a Composable receives an unstable parameter, the compiler **always** recomposes it, bypassing skippability.
Why is a standard java.util.List considered Unstable by the Compose compiler, and how do you fix it?
Answer
The compiler cannot guarantee that a List reference is immutable (it could be cast to a mutable list). Since it cannot track mutations of standard lists, it marks them as unstable. To fix this: 1. Annotate the model class containing the list with @Immutable or @Stable. 2. Use Kotlinx Immutable Collections (like PersistentList). 3. Annotate the parameter types directly to force compile-time stability.
Senior-Level Follow-up
How do you generate and analyze Compose compiler metrics (composables.txt) to identify unstable types?
Common Mistake
Passing lambdas (like `onClick = { model.action() }`) that capture unstable closures, which invalidates skippability unless wrapped in a `remember` block.

8. Performance: Deferring reads (lambda modifiers) Performance

Quick Cheat Sheet

  • Compose Phases: Composition (what to show) -> Layout (where to place) -> Draw (how to render).
  • Deferring Reads: Wrap state reads in a lambda (e.g., inside Modifier.offset { IntOffset(...) }). This defers the state read to the Layout or Draw phase, bypassing the Composition phase entirely and preventing the enclosing Composable function from recomposing.
Why is the lambda version of Modifier.offset preferred during scroll animations?
Answer
The lambda version Modifier.offset { IntOffset(0, scrollState.value) } wraps the state read inside a lambda. This defers the state read to the Layout phase, bypassing the Composition phase entirely. Only layout measurement and drawing are re-executed, preventing full recompositions of the UI tree on every pixel scrolled. The non-lambda version reads state during Composition, causing continuous recompositions and UI lag.
Senior-Level Follow-up
How does using graphicsLayer with lambda arguments optimize GPU hardware layers during rendering?
Common Mistake
Reading high-frequency scroll positions or offsets in the composition scope instead of deferring them to the layout/draw phases.
