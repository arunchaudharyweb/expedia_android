Problem 1: Debounced Search Autocomplete Search

Prompt

Implement a search text handler that takes user input queries, waits for a 300ms pause in typing, and then fetches results from a remote API. Ensure that in-flight requests are cancelled if the user types a new query, and handle network errors without crashing.

Clean Kotlin Solution

import kotlinx.coroutines.FlowPreview
import kotlinx.coroutines.flow.*

class SearchViewModel(private val apiService: ApiService) : ViewModel() {
    private val queryFlow = MutableStateFlow("")

    @OptIn(FlowPreview::class)
    val searchResults: StateFlow = queryFlow
        .debounce(300) // Delay execution by 300ms
        .filter { it.isNotBlank() } // Skip empty inputs
        .distinctUntilChanged() // Ignore duplicate values
        .flatMapLatest { query -> // Cancel previous in-flight search requests
            flow {
                emit(SearchResult.Loading)
                val results = apiService.searchHotels(query)
                emit(SearchResult.Success(results))
            }.catch { err -> 
                emit(SearchResult.Error(err.message ?: "Unknown error"))
            }
        }
        .stateIn(
            scope = viewModelScope,
            started = SharingStarted.WhileSubscribed(5000),
            initialValue = SearchResult.Idle
        )

    fun onQueryChanged(newQuery: String) {
        queryFlow.value = newQuery
    }
}

sealed interface SearchResult {
    object Idle : SearchResult
    object Loading : SearchResult
    data class Success(val hotels: List) : SearchResult
    data class Error(val message: String) : SearchResult
}
Why is the catch operator placed inside flatMapLatest?
Answer
If catch is placed at the end of the Flow chain, it will intercept exceptions but terminate the entire stream, stopping any future search queries. Placing catch inside the nested flatMapLatest flow intercepts the error, emits a failure state, and keeps the parent query stream alive for subsequent queries.
Common Mistake
Using a standard `flatMapMerge` instead of `flatMapLatest`, which processes queries concurrently without cancelling obsolete requests.

Problem 2: Retry with Exponential Backoff Async

Prompt

Write a generic suspending function that retries a network request up to N times, applying an exponential backoff delay (e.g. 1s, 2s, 4s...) between attempts. The retry loop should stop immediately if a non-retryable exception is encountered.

Clean Kotlin Solution

import kotlinx.coroutines.delay
import java.io.IOException

suspend fun <T> retryWithBackoff(
    times: Int = 3,
    initialDelayMs: Long = 1000,
    factor: Double = 2.0,
    isRetryable: (Throwable) -> Boolean = { it is IOException },
    block: suspend () -> T
): T {
    var currentDelay = initialDelayMs
    repeat(times - 1) {
        try {
            return block() // Success: return result immediately
        } catch (e: Throwable) {
            // Terminate if the exception is non-retryable
            if (!isRetryable(e)) throw e
            
            delay(currentDelay)
            currentDelay = (currentDelay * factor).toLong()
        }
    }
    return block() // Final attempt: propagates any exceptions
}
How does this implementation respect coroutine cancellation?
Answer
The delay() function is cancellable. If the coroutine is cancelled, delay() throws a CancellationException, exiting the retry loop immediately. It also propagates the cancellation exception up the tree, supporting structured cancellation.
Common Mistake
Swallowing all exceptions (including `CancellationException`) inside the catch block, which breaks coroutine cancellation.

Problem 3: Thread-Safe In-Memory Cache with TTL Caching

Prompt

Implement a generic, thread-safe, in-memory cache that expires entries after a configurable Time-To-Live (TTL). Ensure the implementation is testable with custom time providers.

Clean Kotlin Solution

import java.util.concurrent.ConcurrentHashMap

fun interface TimeProvider {
    fun currentTimeMillis(): Long
}

class TtlCache<K : Any, V : Any>(
    private val ttlMs: Long,
    private val timeProvider: TimeProvider = TimeProvider { System.currentTimeMillis() }
) {
    private val store = ConcurrentHashMap<K, CacheEntry<V>>()

    fun put(key: K, value: V) {
        val expiryTime = timeProvider.currentTimeMillis() + ttlMs
        store[key] = CacheEntry(value, expiryTime)
    }

    fun get(key: K): V? {
        val entry = store[key] ?: return null
        if (timeProvider.currentTimeMillis() > entry.expiryTime) {
            store.remove(key) // Evict expired entry
            return null
        }
        return entry.value
    }

    fun clear() = store.clear()

    private data class CacheEntry(val value: T, val expiryTime: Long)
}
Why is TimeProvider abstracting System.currentTimeMillis()?
Answer
To support unit testing. By injecting a mock or fake TimeProvider, we can manipulate time in tests (e.g. fast-forwarding time past the TTL threshold) and verify eviction behavior without waiting for real clock ticks.
Common Mistake
Using a standard `HashMap` in multithreaded environments, which can cause race conditions or infinite loops.

Problem 4: MVI ViewModel & Reducer Pattern MVI

Prompt

Implement an Android ViewModel using the MVI pattern. The class should expose a single StateFlow for state and process inputs using a pure reducer function.

Clean Kotlin Solution

import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow
import kotlinx.coroutines.launch

data class HotelUiState(
    val hotels: List = emptyList(),
    val isLoading: Boolean = false,
    val error: String? = null
)

sealed interface HotelIntent {
    object LoadHotels : HotelIntent
    data class ApplyFilter(val query: String) : HotelIntent
}

class HotelViewModel(private val repository: HotelRepository) : ViewModel() {
    private val _uiState = MutableStateFlow(HotelUiState())
    val uiState: StateFlow = _uiState.asStateFlow()

    fun processIntent(intent: HotelIntent) {
        viewModelScope.launch {
            when (intent) {
                is HotelIntent.LoadHotels -> loadHotels()
                is HotelIntent.ApplyFilter -> filterHotels(intent.query)
            }
        }
    }

    private suspend fun loadHotels() {
        _uiState.value = _uiState.value.copy(isLoading = true, error = null)
        try {
            val list = repository.fetchHotels()
            _uiState.value = _uiState.value.copy(isLoading = false, hotels = list)
        } catch (e: Exception) {
            _uiState.value = _uiState.value.copy(isLoading = false, error = e.message)
        }
    }

    private fun filterHotels(query: String) {
        val filtered = _uiState.value.hotels.filter { it.contains(query, ignoreCase = true) }
        _uiState.value = _uiState.value.copy(hotels = filtered)
    }
}
What are the benefits of exposing a single UI state object?
Answer
It creates a single source of truth for the View. It prevents UI bugs where different variables (like loading state and lists) are out of sync, and simplifies state reproduction during unit testing.
Common Mistake
Exposing multiple independent state flows in the ViewModel, leading to state synchronization issues.

Problem 5: Offline-First Repository Flow Offline-First

Prompt

Implement a repository that retrieves cached data immediately and then fetches fresh data from the network. Save the network results to the local database and emit the updated listings via a Flow, handling errors gracefully.

Clean Kotlin Solution

import kotlinx.coroutines.flow.*

interface LocalSource {
    fun getCachedData(): Flow>
    suspend fun saveCachedData(data: List)
}

interface RemoteSource {
    suspend fun fetchFreshData(): List
}

class OfflineFirstRepository(
    private val local: LocalSource,
    private val remote: RemoteSource
) {
    fun getHotelsStream(): Flow>> = flow {
        emit(Resource.Loading(null))
        
        // 1. Emit current cache immediately
        val initialCache = local.getCachedData().firstOrNull() ?: emptyList()
        emit(Resource.Loading(initialCache))
        
        // 2. Fetch fresh network data and write to cache
        try {
            val freshData = remote.fetchFreshData()
            local.saveCachedData(freshData)
            
            // 3. Emit updated data from the cache source of truth
            local.getCachedData().collect { updatedList ->
                emit(Resource.Success(updatedList))
            }
        } catch (e: Exception) {
            emit(Resource.Error(e.message ?: "Network error", initialCache))
        }
    }
}

sealed class Resource {
    data class Loading(val data: T?) : Resource()
    data class Success(val data: T) : Resource()
    data class Error(val message: String, val data: T?) : Resource()
}
Why do we use firstOrNull() on the database flow instead of collecting it?
Answer
We call firstOrNull() to get the initial cached snapshot to show immediate data. We don't want to start an infinite collection loop inside the repository's loader block, which would block execution of the network request. Once the network request finishes, we collect the database stream dynamically.
Common Mistake
Emitting network results directly to the UI without writing them to the database first, violating the single source of truth principle.

Problem 6: Cancellable Sequential Execution Queue Queue

Prompt

Implement a task execution queue that processes image uploads sequentially, one at a time. The queue should support canceling a specific task and must prevent coroutine context leaks.

Clean Kotlin Solution

import kotlinx.coroutines.*
import kotlinx.coroutines.channels.Channel
import java.util.concurrent.ConcurrentHashMap

class UploadQueue(private val scope: CoroutineScope) {
    private val taskChannel = Channel(Channel.UNLIMITED)
    private val activeJobs = ConcurrentHashMap()

    init {
        scope.launch {
            for (task in taskChannel) {
                // Execute tasks sequentially
                val job = launch {
                    try {
                        task.action()
                    } catch (e: CancellationException) {
                        // Task was cancelled
                    } finally {
                        activeJobs.remove(task.id)
                    }
                }
                activeJobs[task.id] = job
                job.join() // Wait for task completion before continuing the loop
            }
        }
    }

    fun enqueue(id: String, action: suspend () -> Unit) {
        val task = UploadTask(id, action)
        taskChannel.trySend(task)
    }

    fun cancelTask(id: String) {
        activeJobs[id]?.cancel() // Cancel specific task execution
    }

    private data class UploadTask(val id: String, val action: suspend () -> Unit)
}
How does job.join() guarantee sequential task execution?
Answer
The consumer loop reads tasks from the channel. For each task, it spawns a coroutine and stores the reference. By calling job.join(), it suspends the consumer loop until the active task completes or is cancelled, ensuring that the next task in the queue is not processed until the current one exits.
Common Mistake
Launching tasks inside a loop without calling join(), which causes them to execute concurrently.

Problem 7: SOLID Clean Code Refactoring SOLID

Prompt

The following class is highly coupled, violates Single Responsibility Principle, and is untestable because it instantiates its own dependencies. Refactor it using SOLID principles and construct a mockable implementation.

Clean Kotlin Solution

interface BookingRepository {
    suspend fun saveBookingLocal(hotelId: String, userId: String)
    suspend fun confirmBookingRemote(hotelId: String, userId: String)
}

interface NotificationSender {
    suspend fun sendEmail(message: String, email: String)
}

class CleanBookingService(
    private val repository: BookingRepository,
    private val notifier: NotificationSender
) {
    suspend fun executeBooking(hotelId: String, userId: String, userEmail: String): Boolean {
        return try {
            repository.confirmBookingRemote(hotelId, userId)
            repository.saveBookingLocal(hotelId, userId)
            notifier.sendEmail("Booking confirmed for $hotelId", userEmail)
            true
        } catch (e: Exception) {
            false
        }
    }
}
Which SOLID principles are satisfied by this refactoring?
Answer
1. **Single Responsibility (SRP)**: The booking service coordinates execution flows; data persistence and notification actions are delegated to dedicated interfaces. 2. **Dependency Inversion (DIP)**: The service depends on interfaces (abstractions) instead of concrete database and API clients. 3. **Interface Segregation (ISP)**: Dependencies are split into small, focused interfaces.
Common Mistake
Hardcoding concrete implementations (like API clients) inside constructors, which prevents mocking.

Problem 8: Unit Testing with Fake Repository Testing

Prompt

Write a unit test suite for a ViewModel that fetches travel deals. Instead of using MockK, implement a Fake Repository to simulate network latency, successes, and resource errors.

Clean Kotlin Solution

interface DealsRepository {
    suspend fun getDeals(): List
}

// Stateful Fake Repository for testing
class FakeDealsRepository : DealsRepository {
    private val deals = mutableListOf()
    var shouldThrowError = false

    fun addDeals(newList: List) {
        deals.clear()
        deals.addAll(newList)
    }

    override suspend fun getDeals(): List {
        if (shouldThrowError) throw Exception("Network failed")
        return deals
    }
}

class DealsViewModel(private val repository: DealsRepository) : ViewModel() {
    private val _uiState = MutableStateFlow<DealsUiState>(DealsUiState.Loading)
    val uiState = _uiState.asStateFlow()

    fun fetchDeals() {
        viewModelScope.launch {
            _uiState.value = DealsUiState.Loading
            try {
                val list = repository.getDeals()
                _uiState.value = DealsUiState.Success(list)
            } catch (e: Exception) {
                _uiState.value = DealsUiState.Error(e.message ?: "Failed")
            }
        }
    }
}
Why are Fakes preferred over Mocks in large-scale testing?
Answer
Fakes provide a lightweight, consistent, stateful implementation (like an in-memory database). They run faster, make tests easier to read by avoiding verbose stubbing configuration, and prevent tests from breaking when internal method invocation orders change.
Common Mistake
Over-mocking simple entities (like data models), which increases configuration boilerplate.
