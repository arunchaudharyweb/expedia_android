Code Testability & Testing Guidelines

⏱ 12 min read ✓ Round 2 Core

Best practices for testing mobile applications, focusing on state transitions, test double design, concurrent test environments, and database cache validation.

1. ViewModel Testing: States vs Implementations ViewModel

When testing ViewModels, focus on verifying **state transitions** and **outputs**, rather than private implementation details. This ensures tests remain valid even if the internal implementation is refactored.

Test Design: Instead of verifying that a specific repository method was called (which is an implementation detail), verify that triggering an intent transitions the UI state correctly (e.g. from Loading to Success or Error). 

// CORRECT: Testing state transitions
@Test
fun testLoadingStateTransitions() = runTest {
    viewModel.processIntent(HotelIntent.LoadHotels)
    
    // Assert State Flow transitions through Loading to Success
    viewModel.uiState.test {
        assertEquals(HotelUiState(isLoading = true), awaitItem())
        assertEquals(HotelUiState(hotels = expectedList), awaitItem())
    }
}
Interview Question
What should you verify when writing unit tests for a ViewModel?
Answer
Focus on verifying state transitions and output emissions (such as state flow updates or navigation events). Avoid verification of internal method calls or private fields, as this couples tests to the implementation and makes refactoring difficult.
Senior Follow-Up
How do you mock or fake saved state handles during ViewModel unit testing?
Common Candidate Mistake
Asserting calls to internal ViewModel methods instead of verifying changes in the exposed public state flows.

2. Fake vs Mock Trade-offs Methodology

Understand the trade-offs between different categories of test doubles:

  • Fakes: Working implementations of a component with simplified behavior (e.g., an in-memory repository). They are resilient to internal interface changes and simulate real system behavior, making tests more reliable.
  • Mocks: Objects configured with pre-programmed expectations (e.g., MockK setups). While quick to configure, mocks couple tests to the component's internal calls, requiring test updates whenever the code is refactored.
// A Fake Repository is a simple class rather than a MockK setup
class FakeHotelRepository : HotelRepository {
    private val list = mutableListOf()
    
    override suspend fun fetchHotels() = list
    fun addHotels(items: List) { list.addAll(items) }
}
Interview Question
Explain why fakes are often preferred over mocks for repository and service test dependencies.
Answer
Fakes provide a simplified, self-contained implementation (such as an in-memory cache) that behaves consistently across tests. This makes tests more resilient to refactoring, as they do not depend on the exact method invocation patterns required by mocks.
Senior Follow-Up
Under what conditions are mocks more appropriate than fakes?
Common Candidate Mistake
Over-mocking simple models and utilities (like data classes or logger wrappers), which adds configuration overhead and slows down test execution.

3. Coroutine and Flow Testing (runTest, Turbine) Concurrency

Testing concurrent coroutines requires using runTest to skip delays and control execution flow via a test scheduler.

Use the **Turbine** library to test flow emissions. It captures emissions and lets you assert values sequentially, avoiding the need to collect flows manually or manage test scopes inside tests. 

@Test
fun testPriceUpdatesFlow() = runTest {
    val priceFlow = repository.observePrices()
    
    // Turbine test wrapper
    priceFlow.test {
        assertEquals(100, awaitItem())
        repository.updatePrice(120)
        assertEquals(120, awaitItem())
        cancelAndConsumeRemainingEvents()
    }
}
Interview Question
Explain how runTest manages virtual time delays during testing.
Answer
runTest uses a TestCoroutineScheduler to intercept delays (such as delay(1000)) and advance virtual time instantly, allowing time-dependent tests to complete immediately without blocking the test runner.
Senior Follow-Up
How do you handle tests where a coroutine dispatcher (like Dispatchers.IO) is hardcoded inside a repository?
Common Candidate Mistake
Using a standard runBlocking for coroutines that contain delays. This causes the test suite to block and slow down, rather than executing instantly with virtual time.

4. Repository Testing: In-Memory Room vs Fake Database

When testing data repositories, choose between an in-memory database and fakes based on the requirements:

  • In-Memory Room Database: Runs on a local SQLite instance in memory. This is slower to execute but validates queries, database triggers, and schema migrations against real SQLite constraints.
  • Fake Data Source: A simple implementation (such as a map-backed cache). It is faster to execute but does not validate SQL syntax or foreign key constraints.
// Room in-memory builder instance
val db = Room.inMemoryDatabaseBuilder(
    ApplicationProvider.getApplicationContext(),
    HotelDatabase::class.java
).allowMainThreadQueries().build() // Allowed only for local unit tests
Interview Question
What are the trade-offs of using an in-memory Room database versus a fake for repository testing?
Answer
An in-memory Room database validates SQL queries, constraint checks, and database schemas, but is slower to initialize and execute. A fake data source is faster to execute but does not validate SQL queries or schema definitions, which can hide database-related bugs.
Senior Follow-Up
How do you close the in-memory database instance to prevent memory leaks between test runs?
Common Candidate Mistake
Sharing a single in-memory database instance across multiple tests, causing database locks and test pollution.

5. Designing for Code Testability Architecture

Code testability is an architectural requirement. To ensure components can be isolated and tested effectively:

  • Constructor Injection: Avoid initializing dependencies inside classes. Pass dependencies (such as APIs or repositories) as constructor arguments to allow them to be mocked or faked during tests.
  • Avoid Static Singletons: Do not access global singletons or static instances directly. Accessing static singletons introduces hidden dependencies and side effects, making it difficult to isolate components for testing.
  • Avoid Hardcoded Dispatchers: Do not use hardcoded dispatchers (e.g. Dispatchers.IO) inside classes. Instead, pass dispatchers as constructor parameters to allow them to be overridden with a TestDispatcher during tests.
Interview Question
What are the key architectural changes needed to make a legacy class testable?
Answer
Use constructor injection to pass dependencies instead of initializing them inside the class. Avoid accessing static singletons directly, and inject coroutine dispatchers as constructor parameters so they can be overridden with a TestDispatcher during tests.
Senior Follow-Up
How do DI frameworks (like Hilt or Dagger) simplify testing with custom test doubles?
Common Candidate Mistake
Hardcoding coroutine dispatchers directly in helper methods, preventing test runners from controlling execution flow using virtual time.
