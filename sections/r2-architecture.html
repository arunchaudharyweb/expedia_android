Streamlined overview of modern presentation, domain, and data layer architectures, data flows, and dependency injection scopes.

1. MVVM (Model-View-ViewModel) Architecture

Quick Cheat Sheet

  • MVVM Structure:
    • View: Renders the UI and routes events. Confined to UI elements.
    • ViewModel: Holds UI state and encapsulates business logic. Does not reference the View, exposing updates via reactive flows.
    • Model: Business logic, database, and network interfaces.
  • Pros/Cons:
    • Pros: Decouples UI from logic, lifecycle awareness via Jetpack ViewModel, eliminates MVP view interface boilerplate.
    • Cons: Complex screens can lead to bloated ViewModels.
What is the key difference between MVVM and MVP?
Answer
In MVP, the Presenter holds a direct reference to the View interface, directing updates via explicit function calls. In MVVM, the ViewModel has no reference to the View; instead, it exposes state properties (such as Flow or LiveData), and the View observes these properties. This decouples the presentation layer from specific UI implementations.
Senior-Level Follow-up
How does the viewModelScope handle automatic cancellation of background tasks when the activity is finished?
Common Mistake
Exposing mutable states (like MutableStateFlow) directly to the View, allowing views to modify the state directly instead of routing actions through the ViewModel.

2. MVI & Unidirectional Data Flow Architecture

Quick Cheat Sheet

  • MVI Structure:
    • Intent: Represents user actions (e.g. `LoadHotelDetails`).
    • State: Single, immutable state object representing the entire screen UI.
    • Model: Receives Intents, processes them, and outputs states using a state reducer.
  • Pros/Cons:
    • Pros: Single source of truth, predictable state transitions, and easier debugging using state logs.
    • Cons: Boilerplate overhead for simple screens, and memory allocations for state copies on every update.
Explain Unidirectional Data Flow (UDF) inside MVI.
Answer
UDF means data flows in a single circular path: 1. The View dispatches user intents (e.g. click button). 2. The ViewModel processes the intent, queries repositories if needed, and feeds the action to a Reducer. 3. The Reducer creates a new immutable State based on the previous State and current action. 4. The View observes and renders this new State. This prevents out-of-sync states.
Senior-Level Follow-up
How do you prevent state duplication overhead when copy-allocating large immutable data classes?
Common Mistake
Treating transient events (like toast alerts or navigation) as persistent state properties inside the screen's main state flow, causing duplicate alerts on rotation.

3. Clean Architecture Layers Architecture

Quick Cheat Sheet

  • Presentation Layer: ViewModels and UI views. Depends on Domain Layer.
  • Domain Layer: Core business entities and Use Cases. Depends on nothing (independent). No platform or framework references.
  • Data Layer: Repositories, database (Room), and network (Retrofit) clients. Depends on Domain Layer.
What are the three core layers of Clean Architecture, and what are their dependencies?
Answer
1. **Presentation Layer**: ViewModels and UI views. Depends on Domain Layer. 2. **Domain Layer**: Core business entities and Use Cases. Depends on nothing (independent). 3. **Data Layer**: Repositories, database (Room), and network (Retrofit) clients. Depends on Domain Layer. The dependency rule specifies that source code dependencies must point inward toward the Domain Layer.
Senior-Level Follow-up
How does Dependency Inversion allow the Domain layer to define repository interfaces implemented in the Data layer?
Common Mistake
Allowing Android platform dependencies (like Context or LiveData) to leak into the Domain layer Use Cases.

4. Repository Pattern & Single Source of Truth Design Pattern

Quick Cheat Sheet

  • Repository Pattern: Decouples the presentation layer from specific data sources (API vs local database).
  • Single Source of Truth (SSOT): The UI observes only the local database stream. Network requests update the database, which automatically updates the UI stream, ensuring state consistency.
Why should a Repository hide its data sources?
Answer
To decouple business and presentation logic from data retrieval mechanisms. It allows migrating local databases or network libraries without modifying the domain or presentation layers. It also coordinates caching policies and handles sync conflicts.
Senior-Level Follow-up
How do you implement an offline-first caching policy with expiration times inside a Repository?
Common Mistake
Exposing database entities (Room models) or network models (Retrofit GSON objects) directly to the presentation layer instead of mapping them to domain models.

5. Dependency Injection Principles DI

Quick Cheat Sheet

  • Dependency Injection (DI): Software design pattern where a class's dependencies are supplied by external entities rather than created by the class itself.
  • Constructor Injection: Preferred pattern. Declares dependencies explicitly in the class constructor, facilitating easy unit testing.
  • Field Injection: Used when the class is instantiated by the Android platform framework (like Activities or Fragments), using annotations.
What is the difference between Constructor Injection and Field Injection?
Answer
Constructor Injection injects dependencies via the class constructor. It is preferred because it makes dependencies explicit and allows instantiating the class without a DI framework during testing. Field Injection is used when the class is instantiated by the Android platform framework (like Activities or Fragments). The DI framework injects dependencies into annotated fields after initialization.
Senior-Level Follow-up
Compare Dagger 2's compile-time graph validation with runtime service locator patterns.
Common Mistake
Instantiating singletons manually (e.g. using `object` singletons or static fields), which hardcodes dependencies and complicates unit testing.

6. Hilt in Android (Scopes & Components) DI

Quick Cheat Sheet

  • Hilt Components: Built-in components that align with Android's lifecycles:
    • SingletonComponent: Tied to the Application lifecycle.
    • ActivityComponent: Tied to the Activity lifecycle.
    • ViewModelComponent: Tied to the ViewModel lifecycle.
  • Hilt Scopes: Restricts instance creation to specific lifecycles (e.g. `@Singleton`, `@ActivityScoped`, `@ViewModelScoped`).
What is the role of the @InstallIn annotation in Hilt modules?
Answer
The @InstallIn annotation tells Hilt which standard Dagger component the module's dependencies should be bound to. For example, @InstallIn(SingletonComponent::class) binds dependencies to the application-level lifecycle (singleton), while @InstallIn(ActivityComponent::class) limits dependencies to the lifecycle of individual activities.
Senior-Level Follow-up
Explain the difference between @Binds and @Provides in a Hilt module.
Common Mistake
Failing to annotate the custom Application class with @HiltAndroidApp, causing compilation errors or runtime injection failures.
