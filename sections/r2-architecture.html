Section 6: Android Architecture

⏱ 15 min read ✓ Part 2 Design

Modern architectural paradigms, structural design patterns, data flows, decoupling principles, and dependency injection implementations for enterprise-grade mobile systems.

1. MVVM (Model-View-ViewModel) Architecture

Pros and Cons

Pros: Separates UI rendering from business logic, leverages lifecycle-aware components (ViewModel), reduces boilerplate compared to MVP, and supports reactive data binding.
Cons: Complex screens can lead to bloated ViewModels (god classes), and managing multiple UI state flows can become difficult if not unified.

Testability

High testability. ViewModels do not contain Android framework references, allowing them to be tested using standard JVM unit tests with mocked repositories.

Scalability

Scales well for medium-sized applications. However, as the number of features grows, sharing state between ViewModels requires clear repository caching or coordination layers.

Real Production Examples

A hotel details page. The View observes a hotelDetailsState flow exposed by the ViewModel. When user selects a date, the ViewModel updates the state, and the UI recomposes automatically.

Interview Questions

How does MVVM differ from MVP (Model-View-Presenter)?
Answer
In MVP, the Presenter holds a direct reference to the View interface, directing updates via explicit function calls. In MVVM, the ViewModel has no reference to the View; instead, it exposes state properties (such as Flow or LiveData), and the View observes these properties. This decouples the presentation layer from specific UI implementations.
Senior-Level Follow-up
How does the viewModelScope handle automatic cancellation of background tasks when the activity is finished?
Common Mistake
Exposing mutable states (like MutableStateFlow) directly to the View, allowing views to modify the state directly instead of routing actions through the ViewModel.

2. MVI (Model-View-Intent) Architecture

Pros and Cons

Pros: Unidirectional Data Flow (UDF), single immutable state container representing the screen, predictable transitions, and simplified debugging using state logs.
Cons: High boilerplate overhead for simple screens, and memory allocations for state copies on every update.

Testability

Excellent testability. State transitions can be tested by sending Intents to a state reducer and asserting the resulting state outputs.

Scalability

Highly scalable for complex, multi-state screen flows. It prevents state conflicts (race conditions) by funneling all mutations through a single reducer queue.

Real Production Examples

An Expedia search checkout screen. User interactions (e.g. input voucher, change payment method) are dispatched as Intents (events). The reducer combines the current state and intent, outputting a new, single immutable CheckoutState.

Interview Questions

Explain Unidirectional Data Flow (UDF) inside MVI.
Answer
UDF means data flows in a single circular path: 1. The View dispatches user intents (e.g. click button). 2. The ViewModel processes the intent, queries repositories if needed, and feeds the action to a Reducer. 3. The Reducer creates a new immutable State based on the previous State and current action. 4. The View observes and renders this new State. This prevents out-of-sync states.
Senior-Level Follow-up
How do you prevent state duplication overhead when copy-allocating large immutable data classes?
Common Mistake
Treating transient events (like toast alerts or navigation) as persistent state properties inside the screen's main state flow, causing duplicate alerts on rotation.

3. Clean Architecture Architecture

Pros and Cons

Pros: Strict separation of concerns, decouples core business logic from framework libraries (like Room or Retrofit), and makes codebase maintenance straightforward.
Cons: Overkill for small projects, introduces a large number of model mappings and interface wrapper classes.

Testability

Outstanding testability. Business logic in Use Cases contains no platform or framework references, allowing fast unit testing on the JVM.

Scalability

Perfect for large, multi-module enterprise codebases. Features can be developed independently inside separate domain and data modules.

Real Production Examples

A Booking module. The View notifies the ViewModel, which calls the ConfirmBookingUseCase (domain layer). The Use Case coordinates data verification and updates the repository layer, completely isolated from network or DB frameworks.

Interview Questions

What are the three core layers of Clean Architecture, and what are their dependencies?
Answer
1. **Presentation Layer**: ViewModels and UI views. Depends on Domain Layer. 2. **Domain Layer**: Core business entities and Use Cases. Depends on nothing (independent). 3. **Data Layer**: Repositories, database (Room), and network (Retrofit) clients. Depends on Domain Layer. The dependency rule specifies that source code dependencies must point inward toward the Domain Layer.
Senior-Level Follow-up
How does Dependency Inversion allow the Domain layer to define repository interfaces implemented in the Data layer?
Common Mistake
Allowing Android platform dependencies (like Context or LiveData) to leak into the Domain layer Use Cases.

4. Repository Pattern Design Pattern

Pros and Cons

Pros: Decouples data sources from domain logic, acts as a single source of truth, manages caching strategies, and coordinates network and database synchronization.
Cons: Adds minor implementation overhead, and can become a bottleneck if too many caching rules are implemented in a single class.

Testability

Allows easy testing of domain logic. By mocking the repository interface, you can simulate network errors, disk latency, or database misses without launching network services.

Scalability

Essential for offline-first applications, as it provides a clean interface that hides changes in data sources (e.g. migrating database schema or changing networking libraries).

Real Production Examples

Expedia Hotel details repository. When requested, HotelRepository first returns cached data from a local Room database, while simultaneously launching a background network fetch. Once complete, it writes to Room, updating the UI stream.

Interview Questions

Why should a Repository hide its data sources, and how does it implement a single source of truth?
Answer
A repository should hide data sources to decouple business logic from data retrieval mechanisms. It implements a single source of truth by directing all reads through a local database cache (like Room). Network updates write directly to the database; the domain layer collects state changes from the database stream, ensuring a single, consistent state representation.
Senior-Level Follow-up
How do you implement an offline-first caching policy with expiration times inside a Repository?
Common Mistake
Exposing database entities (Room models) or network models (Retrofit GSON objects) directly to the presentation layer instead of mapping them to domain models.

5. Dependency Injection (DI Principles) DI

Pros and Cons

Pros: Promotes loose coupling, facilitates mock-based unit testing, defines clear scopes and lifecycles for shared objects, and prevents boilerplate code.
Cons: Learning curve, compile-time overhead (for annotation processors), and complex dependency configurations can be difficult to debug.

Testability

Crucial for testability. Allows replacing real network clients or database instances with mock or fake implementations during testing, avoiding real-world side effects.

Scalability

Crucial for large codebases. Allows developers to modularize the app, injecting mock components in development modes to compile and test individual features quickly.

Real Production Examples

A network client. Instead of each repository instantiating its own OkHttpClient, the network client is declared as a singleton inside a DI module and injected into all repository constructors.

Interview Questions

What is the difference between Constructor Injection and Field Injection, and when is each used?
Answer
Constructor Injection injects dependencies via the class constructor. It is preferred because it makes dependencies explicit and allows instantiating the class without a DI framework during testing. Field Injection is used when the class is instantiated by the Android platform framework (like Activities or Fragments). The DI framework injects dependencies into annotated fields after initialization.
Senior-Level Follow-up
Compare Dagger 2's compile-time graph validation with runtime service locator patterns.
Common Mistake
Instantiating singletons manually (e.g. using `object` singletons or static fields), which hardcodes dependencies and complicates unit testing.

6. Hilt (Jetpack DI implementation) DI

Pros and Cons

Pros: Built-in integration with Jetpack components, pre-defined components and lifecycles (Activity, Fragment, ViewModel), and simplified Dagger configuration.
Cons: Adds build-time overhead, and custom scopes require additional configuration.

Testability

Provides helper tools (like HiltAndroidTest) to swap dependencies in integration tests, allowing you to test UI components with fake data sources.

Scalability

Simplifies dependency configuration in multi-module projects, allowing modules to expose their dependencies automatically to the main app graph.

Real Production Examples

An Expedia application. The ViewModel constructor is annotated with @HiltViewModel, and Hilt automatically injects the repository dependency at runtime without requiring custom factory boilerplate.

Interview Questions

What is the role of @InstallIn annotation in Hilt modules?
Answer
The @InstallIn annotation tells Hilt which standard Dagger component the module's dependencies should be bound to. For example, @InstallIn(SingletonComponent::class) binds dependencies to the application-level lifecycle (singleton), while @InstallIn(ActivityComponent::class) limits dependencies to the lifecycle of individual activities.
Senior-Level Follow-up
Explain the difference between @Binds and @Provides in a Hilt module.
Common Mistake
Failing to annotate the custom Application class with @HiltAndroidApp, causing compilation errors or runtime injection failures.
