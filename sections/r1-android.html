1. Activity Lifecycle & State Restoration Edge Cases Framework

The Activity lifecycle undergoes complex state transitions. Since Android 9 (API 28), onSaveInstanceState() is guaranteed to be called after onStop().

Memory & Process Death: Calling finish() explicitly removes the activity instance and clears its state, signaling that the user has completed the flow. In contrast, system-initiated process death occurs when the OS reclaims resources under low memory conditions. In this case, the process is killed without calling onDestroy(), but the saved state bundle is preserved in the system-wide ActivityManagerService process memory to allow restoration. 

override fun onSaveInstanceState(outState: Bundle) {
    super.onSaveInstanceState(outState)
    // SavedState bundle is sent across Binder to ActivityManagerService
    outState.putString("booking_id", currentBookingId)
}
Interview Question
Explain the lifecycle differences between a user calling finish() and the system killing the process.
Answer
When finish() is called, the activity transitions through onPause, onStop, and onDestroy, releasing all associated memory and state. During system-initiated process death, the process is killed directly (often from the background state). The system preserves the state bundle returned by onSaveInstanceState() in system server memory, bypassing onDestroy(). When the user returns, the activity is recreated, receiving the saved bundle in onCreate().
Senior Follow-Up
Where is the saved state bundle stored during process death, and what are its size limitations?
Common Candidate Mistake
Saving large data structures (like search result lists) in onSaveInstanceState(), triggering a TransactionTooLargeException.

2. Fragment Lifecycle & viewLifecycleOwner Framework

A Fragment has two distinct lifecycles: the Fragment instance lifecycle and the Fragment view lifecycle. The view can be destroyed and recreated (for example, when a fragment is placed on the backstack) while the Fragment instance itself survives.

Memory Implications: Observing LiveData or Flow in a Fragment using the fragment's instance lifecycle (this) leads to memory leaks and duplicate emissions. Because the fragment instance survives view destruction, the observer reference remains active, preventing the view hierarchy from being garbage collected. Using viewLifecycleOwner ensures observers are automatically unregistered when the view is destroyed. 

override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    super.onViewCreated(view, savedInstanceState)
    
    // SAFE: viewLifecycleOwner scope cleans up observers on view destruction
    viewModel.hotelDetails.observe(viewLifecycleOwner) { details ->
        binding.hotelName.text = details.name
    }
}
Interview Question
Why does Fragment have viewLifecycleOwner, and what bugs occur if you use 'this' as the LifecycleOwner?
Answer
Fragment views can be destroyed (e.g., when replaced and placed on the backstack) while the fragment instance remains alive. If you observe LiveData using this, the observer reference is retained. When the fragment returns and recreates its view, a duplicate observer is registered, causing memory leaks and duplicate UI updates. Using viewLifecycleOwner binds observers to the view's lifecycle instead of the fragment instance.
Senior Follow-Up
How should you clean up ViewBinding references in onDestroyView() to prevent leaks?
Common Candidate Mistake
Failing to set the backing property of a ViewBinding reference to null in onDestroyView(), leaking the entire view hierarchy in backstack transitions.

3. ViewModel Survival Mechanics Framework

ViewModels survive configuration changes (such as screen rotations) by using a ViewModelStore. This store is retained by the host activity or fragment using a non-configuration instance mechanism (NonConfigurationInstances).

Memory & Lifecycle: When a configuration change occurs, the activity is destroyed, but the ViewModelStore instance is passed to the new activity instance via the Activity.retainNonConfigurationInstances() hook. The ViewModel is only destroyed and cleared (invoking onCleared()) when the host activity is finished or the fragment is removed. 

// Internals of how ViewModelStore obtains instances
val viewModelProvider = ViewModelProvider(this)
val myViewModel = viewModelProvider.get(MyViewModel::class.java)
// Resolves to: owner.getViewModelStore().get(key)
Interview Question
How does a ViewModel survive configuration changes but not process death?
Answer
During configuration changes, the activity is recreated, but the system passes the original ViewModelStore to the new instance via retainNonConfigurationInstances(). During process death, the host process is terminated, destroying the ViewModelStore and all ViewModels in memory. The system only retains the saved state bundle, which must be restored using a SavedStateHandle.
Senior Follow-Up
How does SavedStateHandle communicate changes back to the system server processes?
Common Candidate Mistake
Storing references to views, contexts, or activity instances inside a ViewModel, causing leaks when the activity is recreated.

4. Looper, MessageQueue, and IdleHandler Internals Threading

Android's main thread runs an infinite loop managed by a Looper, which processes events dispatched to its MessageQueue. The thread is kept alive without consuming 100% CPU because it blocks when the queue is empty using a native blocking mechanism (Linux `epoll`).

IdleHandler: An IdleHandler is a callback interface that executes tasks only when the message queue has processed all pending messages and is about to block for new ones. This is useful for executing low-priority tasks (such as pre-fetching data or reporting analytics) without blocking the startup path. 

// Registering an IdleHandler
Looper.myQueue().addIdleHandler {
    // Run lower priority work here
    reportDeferredAnalytics()
    false // Return false to run once, true to run on subsequent idle cycles
}
Interview Question
How does the main thread Looper stay active without blocking the CPU, and what is an IdleHandler?
Answer
The main thread Looper.loop() is an infinite loop that calls MessageQueue.next(). If no messages are pending, it blocks using native code via the Linux epoll_wait system call, suspending the thread until a new event is written to the file descriptor. An IdleHandler is a callback executed when the queue has completed its current work and is about to block, allowing tasks to run without delaying frame rendering.
Senior Follow-Up
What happens if an IdleHandler returns true instead of false?
Common Candidate Mistake
Running long-running or blocking work inside an IdleHandler, blocking the main thread and causing ANRs when a user interaction event arrives.

5. Handler Memory Leak Pattern and Fix Memory

A common source of leaks is declaring a non-static inner class subclass of Handler. Non-static inner classes retain an implicit reference to their enclosing class (such as an Activity).

Leak Mechanics: When a message is posted to a Handler, it is wrapped in a Message object. The message.target field holds a reference to the Handler. Since the Message is queued in the main thread's MessageQueue (which lives as long as the process), the Handler and its enclosing Activity are leaked until the message is processed. 

// LEAKY: Non-static inner class Handler
class BookingActivity : Activity() {
    private val handler = object : Handler(Looper.getMainLooper()) {
        override fun handleMessage(msg: Message) {
            // Implicitly references BookingActivity.this
        }
    }
}

// FIX: Static class (nested class) + WeakReference
class SafeBookingActivity : Activity() {
    private val handler = SafeHandler(this)
    
    private class SafeHandler(activity: SafeBookingActivity) : Handler(Looper.getMainLooper()) {
        private val activityRef = WeakReference(activity)
        
        override fun handleMessage(msg: Message) {
            val activity = activityRef.get()
            if (activity != null) {
                // Update UI safely
            }
        }
    }
}
Interview Question
Explain the memory reference path that causes a leak when using a non-static Handler.
Answer
The GC reference path is: Main Thread -> Looper -> MessageQueue -> Message -> target (Handler) -> outer class (Activity). Because the message retains a reference to the handler, and the handler retains an implicit reference to the activity, the activity cannot be garbage collected as long as the message is in the queue. The fix is to use a nested (static) handler class that references the activity via a WeakReference.
Senior Follow-Up
Does canceling messages in onDestroy() using handler.removeCallbacksAndMessages(null) prevent leaks?
Common Candidate Mistake
Assuming that because an activity's onDestroy() has been called, all handlers and runnables posted to the main thread are cleaned up automatically.

6. Binder IPC Limits & Mechanics System

Binder is the core Inter-Process Communication (IPC) mechanism in Android. It uses a shared memory buffer mapped between the calling process and the target process. This shared buffer has a **1MB transaction limit** per process, which is shared across all active Binder transactions.

One-way Calls & Death Recipients:

  • One-way Calls: Asynchronous, non-blocking calls. The caller process does not wait for a return value. These transactions have a smaller buffer limit (typically 64KB) to prevent a single process from exhausting the shared Binder buffer.
  • Death Recipient: A callback interface registered by a client process (using IBinder.linkToDeath) to receive a notification if the target service process terminates unexpectedly.
// Listening to process termination
val binder: IBinder = serviceConnection.getBinder()
val deathRecipient = IBinder.DeathRecipient {
    // Handle remote service crash
    reconnectToService()
}
binder.linkToDeath(deathRecipient, 0)
Interview Question
What is the Binder transaction buffer limit, and how do you handle remote service crashes?
Answer
The Binder transaction buffer limit is 1MB, shared across all active transactions for the process. For asynchronous one-way calls, the limit is lower (typically 64KB) to prevent buffer starvation. Service crashes are handled by registering an IBinder.DeathRecipient, which triggers a callback when the Binder channel is severed.
Senior Follow-Up
Why does a one-way Binder call return immediately even if the target service is busy?
Common Candidate Mistake
Passing large bitmaps or lists across Intents or bound services, causing a TransactionTooLargeException.

7. ART Runtime Compilation Tiers System

The Android Runtime (ART) uses a hybrid compilation strategy:

  • Interpreter: Executed immediately upon app startup to parse Dex code, avoiding compilation overhead.
  • Just-In-Time (JIT): Profiles hot code paths during execution, compiling them to native machine code on the fly to improve runtime performance.
  • Ahead-Of-Time (AOT): Profiles from JIT execution are stored in profile files. When the device is idle and charging, the background compilation daemon (dex2oat) compiles these hot code paths to native code, reducing execution overhead on subsequent runs.

Memory Implications: AOT-compiled code resides in shared memory segments, reducing execution overhead and startup times. However, excessive AOT compilation increases storage consumption.

Interview Question
Explain the relationship between the JIT compiler, profiles, and background AOT compilation in ART.
Answer
ART starts by interpreting the dex bytecode. As the app runs, the JIT compiler monitors execution and identifies "hot" methods, compiling them to native code in memory and saving their metadata to profile files. When the device is idle and charging, the system background compiler uses these profiles to perform target AOT compilation for those hot paths, saving storage space while optimizing future startup times.
Senior Follow-Up
How does ART profile-guided compilation improve application memory footprint?
Common Candidate Mistake
Assuming that because an app starts, all code is compiled to native machine code immediately (it runs via the interpreter/JIT hybrid).

8. Garbage Collection and GC Pauses Memory

ART uses a generational garbage collection (GC) scheme. Objects are allocated in the **Young Generation** space. Active objects are promoted to the **Old Generation** space if they survive multiple collection cycles.

GC Pauses & Triggers: ART performs concurrent collections to minimize stop-the-world pauses. However, GC pauses can still occur if the allocation rate exceeds the garbage collection rate, or when moving objects during compaction to resolve memory fragmentation. Frequent allocations in tight loops (such as inside custom drawing loops) trigger GC thrashing, resulting in dropped frames.

Interview Question
What triggers stop-the-world pauses in ART's Generational GC, and how does it affect frame rates?
Answer
Stop-the-world pauses are triggered during the marking phase (to capture a consistent state snapshot of the object reference tree) and the compaction phase (when moving objects to defragment memory). If these pauses exceed the 16.6ms frame rendering threshold, they block the UI thread, resulting in dropped frames (jank).
Senior Follow-Up
How does the ART concurrent copying collector reduce pauses compared to the older Dalvik mark-sweep collector?
Common Candidate Mistake
Creating short-lived objects (such as paint objects, iterators, or formatted strings) inside onDraw(), triggering GC churn and UI lag.

9. Memory Leaks Catalogue Memory

Common Android memory leak patterns and their corresponding fixes:

  • Static Context References: Storing a reference to an Activity context in a static field leaks the Activity after destruction. Fix: Use context.applicationContext or avoid static fields.
  • Anonymous Runnable: Registering an anonymous runnable with a background thread or executor service leaks the enclosing Activity. Fix: Cancel tasks in lifecycle teardown methods.
  • Unregistered Listeners: Registering callback listeners (such as location services or custom bus systems) without unregistering them leaks the observer. Fix: Unregister location listeners in onStop() or onDestroy().
// LEAK: Anonymous listener callback leak
class LocationActivity : Activity() {
    override fun onStart() {
        super.onStart()
        LocationManager.registerListener(object : LocationListener {
            override fun onLocationChanged(loc: Location) {
                updateUI(loc)
            }
        })
    }
    // Leak if not unregistered in onStop()
}
Interview Question
List three common memory leak patterns in Android and their mitigation strategies.
Answer
1. Static context reference: Avoid holding Activity references in static fields; use the Application context instead. 2. Anonymous callback tasks: Cancel pending tasks (such as handlers or coroutines) in lifecycle teardown methods. 3. Unregistered listeners: Unregister listeners (like location updates or custom event bus systems) in the corresponding lifecycle teardown method (e.g. unregister in onStop() if registered in onStart()).
Senior Follow-Up
How does the LeakCanary library detect memory leaks at runtime?
Common Candidate Mistake
Assuming that declaring a listener as a lambda instead of an anonymous class avoids leaks. (Lambdas that access outer class properties compile to classes that retain a reference to the enclosing object).

10. App Startup Optimization & Perfetto Performance

App startup modes are categorized by the state of the process:

  • Cold Start: The system creates the application process from scratch, initializes the class loader, and instantiates components.
  • Warm Start: The process is already running, but the activity must be recreated from scratch (e.g., after the activity is killed in the background).
  • Hot Start: The activity is still in memory and is brought to the foreground, requiring only view traversal and rendering.

Optimization: Measuring startup bottlenecks is done using **Perfetto** tracing. Content Provider initialization runs before the Application.onCreate() method, which can delay cold startup. The **Jetpack App Startup** library addresses this by grouping initializers into a single shared Content Provider, reducing overhead.

Interview Question
Explain the differences between cold, warm, and hot starts, and how you optimize Content Provider initialization overhead.
Answer
Cold starts involve process initialization and class loading, warm starts recreate the activity in an existing process, and hot starts simply bring the activity to the foreground. Content Providers initialize before Application.onCreate(), which can delay startup. To optimize this, use the Jetpack App Startup library to bundle multiple initializers into a single shared provider, reducing initialization overhead.
Senior Follow-Up
How do baseline profiles help optimize app startup times in the Android Runtime?
Common Candidate Mistake
Initializing third-party libraries synchronously inside Application.onCreate() instead of deferring or executing them on background threads.

11. Process Death State Restoration Strategy Framework

To handle system-initiated process death gracefully, Expedia apps implement a three-tiered state restoration strategy:

  • SavedStateHandle: Retains transient UI state (such as text input or selected filters) across process death. This state is stored in the system server's Binder memory.
  • Room/Local Database: Serves as the single source of truth, persisting large user datasets (such as cached search results) that exceed Binder size limits.
  • Network: Re-fetches fresh data if the cached local database records have expired or are missing.
class SearchViewModel(
    private val savedStateHandle: SavedStateHandle,
    private val repository: HotelRepository
) : ViewModel() {
    
    // UI state flows from SavedStateHandle, surviving process death
    val hotelQuery = savedStateHandle.getStateFlow("query", "Expedia Hotel")
    
    fun updateQuery(newQuery: String) {
        savedStateHandle["query"] = newQuery
    }
}
Interview Question
Design a state restoration strategy that handles process death for a hotel booking checkout screen.
Answer
Combine SavedStateHandle and a local Room database. Store transient inputs (such as names or selections) in SavedStateHandle, and persist heavy booking data in the Room database. On recreation, initialize the ViewModel using the ID retrieved from SavedStateHandle, and load the booking details from the local Room database to restore the screen state.
Senior Follow-Up
What is the difference between ViewModel state preservation and SavedStateHandle state preservation?
Common Candidate Mistake
Assuming that because ViewModels survive rotation, they also survive process death. (ViewModels are cleared when the host process is terminated).

12. WorkManager Internals Framework

WorkManager executes deferrable, guaranteed background work. It chooses the underlying scheduler based on the device's API level: JobScheduler on API 23+, and a fallback scheduler using AlarmManager and BroadcastReceiver on older API levels.

Constraint Matching: WorkManager monitors device constraints (such as network availability or charging state) before executing a task. Work tasks can be chained sequentially or in parallel, with intermediate data passed between tasks using key-value bundles. 

val uploadRequest = OneTimeWorkRequestBuilder()
    .setConstraints(Constraints.Builder()
        .setRequiredNetworkType(NetworkType.CONNECTED)
        .setRequiresCharging(true)
        .build())
    .build()

WorkManager.getInstance(context).enqueue(uploadRequest)
Interview Question
How does WorkManager select its scheduling mechanism, and how does it guarantee task execution?
Answer
WorkManager delegates execution to JobScheduler on API 23+ and falls back to a custom combination of AlarmManager and broadcast receivers on older APIs. It guarantees execution by persisting task metadata in an internal SQLite database, allowing tasks to resume even if the device restarts.
Senior Follow-Up
How does WorkManager handle task execution failures and configure retry backoff policies?
Common Candidate Mistake
Using WorkManager to run immediate, short-lived tasks that must execute on the current thread, rather than using standard coroutines.

13. Foreground Service Requirements post Android 14 System

Starting in Android 14 (API 34), applications must declare specific **foreground service types** in their manifest. You must also request the appropriate type-specific permission (e.g. FOREGROUND_SERVICE_LOCATION) before launching the service.

System Restrictions: The OS terminates foreground services that fail to declare a type, lack the appropriate runtime permissions, or violate background execution limits. 

<!-- Manifest Declaration requirement -->
<service
    android:name=".HotelSyncService"
    android:foregroundServiceType="dataSync"
    android:exported="false" />
Interview Question
What changes were introduced for Foreground Services in Android 14, and what steps are required to launch one?
Answer
Android 14 requires declaring specific foreground service types in the manifest. To launch a foreground service, you must: 1. Request the general foreground service permission. 2. Request the type-specific runtime permission (if required). 3. Start the service. 4. Call startForeground(), passing the matching type parameter within 10 seconds of startup.
Senior Follow-Up
Under what conditions does launching a foreground service trigger a BackgroundServiceStartNotAllowedException?
Common Candidate Mistake
Failing to declare the service type in the manifest or attempting to start a foreground service from the background on newer Android versions.
