Section 5: Android Platform & Internals

⏱ 30 min read ✓ Part 1 Core

Low-level Android OS architectures, core framework lifecycle states, thread dispatch loop schedulers, Binder IPC mapping, garbage collection compaction pauses, and background task frameworks.

1. Activity Lifecycle & State Restoration Framework

What happens internally

The OS manages activities using the ActivityTaskManagerService (ATMS) running in the system server process. State transitions are dispatched to the application process via binder IPC, where the ActivityThread schedules the callbacks. After API 28, onSaveInstanceState() is guaranteed to be called after onStop(), serializing critical state to be kept in ATMS memory.

Threading Model

All lifecycle callbacks run synchronously on the application's Main Thread. Blocking these callbacks delays the next transition and stalls UI frame rendering.

Performance Impact

Performing heavy operations (like DB access or serialization) during lifecycle hooks causes frame drops (jank) and can trigger ANRs if a transition takes longer than 5 seconds.

Common Production Issues

System-initiated process death (low memory) re-instantiates the activity using the saved bundle, but static variables are re-initialized to null, leading to NullPointerException if not restored properly.

Interview Questions

What is the difference between onDestroy() called during finish() vs during process death?
Answer
When finish() is invoked, the activity undergoes a complete teardown, calling onDestroy(), and its state is removed from memory. During low-memory process death, the process is killed directly in the background without calling onDestroy(), but its saved state is retained by the system server process. When the user returns, the activity is rebuilt, receiving the saved state bundle in onCreate().
Senior-Level Follow-up
Where is the saved state bundle stored during process death, and what is its transaction size limit?
Common Mistake
Saving heavy bitmaps or lists in the saved state, which triggers a `TransactionTooLargeException`.

2. Fragment Lifecycle & View Lifecycle Framework

What happens internally

Fragments contain two distinct lifecycles: the fragment instance lifecycle and the fragment view lifecycle. When a fragment is replaced but placed on the backstack, its view is destroyed, but the fragment instance survives in memory.

Threading Model

All transactions are managed by the FragmentManager on the Main Thread. Fragment transactions can be asynchronous (e.g. commit()) or synchronous (e.g. commitNow()).

Performance Impact

Retaining views inside inactive fragment instances prevents GC of the entire view hierarchy, leading to substantial memory overhead.

Common Production Issues

Registering lifecycle-aware observers (like Flows or LiveData) using the fragment instance (this) instead of the viewLifecycleOwner. Because the fragment instance survives view destruction, the observers accumulate, causing memory leaks and duplicate triggers.

Interview Questions

Why does Fragment have two separate Lifecycles?
Answer
Fragment views can be destroyed (e.g., when replaced on the backstack) while the fragment instance remains alive. To prevent memory leaks of the view tree, the view lifecycle is destroyed while the instance lifecycle remains active. Observers must bind to the view's lifecycle (viewLifecycleOwner) to automatically clean up.
Senior-Level Follow-up
How do you manage ViewBinding cleanup inside onDestroyView()?
Common Mistake
Holding a non-null reference to view binding in onDestroyView(), which leaks the whole view hierarchy.

3. ViewModel Internals and Lifecycle Framework

What happens internally

A ViewModelStore holds ViewModel instances. The store is retained across configuration changes by registering it inside NonConfigurationInstances. When the activity rotates, the system passes this store from the old activity instance to the new activity instance, preserving state.

Threading Model

Instance lookup and initialization occur on the Main Thread. Network or database actions launched from within ViewModels (via viewModelScope) should be offloaded to background dispatchers.

Performance Impact

Reduces UI recreation latency. However, holding strong references to UI elements (like activities, views, or contexts) inside the ViewModel causes severe memory leaks across rotations.

Common Production Issues

Assuming ViewModels survive process death. When the OS terminates the app process, the ViewModel store is destroyed. To survive process death, you must use SavedStateHandle.

Interview Questions

How does ViewModel survive configuration changes but not process death?
Answer
During rotation, the ViewModelStore is passed to the new Activity instance using retainNonConfigurationInstances(), preserving the same ViewModel in memory. During process death, the VM host process is destroyed, wiping all memory. The system server only preserves a small serialized bundle, which is delivered via SavedStateHandle on recreation.
Senior-Level Follow-up
How does the SavedStateFlow integration in SavedStateHandle update state under the hood?
Common Mistake
Passing Activity Context to a ViewModel constructor instead of Application Context.

4. Process Lifecycle (Foreground, Visible, Service, Cached) System

What happens internally

Android manages app processes dynamically. Based on component states, the OS assigns an OOM adjustment score (oom_adj) representing process importance:

  • Foreground (oom_adj = 0): Host of the active screen or a foreground service.
  • Visible: Partially obscured (like a dialog background).
  • Service: Hosting active background service.
  • Cached: Retained in background for quick resume. Killed first if memory is low.

Threading Model

Managed by the Low Memory Killer Daemon (lmkd) in kernel space, executing kills asynchronously based on oom_adj thresholds.

Performance Impact

Limits memory footprints dynamically, ensuring background apps do not starve the active user interface.

Common Production Issues

Background tasks that run without raising the process priority get terminated abruptly by the LMK when the user leaves the application.

Interview Questions

What is the oom_adj score, and how does the Low Memory Killer use it?
Answer
The oom_adj is a score from -1000 to 1000 assigned to every running process by the Activity Manager. The lower the score, the more important the process is to the user. When physical RAM is depleted, the Low Memory Killer daemon reads these scores and terminates processes starting with the highest scores (e.g. Cached processes) to reclaim memory pages.
Senior-Level Follow-up
How does declaring a service as a foreground service change its oom_adj score?
Common Mistake
Assuming that background thread operations keep an app safe from being killed when the user puts the app in the background.

5. Application Lifecycle and Callbacks Framework

What happens internally

The Application instance is instantiated by the ActivityThread immediately after the process forks from Zygote. It lives as long as the process is alive. It receives system events via callbacks (such as ComponentCallbacks2 for memory trim events).

Threading Model

Initialization (onCreate) runs synchronously on the Main Thread. Any blocking initialization here delays the initial cold startup of the application.

Performance Impact

Synchronous SDK initializations inside Application.onCreate() delay the first frame draw, inflating Cold Start time metrics.

Common Production Issues

Leaking contexts by registering global static variables or singletons that keep strong references to individual activity contexts instead of the application context.

Interview Questions

How can you monitor Activity lifecycle states globally across your entire app?
Answer
You can register a global callback in your Application class using registerActivityLifecycleCallbacks(Application.ActivityLifecycleCallbacks). This receives events for every Activity state transition, allowing centralized logging, tracking, or state management.
Senior-Level Follow-up
Explain how the onTrimMemory() callback helps prevent your process from being killed by the LMK.
Common Mistake
Initializing third-party libraries synchronously in `Application.onCreate` without deferring or offloading them to background threads.

6. App Startup (Jetpack Startup & Initialization Paths) Performance

What happens internally

During cold start, traditional libraries initialize themselves automatically using independent ContentProvider classes. Content Providers are initialized before `Application.onCreate`. The Jetpack App Startup library replaces these multiple providers with a single shared provider, executing initializations in a defined dependency tree.

Threading Model

By default, initializers execute on the Main Thread. However, you can configure tasks to run asynchronously on background threads using custom executors.

Performance Impact

Reduces cold startup overhead by avoiding the creation of separate content provider components, which require expensive binder transactions during startup.

Common Production Issues

Libraries initializing synchronously during startup that block the UI thread, causing frames to freeze before the app UI can display.

Interview Questions

Why are ContentProviders bad for cold start performance, and how does App Startup fix this?
Answer
Each ContentProvider declared in the manifest requires process startup and binder mapping, adding several milliseconds of overhead. If you have 5-10 third-party libraries using providers to self-initialize, the cumulative delay is substantial. The App Startup library defines a single ContentProvider, wrapping the initializations in standard classes that can run sequentially or in parallel under a single entry point.
Senior-Level Follow-up
How do baseline profiles coordinate with App Startup classes to precompile code paths?
Common Mistake
Assuming that App Startup automatically runs all initializations on a background thread.

7. Looper & MessageQueue Internals Threading

What happens internally

A thread has at most one Looper. Under the hood, it uses ThreadLocal storage. The Looper manages a MessageQueue containing messages and runnables. If the queue is empty, the thread blocks using Linux's epoll mechanism, waiting on a file descriptor until a new message is pushed.

Threading Model

Processes tasks sequentially. Message queue synchronization uses a native lock-free queue that allows background threads to safely post events to the loop.

Performance Impact

Ensures that the Main Thread remains responsive by executing tasks in order without polling. Heavy tasks in the queue cause frame rendering delays.

Common Production Issues

Posting too many high-frequency messages (like tracking scroll pixels) can saturate the message queue, delaying rendering and input events.

Interview Questions

Explain how epoll keeps the Main Thread alive without consuming CPU resources.
Answer
The MessageQueue uses native code to invoke epoll_wait on a Linux file descriptor. If there are no messages, the OS puts the thread to sleep, freeing CPU cores. When another thread posts a message, it writes a byte to the file descriptor (via `eventfd`), waking the thread to process the new message. This eliminates active polling.
Senior-Level Follow-up
What is a Sync Barrier, and how does it prioritize rendering messages inside the queue?
Common Mistake
Creating multiple Loopers on the Main Thread (which causes immediate crashes; only one Looper is permitted per thread).

8. Handler & HandlerThread Threading Model Threading

What happens internally

A Handler is bound to a specific Looper. It acts as an interface to push events into the message queue and process them when they exit. A HandlerThread is a helper thread subclass that automatically sets up a Looper and MessageQueue upon starting, providing a background message loop.

Threading Model

Thread-safe message passing. Background threads can call handler.sendMessage(), safely queuing data onto the handler's target thread context.

Performance Impact

HandlerThreads provide a clean, sequential background queue for tasks like Bluetooth sync or sensor processing, offloading work from the UI thread.

Common Production Issues

Memory leaks caused by non-static Handlers holding implicit references to activities, which keeps the activity from being garbage collected if messages are pending.

Interview Questions

Why is HandlerThread useful, and how do you prevent Handler leaks?
Answer
A HandlerThread provides a background thread with an active message loop, which is useful for processing sequential background tasks. To prevent leaks, handlers should be declared as static/nested classes and use a WeakReference to reference the host class (like an Activity). Any pending callbacks should also be cleared in `onDestroy()`.
Senior-Level Follow-up
What happens if you instantiate a Handler without passing a Looper parameter in its constructor?
Common Mistake
Instantiating a Handler inside a background thread without calling Looper.prepare() (results in a runtime crash).

9. ART Runtime (Interpreted, JIT, AOT Compilation) System

What happens internally

The Android Runtime (ART) executes compiled Dex code. It uses a tiered compilation system:

  • Interpreter: Runs immediately at startup without compilation overhead.
  • JIT (Just-In-Time): Profiles code during execution, compiling hot methods to native code in memory.
  • AOT (Ahead-Of-Time): When the device is idle and charging, the background compiler uses stored JIT profiles to pre-compile hot methods to native code.

Threading Model

JIT profiling and compilation run on low-priority background compiler threads, preventing execution stutter on the main application thread.

Performance Impact

Balances fast app installation times with high runtime execution speeds. AOT-compiled code runs at native speed, reducing CPU usage and startup times.

Common Production Issues

App performance differences on older Android devices where AOT compilation profiles are missing or corrupted, falling back to slow interpreter paths.

Interview Questions

Explain the benefits of ART compilation profiles.
Answer
ART compilation profiles store information about which methods are frequently executed (hot code paths) during JIT execution. Instead of compiling the entire app to native code (which takes time and consumes storage), the AOT compiler compiles only these hot methods during idle maintenance cycles, optimizing startup performance without wasting storage.
Senior-Level Follow-up
How does ART's bytecode execution differ from Dalvik's register-based execution model?
Common Mistake
Assuming that because Java code is compiled to dex, it is AOT compiled on installation for all device configurations.

10. Garbage Collection Generational Mechanics Memory

What happens internally

ART uses a generational collection scheme. Newly allocated objects are placed in the Young Gen space. If they survive collections, they are moved to the Old Gen space. It uses a Concurrent Copying (CC) collector to move objects during compaction, reducing memory fragmentation.

Threading Model

Garbage collection runs concurrently with application execution. Brief stop-the-world pauses occur when marking roots and finalizing object movements.

Performance Impact

Compaction resolves memory fragmentation, avoiding OutOfMemory errors. However, triggering frequent GCs (GC churn) leads to stop-the-world pauses, resulting in dropped UI frames.

Common Production Issues

Creating temporary objects (like Paint instances or custom strings) inside onDraw() or high-frequency loops. This triggers GC churn and causes UI stutter.

Interview Questions

How does Generational GC optimize object reclamation, and what causes GC pauses?
Answer
Generational GC assumes most objects die young. Collecting the Young Gen space requires searching fewer objects, making it fast. GC pauses occur when marking root references and when moving objects to defragment memory. If these pauses exceed 16.6ms, they delay frame rendering, causing UI jank.
Senior-Level Follow-up
How does the onTrimMemory() callback help the application preemptively release references before the OS triggers memory compaction?
Common Mistake
Assuming that calling `System.gc()` forces immediate cleanup without any performance penalty.

11. Memory Leaks (Detection, References, Avoidance) Memory

What happens internally

A memory leak occurs when an object is no longer needed by the app, but a reference path to it remains active in the garbage collection root tree. This prevents the GC from reclaiming the object's memory.

Threading Model

Memory analysis tools (like LeakCanary) run in background processes, parsing heap dumps (.hprof files) asynchronously using reference trackers.

Performance Impact

Accumulating leaks reduces available heap memory, triggering frequent GCs and eventually causing process termination due to OutOfMemoryError.

Common Production Issues

Holding strong references to Activities inside singletons, static fields, custom views, or unregistered lifecycle callbacks (like location services).

Interview Questions

How do you detect memory leaks, and how does LeakCanary work under the hood?
Answer
Memory leaks are detected by analyzing heap references. LeakCanary registers activity/fragment lifecycle listeners. When a component is destroyed, it wraps it in a WeakReference with a key. After a short delay, it forces a GC check. If the reference is not cleared, LeakCanary dumps the heap, parses the file, and traces the GC root reference path to identify the leak.
Senior-Level Follow-up
What is the difference between WeakReference, SoftReference, and PhantomReference in Java?
Common Mistake
Assuming that declaring a static object null in onDestroy() immediately executes garbage collection.

12. Binder IPC & AIDL Architecture System

What happens internally

Binder is Android's IPC implementation. It uses a kernel driver (/dev/binder). Communication works by memory mapping (mmap) a shared buffer space between process address spaces. AIDL (Android Interface Definition Language) compiles interfaces into stub and proxy classes that serialize (marshal) objects across this boundary.

Threading Model

Binder transactions are synchronous by default. The calling thread blocks until the target process returns. Async transactions (one-way calls) do not block. Target calls execute on a pool of background binder threads in the receiver process.

Performance Impact

Binder is fast because it requires only a single memory copy (from caller's user space to target's kernel-mapped user space). However, transactions have a strict 1MB size limit per process.

Common Production Issues

Passing too much data (like large intent extras or lists) across IPC boundaries, which triggers `TransactionTooLargeException` and crashes the application.

Interview Questions

Explain the binder memory copy reduction compared to traditional Linux sockets.
Answer
Traditional socket communication requires two copies: caller user-space -> kernel space -> receiver user-space. Binder maps a section of physical memory to both kernel space and the receiver process's address space. The caller copies data once from its address space to this kernel-mapped block, and the receiver reads it directly, reducing copy overhead by 50%.
Senior-Level Follow-up
How does the oneway keyword in AIDL affect thread blocking on the calling side?
Common Mistake
Performing synchronous binder IPC calls on the main thread during UI interactions.

13. PendingIntent Security and Flag Mechanics System

What happens internally

A PendingIntent is a token that you pass to another application (like the system notification manager or alarm manager), granting it permission to execute a pre-defined intent on your app's behalf with your app's security privileges.

Threading Model

Thread-agnostic. The execution of the intent runs asynchronously in your application's process context once triggered by the system process.

Performance Impact

Uses negligible memory. The target intent definition is stored securely by the system server (ActivityManagerService) as a token.

Common Production Issues

Declaring a PendingIntent as mutable (FLAG_MUTABLE) without defining explicit targets. This allows receiver apps to modify the intent fields, executing arbitrary components inside your application.

Interview Questions

What are PendingIntent flags, and why is FLAG_IMMUTABLE recommended?
Answer
PendingIntent flags define its lifecycle and mutability. FLAG_IMMUTABLE prevents other applications from modifying the underlying intent. If mutable, a receiver application could mutate the intent's action or target component, exploiting your app's privileges. FLAG_IMMUTABLE is mandatory for security on newer Android versions unless mutability is explicitly required.
Senior-Level Follow-up
Under what circumstances is FLAG_UPDATE_CURRENT required when updating notification actions?
Common Mistake
Failing to specify either mutable or immutable flags on API 31+, which triggers immediate startup crashes.

14. Notification Channels and System Service Posting System

What happens internally

Notifications are managed by the system process NotificationManagerService. Apps post notifications using NotificationManager.notify(). Since Android 8 (API 26), all notifications must be assigned to a NotificationChannel, which defines user-facing sound, vibration, and priority settings.

Threading Model

Posting notifications is synchronous. The request is serialized and sent via binder IPC to the system server process for rendering on the screen.

Performance Impact

Negligible, but updating notifications at a very high frequency (e.g., inside progress loops) saturates the system server's binder queue, slowing down UI rendering.

Common Production Issues

Attempting to post notifications without declaring a valid channel ID. This is ignored by the OS on newer versions, or crashes the app.

Interview Questions

Why did Android introduce Notification Channels, and can they be modified by the app after creation?
Answer
Notification Channels were introduced to give users granular control over notifications. Once created, the channel settings (such as importance, sound, or vibration) are managed by the system. The application *cannot* modify these settings programmatically. If changes are needed, the app must create a new channel with a different ID.
Senior-Level Follow-up
How do notification action buttons execute background logic without bringing the app to the foreground?
Common Mistake
Creating a new NotificationChannel every time you post a notification (they should be registered once during application startup).

15. WorkManager Internals & SQLite Persistence Framework

What happens internally

WorkManager manages persistent background tasks. When work is enqueued, it writes a task description and execution state to an internal SQLite database. It delegates execution scheduling to JobScheduler on API 23+, and uses an internal broadcast receiver scheduler on older APIs.

Threading Model

Runs tasks asynchronously. By default, task execution occurs on a background worker thread. When the task completes, it updates the SQLite state before exiting.

Performance Impact

Guarantees execution even if the process or device restarts. Writing to the SQLite database adds small disk I/O overhead, so it is not suited for high-speed loops.

Common Production Issues

Enqueuing duplicate unique workers without specifying a conflict resolution policy (like `KEEP` or `REPLACE`), resulting in redundant background tasks.

Interview Questions

How does WorkManager guarantee task execution even across system restarts?
Answer
WorkManager persists all task requests and execution states in a local SQLite database (using Room). When the device reboots, a startup receiver reads this database and schedules any pending tasks back into the OS-level JobScheduler.
Senior-Level Follow-up
How does WorkManager handle task cancellation when constraints (such as network connectivity) are lost during execution?
Common Mistake
Using WorkManager for immediate, in-memory operations that must run synchronously (use Coroutines instead).

16. Services vs WorkManager Framework

What happens internally

A Service is a framework component representing background execution. Standard services run inside the host application process. WorkManager is a library wrapper that manages background tasks, using services under the hood only when persistent background scheduling is required.

Threading Model

Standard Services run on the Main Thread. WorkManager runs tasks asynchronously on background executor threads.

Performance Impact

Standard Services are lightweight but offer no scheduling optimization. WorkManager manages execution scheduling based on battery constraints, optimizing resource usage.

Common Production Issues

Using background services to execute long-running tasks. Background services are killed by the OS when the app goes to the background. Long-running tasks should use WorkManager or Foreground Services instead.

Interview Questions

When should you use a Service instead of WorkManager?
Answer
Use standard Services (like Bound Services) for immediate, non-persistent, local interactions within the app (such as music playback). Use WorkManager for deferrable, persistent tasks that require execution guarantees (like syncing data to a server).
Senior-Level Follow-up
How does the memory footprint of a bound service compare to a WorkManager job executor?
Common Mistake
Thinking that starting a service automatically spawns a background thread.

17. Foreground Services (Android 14 requirements) System

What happens internally

Foreground Services perform user-noticeable operations. The service must display a persistent notification to the user. Starting in Android 14, apps must declare specific foreground service types (e.g. `dataSync`, `location`) in their manifest, along with type-specific permissions.

Threading Model

Runs on the Main Thread. Any network requests or calculations must be offloaded to background threads manually.

Performance Impact

Elevates the application's process priority to Foreground (oom_adj = 0), preventing the Low Memory Killer from terminating the process.

Common Production Issues

Starting a foreground service from the background on newer Android versions. This is blocked by the OS, triggering a `BackgroundServiceStartNotAllowedException`.

Interview Questions

What are the manifest declaration rules for Foreground Services in Android 14?
Answer
In Android 14, you must declare the foreground service in the manifest, specify its foreground service type using the android:foregroundServiceType attribute, and request the type-specific permission (e.g., FOREGROUND_SERVICE_DATA_SYNC) in the manifest.
Senior-Level Follow-up
Under what conditions can you bypass the background service start restriction?
Common Mistake
Failing to call startForeground() within 10 seconds of starting a foreground service, causing the OS to crash the application.

18. ANR (Application Not Responding) Causes and Profiling Performance

What happens internally

An ANR is triggered when the Main Thread Looper takes too long to process an event. The system displays an ANR dialog after 5 seconds of input timeout or 200 seconds of broadcast receiver timeout. When triggered, the OS dumps the thread stacks of all active processes to a `traces.txt` file.

Threading Model

Triggered when background threads block the Main Thread (e.g. through locks or synchronized blocks) or when the Main Thread itself performs blocking I/O.

Performance Impact

Stops all user interactions immediately, degrading user experience and app ratings.

Common Production Issues

Executing database operations synchronously, parsing large payloads on the main thread, or deadlocks between the main thread and background threads.

Interview Questions

How do you debug an ANR using the traces.txt file?
Answer
When an ANR occurs, the system writes the stack traces of all threads to /data/anr/traces.txt. You open this file, locate your application process, find the "main" thread, and examine its call stack to identify the blocking method (such as database read or thread synchronization lock).
Senior-Level Follow-up
How does StrictMode help catch ANR bottlenecks during development?
Common Mistake
Assuming that ANRs are only caused by infinite loops, ignoring blocks caused by database queries or network tasks.
