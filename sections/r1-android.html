Section 5: Android Platform & Internals

⏱ 12 min read ✓ Part 1 Core

Consolidated, high-yield cheat sheet covering Android framework internals, lifecycles, IPC, ART compilers, background execution, and profiling tools.

1. Activity Lifecycle & State Restoration Framework

Quick Cheat Sheet

  • Lifecycle ATMS: The OS coordinates lifecycle changes in the system server process via `ActivityTaskManagerService` (ATMS), communicating via Binder IPC with the app's `ActivityThread`.
  • Process Death: Under low memory, the OS kills background processes without calling `onDestroy()`. It preserves the serialized state bundle returned by onSaveInstanceState() in system server memory to allow restoration when the user returns.
Explain the lifecycle differences between finish() and system-initiated process death.
Answer
When a user calls finish(), the activity transitions through onPause, onStop, and onDestroy, releasing all associated memory and state. During low-memory process death, the process is killed directly in the background without calling onDestroy(). The system preserves the state bundle returned by onSaveInstanceState() in system server memory. When the user returns, the activity is recreated, receiving the saved bundle in onCreate().
Senior-Level Follow-up
Where is the saved state bundle stored during process death, and what is its transaction size limit? (Stored in system server process memory. The limit is 1MB shared across all active binder transactions).
Common Mistake
Saving heavy lists or bitmaps in the saved state, which triggers a `TransactionTooLargeException`.

2. Fragment Lifecycle & viewLifecycleOwner Framework

Quick Cheat Sheet

  • Two Lifecycles: A Fragment has a fragment instance lifecycle and a fragment view lifecycle. The view can be destroyed and recreated (such as when placed on the backstack) while the fragment instance itself remains in memory.
  • Memory Leak Risk: Registering observers inside a Fragment using the instance (this) as the owner causes leaks, as the reference is retained even after the view is destroyed. Always use viewLifecycleOwner.
Why does Fragment have two separate Lifecycles?
Answer
Fragment views can be destroyed (e.g. when replaced on the backstack) while the fragment instance remains alive. To prevent memory leaks of the view tree, the view lifecycle is destroyed while the instance lifecycle remains active. Observers must bind to the view's lifecycle (viewLifecycleOwner) to automatically clean up.
Senior-Level Follow-up
How do you manage ViewBinding cleanup inside onDestroyView()? (Set the backing property reference to null).
Common Mistake
Observing LiveData or Flows in onViewCreated using `this` instead of `viewLifecycleOwner`.

3. ViewModel Internals (ViewModelStore retention) Framework

Quick Cheat Sheet

  • ViewModelStore: Holds ViewModel instances. Retained across configuration changes by passing it to the new activity instance via the retainNonConfigurationInstances() hook.
  • Process Death: ViewModels do **not** survive process death. When the process is terminated, the VM store is wiped. Use SavedStateHandle to restore state.
How does ViewModel survive configuration changes but not process death?
Answer
During configuration changes (like rotation), the activity is recreated, but the system passes the original ViewModelStore to the new Activity instance using retainNonConfigurationInstances(). During process death, the host process is terminated, destroying all memory. The system only retains the saved state bundle, which must be restored using a SavedStateHandle.
Senior-Level Follow-up
How does SavedStateHandle communicate changes back to the system server processes?
Common Mistake
Storing references to views, contexts, or activity instances inside a ViewModel, causing leaks when the activity is recreated.

4. Looper, MessageQueue, and Handler thread loop Threading

Quick Cheat Sheet

  • Looper: Runs an infinite loop on a thread (using ThreadLocal storage), reading tasks from its MessageQueue. Uses Linux's epoll to sleep when the queue is empty.
  • Handler: Interacts with the Looper's message queue to post runnables and messages from background threads safely to the target thread context.
How does the main thread Looper stay active without consuming 100% CPU when idle?
Answer
The main thread Looper.loop() is an infinite loop that calls MessageQueue.next(). If no messages are pending, it blocks using native code via the Linux epoll_wait system call, putting the thread to sleep. When a new event or message is posted to the queue, the OS wakes up the thread by writing to the event file descriptor, eliminating busy polling.
Senior-Level Follow-up
What is an IdleHandler and when should you use it? (A callback executed when the queue has completed its current work and is about to block, useful for low-priority tasks).
Common Mistake
Instantiating a Handler inside a background thread without calling Looper.prepare() first, causing runtime crashes.

5. Memory Leaks (Common patterns and detection via LeakCanary) Memory

Quick Cheat Sheet

  • Definition: Occurs when an object is no longer needed but a reference path to it remains active in the GC root tree, preventing reclamation.
  • Common Patterns: Static references to Context, anonymous Handler runnables, unregistered receivers or listeners.
  • LeakCanary: Registers activity lifecycle listeners. When destroyed, it wraps references in a WeakReference with a key. If the reference is not cleared after a GC check, LeakCanary dumps the heap, parses the file, and traces the GC root reference path.
Explain the memory reference path that leaks an Activity when using a non-static inner Handler class.
Answer
The GC reference path is: Main Thread -> Looper -> MessageQueue -> Message -> target (Handler) -> outer class (Activity). Because the message in the queue retains a reference to the handler, and the handler retains an implicit reference to the activity, the activity cannot be garbage collected until all messages are processed. Declare the Handler class as static/nested with a WeakReference to fix.
Senior-Level Follow-up
What is the difference between WeakReference, SoftReference, and PhantomReference in Java?
Common Mistake
Assuming that declaring a static object null in onDestroy() immediately executes garbage collection.

6. Binder IPC (1MB transaction limit and one-way calls) System

Quick Cheat Sheet

  • Binder IPC: Maps a shared memory buffer space between the calling and target process address spaces, reducing memory copying.
  • Transaction Limit: Limited to **1MB** per process, shared across all active transactions. One-way (asynchronous) calls have a smaller limit (usually 64KB).
Explain the transaction limits of Binder IPC.
Answer
Binder IPC enforces a strict 1MB transaction buffer limit shared across all active transactions in the process. For asynchronous one-way calls, the limit is lower (usually 64KB) to prevent buffer starvation. Passing data that exceeds this limit (such as large images or lists in intents) triggers a TransactionTooLargeException.
Senior-Level Follow-up
What is a Binder Death Recipient, and when should you use it? (A callback registered to receive notification if the target service process terminates unexpectedly).
Common Mistake
Passing heavy serialized objects inside bundle intents across activities or services.

7. ART Compilation Tiers & Baseline Profiles System

Quick Cheat Sheet

  • Compilation Tiers:
    • Interpreter: Runs immediately at startup without compilation overhead.
    • JIT (Just-In-Time): Profiles code during run, compiling hot methods to native code in memory.
    • AOT (Ahead-Of-Time): Pre-compiles hot methods using JIT profiles during idle/charging daemon cycles.
  • Baseline Profiles: Pre-defined JIT profiles shipped inside the APK. ART uses them to pre-compile critical code paths (like startup flows) on installation, bypassing initial interpretation steps.
How do Baseline Profiles optimize cold startup time in ART?
Answer
Usually, ART interprets code at startup and profiles it using JIT. Compilation to native code occurs only after code paths become "hot". Baseline Profiles contain a pre-compiled profile list of classes and methods (like startup flows). Upon installation, ART uses this profile to AOT-compile these paths immediately, avoiding interpreter overhead on the very first launch.
Senior-Level Follow-up
How does ART's concurrent copying garbage collector reduce compaction pauses?
Common Mistake
Assuming that all Java/Kotlin code is compiled to native machine code immediately upon installation on all versions of Android.

8. WorkManager vs Services vs Foreground Services Framework

Quick Cheat Sheet

  • Service: Framework component for background tasks. Runs on the main thread by default. Kept alive relative to the host process priority.
  • Foreground Service: Displays a persistent notification. Elevates process priority to Foreground (oom_adj = 0), preventing termination. Android 14 requires declaring specific service types.
  • WorkManager: Library for persistent, guaranteed background tasks. Persists tasks in SQLite database, scheduling execution via JobScheduler when constraints are met.
When should you use WorkManager instead of a Foreground Service?
Answer
Use WorkManager for deferrable, persistent tasks that require execution guarantees (like uploading analytics or syncing database deltas) even if the app restarts. Use a Foreground Service for immediate, user-noticeable tasks that require continuous execution (like navigation navigation or music playback).
Senior-Level Follow-up
Under what conditions does starting a foreground service trigger a BackgroundServiceStartNotAllowedException? (When trying to start it from the background without specific system permissions or intent triggers).
Common Mistake
Thinking that starting a standard Service automatically runs it on a background thread. (Services run on the main thread by default).

9. PendingIntent Security (FLAG_IMMUTABLE) System

Quick Cheat Sheet

  • PendingIntent: A token granting external processes (like notification manager) permission to execute an intent on your app's behalf with your app's security privileges.
  • Security: On API 31+, developers must specify either FLAG_IMMUTABLE (preferred) or FLAG_MUTABLE. Immutable flags prevent the receiving app from modifying the intent parameters.
Why did Android 12 make FLAG_IMMUTABLE mandatory for most PendingIntents?
Answer
To prevent security vulnerability exploits. If a PendingIntent is mutable, a malicious receiving application can mutate the underlying intent's parameters or target component, executing arbitrary actions with your application's privileges. FLAG_IMMUTABLE locks the intent, preventing modifications.
Senior-Level Follow-up
When is FLAG_MUTABLE explicitly required? (When the system needs to append extra information, such as direct reply text from notifications or location updates).
Common Mistake
Failing to declare any mutability flag on API 31+, which triggers immediate runtime crashes.

10. ANR (Application Not Responding) profiling Performance

Quick Cheat Sheet

  • ANR Trigger: Occurs when the main thread is blocked for longer than 5 seconds (input timeout) or 200 seconds (broadcast receiver). The OS generates a thread dump file (`traces.txt`).
  • Common Causes: Main thread performing disk I/O, network requests, locks on background threads, or long-running computations.
How do you debug an ANR using the traces.txt file?
Answer
When an ANR occurs, the system writes the stack traces of all active threads to /data/anr/traces.txt. To debug, locate your application process in this file, find the "main" thread, and examine its call stack to identify what blocked execution (like database queries or synchronization lock waiting).
Senior-Level Follow-up
How does StrictMode help catch ANR bottlenecks during development?
Common Mistake
Assuming that ANRs are only caused by infinite loops, ignoring blocks caused by synchronized database writes.
