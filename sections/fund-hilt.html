Dependency Injection with Hilt

⏱ 10 min read ✓ Part 0 Fundamentals

Understand Dependency Injection principles using Hilt, class components, scope constraints, modules design (@Binds vs @Provides), and construction configurations.

1. Hilt Components, Scopes & Lifecycles DI Scopes

Concepts & Explanations

  • Hilt Components: Built-in scopes generated at compile-time that align with Android's lifecycles:
    • SingletonComponent: Tied to the Application class (retains `@Singleton` instances).
    • ActivityComponent: Tied to the Activity lifecycle (retains `@ActivityScoped` instances).
    • ViewModelComponent: Tied to ViewModel scope (retains `@ViewModelScoped` instances).
  • Hilt Scopes: Limit object creation to specific lifecycles. If no scope is declared, the component creates a new instance on every injection request.

Code Example

import javax.inject.Inject
import javax.inject.Singleton
import dagger.hilt.android.scopes.ActivityScoped

// Singleton Scope: Retained in SingletonComponent throughout app lifecycle
@Singleton
class SharedRepository @Inject constructor() {
    fun getDeals() = listOf("Deal A", "Deal B")
}

// Activity Scope: Retained in ActivityComponent; new instance created per Activity
@ActivityScoped
class ScreenUIManager @Inject constructor() {
    fun renderDetails() = println("Details rendered on screen")
}
What are the performance implications of scoping every dependency in Hilt?
Answer
Scoping every dependency increases memory footprint and initialization times. Scoped classes must be stored in the component's internal state map and remain in memory for the duration of that component's lifecycle. Leave dependencies unscoped unless they must share internal state or manage shared resources.

2. Constructor vs Field Injection Injection

Concepts & Explanations

  • Constructor Injection: Preferred type. Annotating constructor with `@Inject` makes dependencies explicit and testable.
  • Field Injection: Injecting public fields annotated with `@Inject` in system-instantiated components (Activities, Fragments).

Code Example

import android.os.Bundle
import androidx.appcompat.app.AppCompatActivity
import dagger.hilt.android.AndroidEntryPoint
import javax.inject.Inject

// Constructor Injection: dependencies are explicit in the constructor signature
class DealsUseCase @Inject constructor(
    private val repository: SharedRepository
) {
    fun getTopDeals() = repository.getDeals().take(1)
}

// Field Injection: Hilt injects annotated public fields
@AndroidEntryPoint
class DealsActivity : AppCompatActivity() {

    @Inject lateinit var useCase: DealsUseCase // Field Injection

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        // useCase is initialized by Hilt after super.onCreate()
        val deals = useCase.getTopDeals()
        println("Top Deal: $deals")
    }
}
Why is Constructor Injection preferred over Field Injection?
Answer
Constructor Injection makes dependencies explicit. It allows you to instantiate the class with standard constructor calls in unit tests without starting Hilt, supports immutability (by declaring fields as val), and prevents nullability issues (no lateinit var needed).

3. Hilt Modules (@Binds vs @Provides) Modules

Concepts & Explanations

  • Hilt Modules: Annotating with @Module and @InstallIn(...) configures dependencies that Hilt doesn't know how to instantiate.
  • @Provides: Concrete functions that construct and return third-party objects (like Retrofit clients or Room databases).
  • @Binds: Abstract functions that map implementation classes to interfaces, reducing compiler boilerplate.

Code Example

import dagger.Module
import dagger.Provides
import dagger.Binds
import dagger.hilt.InstallIn
import dagger.hilt.components.SingletonComponent
import javax.inject.Singleton

// Interface definition
interface AnalyticsService {
    fun logEvent(event: String)
}

class FirebaseAnalytics @Inject constructor() : AnalyticsService {
    override fun logEvent(event: String) = println("Logged: $event")
}

@Module
@InstallIn(SingletonComponent::class)
abstract class AnalyticsModule {

    // 1. @Binds maps implementation class to interface
    @Binds
    @Singleton
    abstract fun bindAnalyticsService(impl: FirebaseAnalytics): AnalyticsService
}

@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {

    // 2. @Provides constructs third-party Retrofit mock client
    @Provides
    @Singleton
    fun provideRetrofitClient(): String {
        return "RetrofitHttpClientInstance"
    }
}
Why is @Binds preferred over @Provides?
Answer
@Binds is preferred because it generates no bytecode representation at compile-time. Unlike @Provides (which compiles into a dedicated class and factory instance), @Binds only informs the Dagger compiler of the type mapping, improving compile times and reducing the final APK size.
