Coroutines & Concurrency Fundamentals

⏱ 15 min read ✓ Part 0 Fundamentals

In-depth exploration of Kotlin coroutines, processes vs threads, Continuation Passing Style, state machine compilation, structured concurrency, and async streams.

1. Processes, Threads & Concurrency vs Parallelism Foundations

Concepts & Explanations

  • Processes vs Threads: A process represents an active instance of an application running in its own private memory heap. A thread is an execution pathway inside a process, sharing memory segments but allocating its own stack.
  • Concurrency vs Parallelism: Concurrency is the scheduling of multiple execution flows within overlapping time intervals (interleaving execution, e.g. single-core fast switching). Parallelism is the simultaneous execution of multiple tasks at the exact same instant (multiple CPU cores).

Code Example

import kotlin.concurrent.thread

fun runClassicThreads() {
    // Basic Process-bound JVM Threads
    val thread1 = thread(start = true) {
        println("Thread 1: Performing background task on: ${Thread.currentThread().name}")
        Thread.sleep(1000) // Blocks active JVM thread
    }

    val thread2 = thread(start = true) {
        println("Thread 2: Performing background task on: ${Thread.currentThread().name}")
        Thread.sleep(1000)
    }

    thread1.join()
    thread2.join()
}
Why are JVM threads considered expensive compared to Kotlin coroutines?
Answer
JVM threads map directly to OS kernel threads. Each thread requires a dedicated memory stack allocation (typically 1MB). Context switching between threads requires saving register states, switching virtual memory spaces, and modifying scheduler queues in the OS kernel. Coroutines are purely user-land abstraction entities that run inside thread pools, sharing stack space and context switching in user memory, allowing millions of active instances.

2. CoroutineScope, Jobs & Builders (launch vs async) Builders

Concepts & Explanations

  • CoroutineScope: Establishes a boundary of execution context for new coroutines. Every builder (like launch, async) is an extension of this scope.
  • Job vs SupervisorJob: A Job acts as a lifecycle handle. Standard Jobs propagate failure upwards, cancelling all other siblings and the parent. A SupervisorJob isolates failures; a child's failure does not impact siblings or parent scopes.
  • launch vs async: launch returns a fire-and-forget Job with no return value. async returns a Deferred which holds a future result retrievable via the suspending call await().

Code Example

import kotlinx.coroutines.*

val customScope = CoroutineScope(Dispatchers.Default + SupervisorJob())

fun executeBuilders() {
    // 1. launch: Fire-and-forget
    val job: Job = customScope.launch {
        println("Running launch worker")
        delay(500)
    }

    // 2. async: Holds deferred result
    val deferred: Deferred = customScope.async {
        println("Calculating result...")
        delay(1000)
        42
    }

    customScope.launch {
        val result = deferred.await() // Suspends until async returns value
        println("Async result received: $result")
    }
}
What happens if a coroutine inside a supervisorScope throws an exception?
Answer
Unlike a standard coroutineScope where child failures automatically cascade to cancel the entire tree, a supervisorScope overrides child failure propagation. The thrown exception does not cancel other active sibling coroutines or the parent scope, allowing you to catch and handle the exception locally.

3. Suspend Functions, Continuation & State Machine Transformation Internals

Concepts & Explanations

  • Suspend Functions: Suspend execution without blocking the host thread, allowing other tasks to run on that thread.
  • Continuation & CPS: Under the hood, the compiler rewrites suspending methods using Continuation Passing Style (CPS). The compiler appends an implicit Continuation callback argument to the function signature.
  • State Machine Transformation: The compiled function's body is transformed into an efficient state machine (compiled as a switch statement in bytecode). Each suspension point maps to a state label index. Before suspension, local variables are saved to the continuation instance, and the function exits, returning a COROUTINE_SUSPENDED token. Upon completion, the continuation's resumeWith() is called to re-enter the function and jump back to the saved state label.

Code Example

import kotlinx.coroutines.delay

// Standard suspending function
suspend fun fetchDeals(): List {
    println("Fetch deals start")
    delay(500) // Suspension point 1 (State 0 -> State 1)
    println("Processing deals...")
    delay(300) // Suspension point 2 (State 1 -> State 2)
    return listOf("Deal A", "Deal B")
}

/* 
  Conceptual JVM Bytecode representation generated by compiler:
  
  fun fetchDeals(completion: Continuation>): Any? {
      val stateMachine = completion as? FetchDealsContinuation ?: object : FetchDealsContinuation() {
          var label = 0
          var result: Any? = null
          override fun resumeWith(res: Result) {
              this.result = res
              this.label++
              fetchDeals(this)
          }
      }
      
      switch(stateMachine.label) {
          case 0:
              stateMachine.label = 1
              if (delay(500, stateMachine) == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
          case 1:
              stateMachine.label = 2
              if (delay(300, stateMachine) == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
          case 2:
              return listOf("Deal A", "Deal B")
      }
  }
*/
Does a suspend function allocate a new thread when called?
Answer
No. Suspend functions do not create or manage threads. They are compiler-rewritten callback mechanisms. When suspending, the execution state is saved to a callback object (Continuation) on the heap, and the thread is released to execute other tasks until the callback invokes its resume method.

4. Dispatchers (Main, IO, Default, Unconfined) Execution

Concepts & Explanations

  • Dispatchers.Main: Bound to the UI main thread loop (via Looper/Handler on Android). All UI updates must execute here.
  • Dispatchers.IO: Shared elastic pool (up to 64 threads or matching core counts) optimized for blocking input/output operations (network, database).
  • Dispatchers.Default: Fixed pool of threads matching CPU cores count. Designed for high CPU-intensive work.
  • Dispatchers.Unconfined: Executes code immediately on the current calling thread context, resuming execution on whatever thread the suspending function completes on.

Code Example

import kotlinx.coroutines.*

fun runOnDifferentDispatchers() = runBlocking {
    launch(Dispatchers.Default) {
        println("Default CPU thread: ${Thread.currentThread().name}")
        
        withContext(Dispatchers.IO) {
            println("IO Database thread: ${Thread.currentThread().name}")
        }
        
        launch(Dispatchers.Unconfined) {
            println("Unconfined pre-suspend thread: ${Thread.currentThread().name}")
            delay(100) // suspends
            println("Unconfined post-suspend thread: ${Thread.currentThread().name}")
        }
    }
}
When would you choose to use Dispatchers.Unconfined?
Answer
Dispatchers.Unconfined is rarely used in application code. It is suitable for unit tests or performance-critical utility callbacks where no thread confinement is required, eliminating dispatch queuing latency by executing tasks immediately on the caller's thread context.

5. Structured Concurrency, Cancellation & Exception Handling Control Flow

Concepts & Explanations

  • Structured Concurrency: Ensures that a parent scope does not complete until all of its spawned child coroutines complete. This prevents background resource leaks.
  • Cancellation: Cooperative in nature. Coroutines periodically inspect their status (checking isActive or throwing CancellationException during suspension calls).
  • Exception Handling: Uncaught errors propagate up to the root parent, canceling the entire scope. Use CoroutineExceptionHandler to catch unhandled errors at the root scope, or utilize supervisorScope to handle failures locally.

Code Example

import kotlinx.coroutines.*

fun runCooperativeScope() = runBlocking {
    val exceptionHandler = CoroutineExceptionHandler { _, exception ->
        println("Caught root exception: ${exception.message}")
    }

    val job = launch(exceptionHandler) {
        // Child 1: Cooperative cancellation loop
        val child1 = launch {
            try {
                while (isActive) {
                    println("Worker running...")
                    delay(100)
                }
            } catch (e: CancellationException) {
                println("Child 1 cancelled clean")
            }
        }
        
        // Child 2: Throws runtime error
        val child2 = launch {
            delay(200)
            throw RuntimeException("Child 2 failed!")
        }
    }
    
    job.join() // Waits for execution to finish
}
Why is catching CancellationException in a catch-all block considered dangerous?
Answer
If you catch and swallow CancellationException (e.g. catch (e: Exception) without rethrowing it), the coroutine stays active, ignoring cancellation commands from its parent scope. This breaks cooperative cancellation, leading to resource leaks. Always rethrow CancellationException or avoid catching it directly.

6. Flows, StateFlow, SharedFlow & Channels Streams

Concepts & Explanations

  • Flow: Cold reactive stream. Emits values only when a collector subscribes.
  • StateFlow: Hot state-holder. Retains its latest value (requires initial value) and replays it to new collectors.
  • SharedFlow: Hot event-emitter. Broadcasts one-off events (like alerts or navigation commands) to active subscribers.
  • Channels: Hot queue-like communication pipeline. Synchronizes data transfer between producer and consumer coroutines.

Code Example

import kotlinx.coroutines.*
import kotlinx.coroutines.flow.*
import kotlinx.coroutines.channels.Channel

fun runStreamsDemo() = runBlocking {
    // 1. Cold Flow
    val coldFlow: Flow = flow {
        emit(1); delay(100); emit(2)
    }
    
    // 2. StateFlow (Hot)
    val stateFlow = MutableStateFlow("InitialState")
    
    // 3. SharedFlow (Hot)
    val sharedFlow = MutableSharedFlow(replay = 0)
    
    // 4. Channel (Queue)
    val channel = Channel(Channel.BUFFERED)
    
    launch {
        coldFlow.collect { println("Collected: $it") }
        
        stateFlow.value = "UpdatedState" // updates StateFlow value
        
        sharedFlow.emit("NavigateToHome") // emits SharedFlow event
        
        channel.send(100) // sends Channel item
        println("Channel received: ${channel.receive()}")
    }
}
What is the key difference between StateFlow and SharedFlow?
Answer
StateFlow represents state (always has an initial value, holds state, filters duplicate values, and replays the latest state to new collectors). SharedFlow represents events (no initial value, does not filter duplicates, and events are processed once and discarded).
