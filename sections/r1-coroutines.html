Section 2: Coroutines & Concurrency

⏱ 10 min read ✓ Part 1 Core

Consolidated, high-yield overview of coroutines execution pools, continuation state machines, structured concurrency rules, and thread pools.

1. Threads vs Coroutines Concurrency

Quick Cheat Sheet

  • Threads: Managed by the OS kernel. Mapping is 1:1. Each allocates a fixed stack (typically 1MB). Context switching involves kernel-space transition and is resource-heavy.
  • Coroutines: Managed in user-space by the library runtime. They are lightweight (often called "cooperative threads"). They do not have dedicated stacks and run on top of shared thread pools. Context switching is a lightweight stack-frame switch.
Why are coroutines lighter than standard Java threads?
Answer
Standard threads map 1:1 to OS-level kernel threads and reserve 1MB of memory for call stacks upfront, limiting concurrent execution. Coroutines run in user space, sharing threads from a common pool. When a coroutine suspends, it does not block the thread; instead, it saves its state as a lightweight object on the heap, allowing thousands of coroutines to share a single thread.
Senior-Level Follow-up
What happens if you run a blocking thread operation (like Thread.sleep) inside a coroutine? (It blocks the underlying dispatcher thread, starving other coroutines).
Common Mistake
Running blocking calculations on the main dispatcher, assuming that "coroutines are asynchronous" automatically protects the UI thread.

2. CoroutineScope & Structured Concurrency Scope

Quick Cheat Sheet

  • CoroutineScope: Wraps a CoroutineContext. Serves as the entry point to launch coroutines, establishing execution boundaries.
  • Structured Concurrency: Guarantees that:
    • Parent scopes wait for all their children to complete.
    • Cancelling a scope automatically cancels all child jobs launched within it, preventing leaks.
    • Child failures automatically propagate up to cancel the parent.
Explain the principles of Structured Concurrency.
Answer
Structured concurrency ensures that new coroutines are launched within a defined scope, linking their lifecycles. A parent coroutine is not complete until all its child coroutines complete. If a parent scope is cancelled, all its children are cancelled, preventing memory and background task leaks.
Senior-Level Follow-up
What are the security and resource leak risks of using GlobalScope?
Common Mistake
Launching coroutines in GlobalScope inside ViewModels, causing tasks to continue running even after the ViewModel is cleared.

3. Job vs SupervisorJob Lifecycle

Quick Cheat Sheet

  • Job: A standard background task. If any child Job fails, it immediately cancels its parent and all other sibling coroutines.
  • SupervisorJob: Overrides default failure propagation. A failure in a child coroutine does not cancel its siblings or parent, isolating the error.
When would you use a SupervisorJob instead of a standard Job?
Answer
Use a SupervisorJob when launching multiple independent tasks where a failure in one should not affect the others. For example, on a dashboard showing independent widgets (flights, hotels, weather), you want to load the other widgets even if the weather request fails. Use a standard Job when tasks are interdependent and a single failure invalidates the entire operation.
Senior-Level Follow-up
Why does calling launch(SupervisorJob()) inside a standard scope still cancel siblings on failure? (Because the new job is a child of the scope's standard Job; the scope itself must be initialized with a SupervisorJob).
Common Mistake
Passing a SupervisorJob to the child launcher and expecting it to isolate failures, instead of using supervisorScope block or configuring the parent scope with a SupervisorJob.

4. launch vs async builders Builders

Quick Cheat Sheet

  • launch: Fire-and-forget coroutine builder. Returns a Job. Any unhandled exception is propagated to the parent immediately, which cancels the scope.
  • async: Performs background calculations that return a result. Returns a Deferred<T>. Result is accessed by calling await().
How does exception propagation differ between launch and async?
Answer
launch propagates unhandled exceptions to the parent scope immediately, which cancels the parent and siblings. async captures the exception and throws it when await() is called. However, if async is child to a standard coroutine scope, the failure still cancels the parent scope immediately, regardless of whether await() is invoked.
Senior-Level Follow-up
How does wrapping an async call in supervisorScope isolate its exception?
Common Mistake
Wrapping a call to await() in a try-catch block and expecting it to prevent the parent scope from cancelling.

5. suspend functions & Continuation CPS compilation Compilation

Quick Cheat Sheet

  • Continuation-Passing Style (CPS): The Kotlin compiler rewrites suspending functions by adding an extra parameter of type Continuation<T> to the method signature.
  • State Machine Transformation: The compiler compiles the function body into a switch-case state machine, assigning a state label to each suspension point. When a suspending function yields, the state label and local variables are saved to a Continuation object on the heap, and the function returns COROUTINE_SUSPENDED. When the task completes, resumeWith() is called, loading the saved state and jumping back to the saved state label.
What does the compiler generate for a suspend function?
Answer
The compiler adds a Continuation argument to the function signature and changes the return type to Any?. It compiles the function body into a switch-case state machine. Each case represents a step in execution between suspension points, saving and restoring local variables and execution state labels on the heap.
Senior-Level Follow-up
How does the COROUTINE_SUSPENDED sentinel value indicate suspension to the runtime loop?
Common Mistake
Thinking that the `suspend` keyword automatically executes the function on a background thread.

6. Dispatchers (Main, IO, Default, Unconfined) Dispatchers

Quick Cheat Sheet

  • Dispatchers.Main: Confined to the UI main thread. Uses the Android Looper message queue under the hood. Use `Main.immediate` to bypass queue delays if already on the main thread.
  • Dispatchers.IO: Optimized for blocking I/O (network, database). Shares a thread pool with `Default` but allows dynamic growth up to 64 threads (or core count) to prevent starvation.
  • Dispatchers.Default: Optimized for CPU-bound computations (JSON parsing, sorting). Confines threads to the number of physical CPU cores (at least 2).
  • Dispatchers.Unconfined: Runs the coroutine on the caller thread until the first suspension point. On resumption, it executes on whatever thread resumed the continuation. (Discouraged for production code).
Explain how Dispatchers.IO and Dispatchers.Default share threads.
Answer
They share the same underlying pool of worker threads. However, Dispatchers.Default limits active concurrent threads to the device's CPU core count to prevent CPU thrashing. Dispatchers.IO allocates up to 64 threads (or more) because I/O threads spend most of their time suspended (waiting on network or disk writes), meaning they do not compete for CPU cycles. Shared thread pools avoid the overhead of thread creation.
Senior-Level Follow-up
How does Dispatchers.IO.limitedParallelism(1) serve as a lock-free replacement for synchronized blocks?
Common Mistake
Using Dispatchers.IO for heavy mathematical computations, which can starve actual I/O operations.

7. Coroutine Cancellation & Cooperative Loops Lifecycle

Quick Cheat Sheet

  • Cooperative Cancellation: Coroutines check for cancellation at suspension points. Standard suspending functions (like `delay` or flow collection) throw CancellationException if their job was cancelled.
  • Calculation Loops: If a coroutine executes a tight CPU loop without calling suspending functions, it ignores cancellation. You must explicitly check the status using ensureActive(), yield(), or the isActive property.
How do you make a heavy CPU calculation loop cooperative with cancellation?
Answer
Check the cancellation status on every iteration using ensureActive() (which throws a CancellationException if the job is cancelled) or by checking the isActive property. Alternatively, call yield() inside the loop, which checks cancellation and allows other tasks to run on the thread.
Senior-Level Follow-up
Why must you use withContext(NonCancellable) to run cleanup code inside a finally block if the cleanup requires suspending calls?
Common Mistake
Catching all exceptions (using `catch(e: Exception)`) and swallowing CancellationException, preventing proper cancellation flow.

8. Exception Handling & CoroutineExceptionHandler Exception

Quick Cheat Sheet

  • Propagation: Exceptions propagate up to the root job, cancelling the parent and siblings.
  • CoroutineExceptionHandler: Used to log and handle uncaught exceptions. It must be registered at the **root** coroutine context (passed to the root launcher). Registering a handler on a child coroutine has no effect, as exceptions propagate up to the parent.
Why does registering a CoroutineExceptionHandler on a child coroutine fail to catch errors?
Answer
Because exceptions inside child coroutines always propagate up the tree to the parent. The parent delegates the exception to its parent, continuing until a root coroutine scope is reached. The root coroutine is the only component that invokes the `CoroutineExceptionHandler`. Pass the handler to the root `launch` block or the scope itself.
Senior-Level Follow-up
What is the difference in exception handling between supervisorScope and coroutineScope?
Common Mistake
Wrapping a child launch call in a try-catch block and expecting it to prevent the parent scope from cancelling.

9. Flow (Cold) vs SharedFlow/StateFlow (Hot) Streams

Quick Cheat Sheet

  • Flow (Cold): Runs its producer block on-demand for every collector (e.g. database queries or network requests). It does not buffer values or maintain state.
  • StateFlow (Hot): Retains a single state value, requires an initial value, and replays the latest value to new subscribers. It deduplicates consecutive duplicate updates (conflates values). Perfect for UI state.
  • SharedFlow (Hot): A generic event bus. It does not require an initial value or deduplicate updates. It can buffer and replay events, making it ideal for one-time events (like navigation or toasts).
What are the differences between StateFlow and standard Cold Flow?
Answer
A Cold Flow executes its code block independently for each collector. It does not hold state. StateFlow is a hot stream that runs independently of collectors. It always retains the latest state in memory, requires an initial value, and broadcasts the current value to all active collectors, deduplicating consecutive identical updates.
Senior-Level Follow-up
How does repeatOnLifecycle prevent resource leaks when collecting Flows in Android views?
Common Mistake
Using StateFlow to emit one-time navigation actions, which triggers duplicate navigation actions on screen rotation.

10. Channels & Channel Types Streams

Quick Cheat Sheet

  • Channel: A hot point-to-point communication queue. Each element sent to a channel is processed by exactly one receiver.
  • Channel Types:
    • Rendezvous (Capacity 0): Sender suspends until a receiver retrieves the element.
    • Buffered (Fixed size): Senders only suspend once the buffer is full.
    • Conflated (Capacity 1): Sender never suspends; new elements overwrite older ones.
    • Unlimited (Linked-list): Senders never suspend, risking OOM if the receiver is too slow.
What is the difference between a Channel and a SharedFlow?
Answer
A Channel is a point-to-point (unicast) queue. Each element is consumed once: if multiple coroutines read from a channel, they split the elements. A SharedFlow is a hot broadcast (multicast) stream: every emitted value is received by all active collectors.
Senior-Level Follow-up
What happens when you call trySend on a Rendezvous channel that has no active receivers? (It returns a failed channel result without suspending).
Common Mistake
Using a Channel to distribute UI state updates to multiple views, which results in views receiving partial updates.
