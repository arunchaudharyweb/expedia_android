1. SOLID Principles in Android Design Patterns

Apply SOLID principles to Android architecture:

  • Single Responsibility (SRP): A class should have one reason to change. Example: Separation of hotel network models from UI presentation models.
  • Open/Closed (OCP): Open for extension, closed for modification. Example: Using an interface for payment processors (e.g. Card vs ApplePay) rather than conditional checks inside a checkout manager.
  • Liskov Substitution (LSP): Subclasses should be substitutable for their base class. Example: Custom view classes should override methods without breaking the contract established by the parent View class.
  • Interface Segregation (ISP): Clients should not depend on interfaces they do not use. Example: Splitting a massive list adapter listener interface into smaller, focused interfaces.
  • Dependency Inversion (DIP): Depend on abstractions, not on concrete classes. Example: Injecting a repository interface into a ViewModel rather than instantiating the concrete implementation.
// Open/Closed Principle: Open for expansion, closed for editing
interface PaymentProcessor {
    suspend fun processPayment(amount: Double): Result
}

class GooglePayProcessor : PaymentProcessor {
    override suspend fun processPayment(amount: Double): Result {
        // Implement Google Pay logic
        return Result.success(Unit)
    }
}
Interview Question
Explain how you apply the Interface Segregation and Dependency Inversion principles in Android development.
Answer
Interface Segregation is applied by splitting large interface classes (such as a generic listener) into smaller, focused interfaces (like HotelSelectionListener). Dependency Inversion is applied by injecting interface abstractions into clients (like ViewModels) instead of instantiating concrete implementations, allowing dependencies to be swapped for testing.
Senior Follow-Up
How does the Liskov Substitution Principle guide your custom View class implementations?
Common Candidate Mistake
Writing custom view classes that throw unsupported operation exceptions in overridden lifecycle methods, violating Liskov Substitution.

2. Senior Naming Conventions Code Review

Senior naming conventions prioritize clarity, intent, and domain vocabulary over brevity:

  • State Variables: Suffix state variables with UiState (e.g. bookingUiState) to clarify that they represent screen state.
  • ViewModel Intent actions: Prefix intent event names with verbs (e.g. LoadDetails, SelectRoom) to indicate actions.
  • Interface implementations: Use descriptive names instead of simple "Impl" suffixes (e.g. RoomRoomDatabaseCache instead of DatabaseCacheImpl) to document the concrete technology choice.
Interview Question
Why are generic suffixes like 'Impl' discouraged, and what naming patterns indicate seniority?
Answer
"Impl" suffixes do not describe the class's purpose or technology choice. Suffix implementations with descriptive terms (such as RoomLocalCache or RetrofitHotelApi) to document their design and dependencies clearly.
Senior Follow-Up
How do you name callback parameters in asynchronous function declarations to clarify thread safety?
Common Candidate Mistake
Using generic names (like data, list, or manager) that fail to convey the domain concept or purpose of the class.

3. Function Extraction Guidelines Refactoring

Extract logic into helper functions based on these criteria:

  • High Cognitive Complexity: Extract nested loops or complex conditional blocks to improve readability.
  • Reusability & DRY: Extract logic that is repeated across different parts of the application to prevent code duplication.
  • Testability: Extract complex algorithms into pure functions, allowing them to be unit tested in isolation without mocking dependencies.

When not to extract: Do not extract simple, sequential initialization steps that are run only once, as this can make the code harder to follow.

Interview Question
What criteria do you use to decide whether to extract logic into a separate function?
Answer
Extract logic if it contains complex nested conditions, is repeated across classes, or can be isolated as a pure function to simplify unit testing. Avoid extracting simple, sequential code paths, as this can fragment the logic and make it harder to read.
Senior Follow-Up
How does extracting inline functions in Kotlin affect performance and bytecode size?
Common Candidate Mistake
Extracting small, single-use utility methods that are closely coupled to class variables, which fragments code readability.

4. Functional Error Handling with Result Functional

Instead of throwing exceptions, use functional error returns with a sealed Result class to make error states explicit: 

// Kotlin functional error handling
suspend fun fetchHotelDetail(id: String): Result {
    return try {
        val details = api.fetchHotel(id)
        Result.success(details)
    } catch (e: Exception) {
        Result.failure(e)
    }
}

// Processing results in UI
val result = fetchHotelDetail("1")
result.fold(
    onSuccess = { displayHotel(it) },
    onFailure = { showErrorMessage(it.message) }
)
Interview Question
Why is functional error handling with a Result class preferred over standard try-catch blocks?
Answer
Functional error handling makes error states explicit in the function signature, forcing callers to handle errors using utilities like fold. This prevents uncaught runtime crashes and helps separate normal control flow from error handling.
Senior Follow-Up
How do you map error subclasses inside a repository layer without losing the original exception stack trace?
Common Candidate Mistake
Swallowing errors by returning null values instead of using an explicit Result type, which hides the failure reason.

5. Callback-to-Coroutine Migration Async

Migrate legacy callback-based APIs to suspending functions using suspendCancellableCoroutine to avoid callback hell and handle cancellations safely. 

import kotlinx.coroutines.suspendCancellableCoroutine
import kotlin.coroutines.resume
import kotlin.coroutines.resumeWithException

// Legacy Callback API
interface BookingCallback {
    fun onSuccess(bookingId: String)
    fun onError(error: Throwable)
}

// Clean suspending implementation
suspend fun executeBooking(details: BookingDetails): String = suspendCancellableCoroutine { continuation ->
    val call = api.requestBooking(details, object : BookingCallback {
        override fun onSuccess(bookingId: String) {
            continuation.resume(bookingId)
        }
        override fun onError(error: Throwable) {
            continuation.resumeWithException(error)
        }
    })
    
    // Clean up or cancel the API call if the coroutine is cancelled
    continuation.invokeOnCancellation {
        call.cancel()
    }
}
Interview Question
How do you wrap callback-based APIs in a suspending function, and how do you handle cancellation?
Answer
Use suspendCancellableCoroutine to obtain a continuation handle. Call continuation.resume() or continuation.resumeWithException() inside the callbacks. Use the continuation.invokeOnCancellation callback to cancel active network or database requests if the coroutine is cancelled.
Senior Follow-Up
What is the difference between suspendCoroutine and suspendCancellableCoroutine?
Common Candidate Mistake
Using the standard suspendCoroutine instead of the cancellable variant for network API calls, leaving request tasks active on cancellation.

6. Common Android Code Smells & Fixes Refactoring

Common architectural code smells in Android and their corresponding fixes:

  • God ViewModel: A ViewModel that contains database queries, network calls, and UI routing logic. Fix: Separate concerns by delegating data operations to Repositories and routing to Navigator classes.
  • Leaking Activity Context: Storing Context references in repositories, singletons, or thread tasks. Fix: Pass application context references only when necessary, or inject isolated platform helper interfaces.
  • Business Logic in Fragment: Placing domain logic or state manipulation inside Fragments or Activities. Fix: Move calculations and state mutations to ViewModels or Domain Use Cases.
Interview Question
Identify three architectural code smells in Android applications and explain how you would resolve them.
Answer
1. God ViewModel: Delegate network/database calls to repositories. 2. Context leaks: Avoid storing context references in long-lived objects. 3. Presentation-layer business logic: Move validation and transformation algorithms to ViewModels or Domain layers.
Senior Follow-Up
How does architectural separation of concerns affect application build times?
Common Candidate Mistake
Writing complex string formatting and currency conversion logic inside Fragments, which couples formatting code to the UI lifecycle and complicates testing.
