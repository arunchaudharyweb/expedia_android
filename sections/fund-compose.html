Jetpack Compose & UI Architecture

⏱ 15 min read ✓ Part 0 Fundamentals

Explore Jetpack Compose compilation, state hoisting, lifecycle side effects, runtime snapshot system mechanics, navigation structures, and compiler optimizations.

1. Composable Functions, Recomposition & Slot Table Compiler

Concepts & Explanations

  • Composable Functions: Annotated with @Composable. The compiler modifies these functions to receive a Composer instance and write layout metadata into the Slot Table database.
  • Slot Table: A sequential array structure that stores composition states, keys, and values. It acts as the database of the UI hierarchy.
  • Recomposition: Re-running composable functions when their state dependencies change. Compose skips executing functions whose arguments are unchanged.

Code Example

import androidx.compose.runtime.*
import androidx.compose.material3.Text

@Composable
fun GreetingWidget(name: String) {
    // Recomposes when 'name' changes
    println("GreetingWidget Recomposing for: $name")
    Text(text = "Welcome to $name")
}
What role does the Composer play in Composable bytecode?
Answer
The Compose compiler injects a Composer parameter into every Composable function signature (e.g. fun Widget($composer: Composer, $key: Int)). During execution, the composer reads and writes nodes, keys, and states to the Slot Table, checking if values are unchanged to bypass rendering.

2. State, remember, rememberSaveable & State Hoisting State

Concepts & Explanations

  • State: An observable wrapper that registers recomposition scopes on read.
  • remember: Caches a value inside the Slot Table across recompositions.
  • rememberSaveable: Automatically persists state inside the Bundle system to survive configuration changes and process death.
  • State Hoisting: Moving state up to make a composable stateless, promoting reusability and testability.

Code Example

import androidx.compose.runtime.*
import androidx.compose.material3.*

@Composable
fun CounterParent() {
    // Hoisted State: Parent owns the state
    var count by rememberSaveable { mutableStateOf(0) }
    
    CounterChild(
        count = count,
        onIncrement = { count++ }
    )
}

@Composable
fun CounterChild(count: Int, onIncrement: () -> Unit) {
    // Stateless Composable: simple input and event triggers
    Button(onClick = onIncrement) {
        Text("Count: $count")
    }
}
What is the difference between remember and rememberSaveable?
Answer
remember stores values in the Slot Table. These values survive recompositions but are lost during activity recreation (like screen rotation). rememberSaveable serializes values into a Bundle, allowing them to survive configuration changes and background process death.

3. Side Effects (LaunchedEffect, DisposableEffect, produceState, derivedStateOf) Side Effects

Concepts & Explanations

  • LaunchedEffect: Launches a coroutine scope when entering composition. It cancels and re-launches when a specified key changes.
  • DisposableEffect: Launches an effect that requires cleanup, executing its onDispose block when the key changes or the composable exits.
  • produceState: Converts non-Compose state flows (like flows or callbacks) into Compose state properties.
  • derivedStateOf: Calculates state based on other states, caching results and only updating when the computed output changes, preventing redundant recompositions.

Code Example

import androidx.compose.runtime.*
import kotlinx.coroutines.delay

@Composable
fun UserStatusDisplay(userId: String) {
    var status by remember { mutableStateOf("Offline") }

    // LaunchedEffect: re-triggers when userId changes
    LaunchedEffect(userId) {
        status = "Loading..."
        delay(1000)
        status = "Online for $userId"
    }

    // DisposableEffect: registers a sensor listener and unregisters on dispose
    DisposableEffect(userId) {
        println("Registering listener for $userId")
        onDispose {
            println("Unregistering listener for $userId")
        }
    }

    // produceState: produces a state from an async source
    val timeState = produceState(initialValue = 0) {
        while(true) {
            delay(1000)
            value++
        }
    }

    Text("Status: $status, Uptime: ${timeState.value}s")
}
Why should you avoid reading mutableState values directly in composition if they change frequently?
Answer
Reading a rapidly changing state (like scroll offsets) directly inside composition triggers recomposition on every value change. Use derivedStateOf to filter updates, or pass the value inside lambda parameters in modifiers to bypass composition, routing updates directly to the layout and draw phases.

4. The Compose Snapshot System Internals

Concepts & Explanations

  • Snapshot System: The database engine of Compose state. It tracks values at a specific point in time, allowing isolated reads and writes across threads (Multi-Version Concurrency Control). Commit actions schedule recomposition.
  • Snapshot Read/Write: When state values are read during composition, the compiler registers a read observer to identify which scopes depend on those states, invalidating them when values change.

Code Example

import androidx.compose.runtime.snapshots.Snapshot
import androidx.compose.runtime.mutableStateOf

fun testSnapshotSystem() {
    val state = mutableStateOf("Initial State")

    // Take a snapshot
    val snapshot = Snapshot.takeSnapshot()

    state.value = "Updated State"

    // Read state within the captured snapshot context
    snapshot.enter {
        println("Value inside snapshot: ${state.value}") // Output: "Initial State"
    }

    println("Value outside snapshot: ${state.value}") // Output: "Updated State"
}
How does the Snapshot system support multithreaded state updates?
Answer
The snapshot system uses Multi-Version Concurrency Control (MVCC). It allows threads to read and write state properties inside their own isolated snapshots. Changes are only committed globally when the snapshot exits, triggering recomposition observers on the main thread safely.

5. Navigation in Compose Navigation

Concepts & Explanations

  • Navigation Compose: Manages UI navigation flows. The NavHost binds a NavController to a navigation graph, swapping composables in the content area based on route strings.

Code Example

import androidx.compose.runtime.Composable
import androidx.navigation.compose.*

@Composable
fun AppNavigation() {
    val navController = rememberNavController()
    NavHost(navController = navController, startDestination = "home") {
        composable("home") {
            HomeScreen(onNavigateToDetails = { navController.navigate("details/123") })
        }
        composable("details/{id}") { backStackEntry ->
            val id = backStackEntry.arguments?.getString("id")
            DetailsScreen(id = id)
        }
    }
}

@Composable fun HomeScreen(onNavigateToDetails: () -> Unit) {}
@Composable fun DetailsScreen(id: String?) {}
What are the pitfalls of passing complex data structures inside Navigation routes?
Answer
Passing complex objects (like custom data classes) in routes as serialized JSON strings violates Android's navigation principles. It leads to bloated route URLs and potential crashes if sizes exceed bundle limits. Instead, pass simple unique IDs as arguments and fetch the details object from a shared repository.

6. Performance Optimization, Stability & Skippability Performance

Concepts & Explanations

  • Stability: Types where values are unchanging, or whose change notifications are visible to Compose. Compose skips rendering composables if their arguments are Stable and unchanged (Skippability).
  • Skippability: Compiler optimization. If a composable's arguments are stable and unchanged, Compose skips recomposing it entirely.
  • Unstable parameters: standard collections (like List) are treated as unstable because their implementations can mutate. Fix by wrapping lists in `@Stable` classes or using kotlinx immutable collections.

Code Example

import androidx.compose.runtime.*

// 1. Unstable parameter class (standard collection list)
@Composable
fun UnstableListWidget(items: List) {
    // Recomposes even if items content is unchanged because List is marked unstable
    Text("Count: ${items.size}")
}

// 2. Stable parameter class
@Stable
data class StableWrapper(val items: List)

@Composable
fun StableListWidget(wrapper: StableWrapper) {
    // Skippable: will skip recomposition if wrapper holds the same items
    Text("Count: ${wrapper.items.size}")
}
How does the Compose compiler classify class stability?
Answer
A class is classified as stable if: 1. All public properties are immutable (declared with val) and stable. 2. It notifies the runtime of state changes (e.g. using mutableStateOf). If any property is mutable (var) or is an interface (like List), the class is classified as unstable.
