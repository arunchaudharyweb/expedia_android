Streamlined Kotlin cheat sheet focusing on high-probability Expedia interview questions, reference mechanics, memory structures, collection scaling, and generic variance.

1. val vs var Reference Mutability Basics

Quick Cheat Sheet

  • val: Declares a read-only, final reference. In bytecode, compiles to a final field with a getter.
  • var: Declares a mutable reference. In bytecode, compiles to a non-final field with both getter and setter.
  • Reference vs Object Immutability: A val collection reference (e.g. val list = mutableListOf(1)) cannot be reassigned, but the contents of the collection can still be modified.
Does declaring a val property with a custom getter generate a backing field in Java bytecode?
Answer
No. If a property is declared as val and has a custom getter (e.g., val isAdult get() = age >= 18) without using the field identifier, the compiler does not generate a backing field. It compiles simply into a standard Java method.
Senior-Level Follow-up
How does the const val compile-time constant differ from a standard val? (Inlined at compile-time directly in calling code, primitives and strings only).
Common Mistake
Assuming that declaring a collection reference as `val` makes the collection immutable.

2. Null Safety & Platform Types Type System

Quick Cheat Sheet

  • Safe Calls (?.): Executes downstream references only if non-null, returning null otherwise.
  • Elvis Operator (?:): Evaluates default expressions if the left-side expression resolves to null.
  • Double Bang (!!): Forces non-null casting, throwing a NullPointerException if null.
  • Platform Types (T!): Types coming from Java without nullability annotations. Kotlin does not enforce compile-time checks on them, which can trigger runtime NPEs on access.
What are Platform Types in Kotlin and how do they impact safety?
Answer
Platform types are types originating from Java code that lack nullability annotations (like @Nullable or @NotNull). Kotlin represents them with a single exclamation mark (e.g., String!). Because they bypass Kotlin's compile-time null safety checks, they can trigger runtime NullPointerException if dereferenced when null. To prevent this, annotate Java APIs or cast them to explicit nullable/non-null Kotlin types immediately.
Senior-Level Follow-up
Explain the overhead of Kotlin's compiler-inserted Intrinsics.checkNotNullParameter assertions on method arguments.
Common Mistake
Using the double bang (!!) operator to resolve compiler errors instead of using safe navigation or smart casting.

3. Data Classes vs Standard Classes Classes

Quick Cheat Sheet

  • Compiler Generation: Automatically generates equals(), hashCode(), toString(), copy(), and componentN() destructuring functions.
  • Rules: Primary constructor must contain at least one parameter, and all parameters must be declared as val or var. Data classes cannot be marked as abstract, open, sealed, or inner.
How does the copy() method behave in a data class containing a nested mutable list?
Answer
The generated copy() method performs a **shallow copy**. It creates a new instance of the data class, but passes the exact same object reference addresses for nested structures. Mutating the nested list in the copied instance will affect the original instance. For deep copies, you must construct a custom copy logic.
Senior-Level Follow-up
What happens if you define a custom equals() method in a data class? (The compiler uses the custom implementation, ignoring primary constructor properties for equality checks).
Common Mistake
Using mutable lists inside a data class, which breaks equals() and hashCode() properties if mutated.

4. Sealed Classes vs Enums Classes

Quick Cheat Sheet

  • Sealed Classes: Closed type hierarchies representing restricted class trees. Subclasses can have multiple instances, each containing unique state and constructor parameters.
  • Enums: Restricted constant value hierarchies. Every enum value is a singleton instance without dynamic constructor state.
  • When Blocks: The compiler enforces exhaustive checks on both sealed classes and enums, avoiding the need for an else branch.
When should you choose a sealed class over an enum?
Answer
Use an enum if you represent a fixed set of static values (like days of the week or booking types). Use a sealed class when subclasses need to maintain dynamic, runtime state (like representing UI states: Success(val hotels: List) vs Error(val exception: Throwable)).
Senior-Level Follow-up
How does the Kotlin compiler generate sealed classes under the hood on JVM 17+? (Uses Java's native sealed class keyword for validation).
Common Mistake
Declaring stateless sealed subclasses as class instead of object, resulting in redundant instance allocations.

5. Object & Companion Objects Classes

Quick Cheat Sheet

  • object: Declares a singleton instance class. Initialized lazily upon first access.
  • companion object: Tied to the enclosing class. Initialized when the enclosing class is loaded by the JVM ClassLoader.
  • Java Interop: Access using `Class.Companion.method()` in Java, or annotate with @JvmStatic to access them as standard static methods.
Is a companion object initialized lazily or eagerly?
Answer
A companion object is initialized when the enclosing class is loaded and initialized by the JVM ClassLoader (typically when the class is first accessed or instantiated). This behaves eagerly relative to class access, unlike a standalone object singleton which is resolved lazily on first direct reference call.
Senior-Level Follow-up
How does the @JvmStatic annotation affect JVM class bytecode structure?
Common Mistake
Assuming companion objects act as native static members in Java without annotations (they are compiled as inner class singletons under the hood).

6. Collections: ArrayList vs HashMap Internals Collections

Quick Cheat Sheet

  • ArrayList: Backed by a dynamic array. Default capacity is 10. Resizes by allocating a new array of 1.5x capacity and copying elements when full. Access: O(1), Insertion: O(1) amortized.
  • HashMap: Backed by an array of buckets (nodes). Resolves index using hashCode() % capacity. Uses linked lists for collision resolution, converting to red-black trees (treeifying) when a bucket contains 8+ items (threshold of 8) and capacity is 64+.
  • Mutable vs Immutable: Kotlin enforces read-only interfaces (List) at compile time. At runtime, these compile to standard mutable JVM collections, meaning they can be cast back if not protected.
What happens internally when a HashMap encounters hash collisions?
Answer
When keys produce the same bucket index, the nodes are chained together in a linked list. If the number of collisions in a bucket exceeds 8, and the total map capacity is 64+, Java 8+ transforms the linked list into a red-black tree (self-balancing binary search tree), optimizing lookup time complexity from O(N) to O(log N).
Senior-Level Follow-up
What is the load factor of a HashMap, and how does it determine rehashing cycles?
Common Mistake
Assuming Kotlin's standard read-only collections (List) are immutable. (They are read-only views of mutable lists and can still be modified at runtime).

7. Extension Functions Functions

Quick Cheat Sheet

  • Static Resolution: Extension functions do not modify the receiver class. They compile to Java static methods that accept the receiver object as the first parameter.
  • Polymorphism: Because they are static methods under the hood, they are resolved statically at compile time based on the reference type, not polymorphic runtime dispatch.
Why are extension functions resolved statically instead of dynamically?
Answer
Because they compile to standard Java static methods. The compiler determines which method to call based on the declared static type of the reference at compile time. If a base class and a subclass both define an extension with the same name, calling it on a base-class reference points to the base class extension, even if the runtime object is a subclass.
Senior-Level Follow-up
What happens if an extension function has the same signature as a member function? (The member function always takes priority).
Common Mistake
Expecting extension functions to support polymorphic overrides.

8. High-Order Functions & inline, crossinline, noinline Functions

Quick Cheat Sheet

  • inline: Tells the compiler to copy the function body and lambda arguments directly to the call site, eliminating class allocation and lambda execution overhead.
  • crossinline: Restricts the lambda argument from using non-local returns (e.g. `return` to exit the enclosing function). Necessary when the lambda is executed inside a nested context.
  • noinline: Prevents inlining of specific lambda parameters, treating them as standard Function objects (useful if you need to store or pass the lambda).
What is a non-local return, and how does inline enable it?
Answer
A non-local return is a return statement inside a lambda that exits the outer function calling the lambda. This is only possible because the compiler copies the lambda body directly to the call site when the function is marked as `inline`. Non-inlined lambdas cannot use non-local returns.
Senior-Level Follow-up
Explain the bytecode file-size growth trade-off of using `inline` on large functions.
Common Mistake
Marking small functions that do not accept lambdas as `inline` (which increases bytecode size without improving performance).

9. Reified Type Parameters Generics

Quick Cheat Sheet

  • Type Erasure: Generics are erased at runtime on the JVM, meaning you cannot access class info using T::class.java directly.
  • reified: Paired with inline. Keeps type information accessible at runtime by replacing generic references with the concrete class literal directly at the inlined call site.
Why do reified parameters require inline functions?
Answer
Because the JVM erases generic type parameters at compile-time. When a function is marked as inline and its type as reified, the compiler copies the function body to the call site, where the concrete type is known. The compiler replaces the generic T with the actual class reference (e.g. String::class.java), preserving type info.
Senior-Level Follow-up
Can you invoke a function containing a reified parameter from Java? (No, because Java does not support Kotlin-style function inlining).
Common Mistake
Attempting to use `reified` parameters in standard, non-inline functions.

10. Property Delegation & by lazy Delegation

Quick Cheat Sheet

  • by lazy: Computes values only on first access, caching the result. Default mode is thread-safe.
  • Thread Safety Modes:
    • LazyThreadSafetyMode.SYNCHRONIZED: Double-checked locking. Only one thread can initialize.
    • LazyThreadSafetyMode.PUBLICATION: Multiple threads can initialize, but only the first returned result is saved.
    • LazyThreadSafetyMode.NONE: No lock checks. Fast, but unsafe for concurrent access.
What are the three lazy initialization modes in Kotlin?
Answer
1. SYNCHRONIZED: Uses double-checked locking to guarantee only one thread initializes the property. 2. PUBLICATION: Allows multiple threads to call the initializer block, but only the first completed value is cached and returned. 3. NONE: Bypasses thread synchronization, providing the fastest access but risking race conditions if accessed concurrently.
Senior-Level Follow-up
How does the ObservableProperty delegate help track changes to properties in ViewModels?
Common Mistake
Using the default `SYNCHRONIZED` mode for UI-bound properties accessed only from the main thread, introducing unnecessary lock overhead.

11. Generics: Covariance & Contravariance Generics

Quick Cheat Sheet

  • Covariance (out T): Restricts the type to read-only positions. Preserves subtyping: `Producer` matches `Producer`. (Equivalent to `? extends T` in Java).
  • Contravariance (in T): Restricts the type to write-only positions. Reverses subtyping: `Consumer` matches `Consumer`. (Equivalent to `? super T` in Java).
  • Star Projection (*): Used when you don't know or care about the type. Restricts writes (accepts nothing) but allows reading `Any?`.
Explain covariance and contravariance in Kotlin.
Answer
Covariance (out T) allows you to read values from a class but prevents writing them. It preserves the subtyping relationship: a collection of dogs can be assigned to a reference expecting a collection of animals. Contravariance (in T) is the opposite: you can write values into the class but cannot read them, reversing subtyping relationships.
Senior-Level Follow-up
How does declaration-site variance in Kotlin compare to use-site variance in Java?
Common Mistake
Declaring a parameter as invariant when it is only used as a producer (read-only), which unnecessarily limits caller compatibility.

12. Scope Functions Basics

Quick Cheat Sheet

Function Context Object Return Value Typical Use Case
let it Lambda result Null check wrappers and object mappings
run this Lambda result Initialization and computation in one block
with this Lambda result Grouping member function calls on an object
apply this Context object Object configuration (builder pattern)
also it Context object Side effects, logging, and validations
Compare let and run in terms of context access and return type.
Answer
Both return the lambda result. The difference is how they access the context object: let accesses the object via the implicit parameter it, while run accesses the object via this, allowing you to call members directly without the object reference prefix.
Senior-Level Follow-up
What is the performance overhead of using scope functions? (None, they are inline functions, compiling to flat, standard sequential bytecode).
Common Mistake
Nesting multiple `let` blocks, which makes code difficult to read (use helper functions or clear parameter naming instead).
