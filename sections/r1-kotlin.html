1. Variables: val, var, and const val Basics

Definition

Kotlin variables are declared using val (read-only reference) and var (mutable reference). Compile-time constants are declared using const val inside object declarations or top-level files.

Syntax

val name: String = "Expedia"
var counter: Int = 0
const val API_TIMEOUT_MS: Long = 5000L

Internal Working

val variables compile to final private fields with a getter method. var variables compile to non-final private fields with both getter and setter methods. const val variables are completely inlined by the compiler at their usage sites, leaving no field lookup in the compiled bytecode.

JVM Generated Code

For a class containing val name: String, the compiler generates a private final field and a public getter:

// Decompiled Java representation
private final String name = "Expedia";
public final String getName() { return this.name; }

Memory Implications

Using val does not make objects immutable; it only makes the reference immutable. If the reference points to a mutable object (like a list), the object's properties can still be modified, maintaining allocations on the heap.

Performance Considerations

Inlining `const val` values avoids method call lookup and field access overhead. Read-only variables (`val`) help JIT compilers perform optimization analysis safely.

Real Example

// Read-only reference to mutable list
val selectedHotels = mutableListOf("Sheraton")
// selectedHotels = mutableListOf("Marriott") // Error: Reference cannot be reassigned
selectedHotels.add("Marriott") // OK: Internal list content can change
Interview Question
Explain the differences between val, var, and const val in terms of compiler behavior.
Answer
val creates a read-only variable represented as a final field with a getter. var creates a mutable variable with both getter and setter. const val represents a compile-time constant that is completely inlined at the bytecode level, avoiding execution overhead.
Senior Follow-Up
What happens when you declare a custom getter on a val property? Does it allocate field memory?
Common Candidate Mistake
Assuming that declaring a list as val prevents its contents from being modified.

2. Nullable Types & Null Safety Type System

Definition

Kotlin separates references into non-nullable types (cannot hold null) and nullable types (can hold null), enforcing safety checks at compile time.

Syntax

val nonNull: String = "Expedia"
val nullable: String? = null
val length = nullable?.length ?: 0 // Safe call & Elvis operator
val forced = nullable!!.length // Non-null assertion (risk of NPE)

Internal Working

Kotlin's null safety is enforced at compile time. At runtime, Kotlin's type system compiles down to Java types, using standard JVM references. The compiler inserts null check assertions (`Intrinsics.checkNotNullParameter`) at the entry points of public API methods to catch null references passed from Java code.

JVM Generated Code

A non-null parameter method generates runtime assertion code in Java:

// Decompiled Java representation
public final void processHotel(String hotel) {
    Intrinsics.checkNotNullParameter(hotel, "hotel");
    // Method body
}

Memory Implications

Nullable values of primitive types (like `Int?`) force boxing (e.g. compiling to `java.lang.Integer` on the JVM), which increases memory footprint by allocating objects on the heap.

Performance Considerations

Safe calls (`?.`) and Elvis operators (`?:`) compile to simple jump instructions (comparable to Java if-null checks) without any function call overhead.

Real Example

fun displayHotelRating(rating: Double?): String {
    // Elvis operator fallback to default rating representation
    val safeRating = rating ?: 0.0
    return "Rating: $safeRating"
}
Interview Question
How does Kotlin implement null safety, and how does it interoperate with Java?
Answer
Kotlin enforces null safety at compile-time by distinguishing nullable and non-nullable types. When compiling, it inserts null checks (`Intrinsics.checkNotNull`) to prevent null values from propagating. When calling from Java, platform types are used, which are not checked unless annotated with `@Nullable` or `@NonNull`.
Senior Follow-Up
Explain what platform types are and how they affect null safety checks during compilation.
Common Candidate Mistake
Using the non-null assertion operator (!!) to bypass compile-time errors instead of handling null checks gracefully, leading to NullPointerExceptions.

3. Kotlin Type System: Any, Unit, Nothing, and Casts Type System

Definition

Kotlin has a unified type hierarchy. Any is the root of the hierarchy. Unit corresponds to void in Java. Nothing represents a type that has no instances (signaling code execution paths that never return).

Syntax

fun logError(msg: String): Unit { println(msg) }
fun throwException(): Nothing { throw RuntimeException("Fatal") }
val value: Any = "Test"
val casted: String? = value as? String // Safe Cast

Internal Working

Any maps to java.lang.Object on the JVM. Unit compiles to a singleton object instance. Nothing compiles to a private constructor class to prevent instantiation, and methods returning Nothing compile to a Java void return type.

JVM Generated Code

A smart cast: if (x is String) x.length compiles to a Java type cast check without redundant allocations:

// Decompiled Java representation
if (value instanceof String) {
    int length = ((String)value).length();
}

Memory Implications

Smart casts occur without memory overhead. Returning `Unit` does not allocate memory on subsequent calls because the compiler uses a static singleton instance reference (`Unit.INSTANCE`).

Performance Considerations

Safe casts (`as?`) combine type checks (`instanceof`) and cast instructions in a single step, preventing runtime `ClassCastException` overhead.

Real Example

fun processInput(input: Any) {
    // Smart casting automatically converts input to String inside the block
    if (input is String) {
        println("String length: ${input.length}")
    }
}
Interview Question
What is the difference between Unit and Nothing in Kotlin?
Answer
Unit is a real type that represents a single value (void return type). Nothing is a subtype of all other types, containing no values. It is used to signal that a function never returns or always throws an exception.
Senior Follow-Up
How does smart casting work in Kotlin, and what are the limitations with open or mutable properties?
Common Candidate Mistake
Attempting to smart cast a mutable property (`var`), which fails because the value can be modified by another thread between the check and usage.

4. Classes, Constructors, and Inheritance OOP

Definition

Kotlin classes encapsulate state and behavior. By default, Kotlin classes are closed for extension (`final`), requiring the `open` modifier to support inheritance.

Syntax

open class BaseHotel(val id: String) {
    init { println("Base initialized") }
}
class LuxuryHotel(id: String, val rating: Int) : BaseHotel(id)

Internal Working

Primary constructors compile directly into class initializer blocks. Init blocks are executed sequentially in order of definition during class construction. Subclass constructor calls invoke the parent constructor via the superclass instruction (`super(...)`).

JVM Generated Code

A class declared in Kotlin is translated to a Java class. By default, it is final:

// Decompiled Java representation
public final class LuxuryHotel extends BaseHotel {
   private final int rating;
   public LuxuryHotel(String id, int rating) {
      super(id);
      this.rating = rating;
   }
}

Memory Implications

Instantiating a subclass allocates a single contiguous block of memory on the heap representing both the parent and child properties, avoiding redundant allocations.

Performance Considerations

Since classes are `final` by default, compiler and JIT analysis can devirtualize method calls, optimizing execution times.

Real Example

open class Resource(val path: String)

class ImageResource(path: String, val width: Int, val height: Int) : Resource(path) {
    init {
        require(width > 0 && height > 0) { "Dimensions must be positive" }
    }
}
Interview Question
Why are Kotlin classes final by default, and how are init blocks executed?
Answer
Kotlin classes are final by default to enforce the principle of designing for inheritance or forbidding it explicitly. Init blocks compile directly into the primary constructor body, executing sequentially in the order they are defined.
Senior Follow-Up
What is the execution order of init blocks, property initializations, and secondary constructors?
Common Candidate Mistake
Accessing open class properties inside an `init` block of a base class. (If the subclass overrides the property, it will be uninitialized at the time of execution).

5. Data Classes Internals OOP

Definition

Data classes are designed to hold state. The compiler automatically generates boilerplate methods (equals, hashCode, toString, copy, componentN) based on properties declared in the primary constructor.

Syntax

data class Hotel(val id: String, val price: Int)

Internal Working

The compiler scans primary constructor parameters and generates matching methods inside the class file. Destructuring declarations compile to sequential method calls: `component1()`, `component2()`, and so on.

JVM Generated Code

The compiled class includes the auto-generated method implementations:

// Decompiled Java representation of componentN methods
public final String component1() { return this.id; }
public final int component2() { return this.price; }

Memory Implications

The `copy()` method performs a shallow copy of constructor properties. If a property is a mutable reference, both the original and copied instances will point to the same nested object in memory, which can lead to unexpected state updates.

Performance Considerations

Avoid using destructuring declarations in high-frequency loops or collection filters, as compiling multiple `componentN` calls can add method lookup overhead.

Real Example

val hotel1 = Hotel("hotel_01", 150)
val hotel2 = hotel1.copy(price = 180) // Clones properties, overriding 'price'
val (id, price) = hotel2 // Destructures using component1() and component2()
Interview Question
Explain how Kotlin implements destructuring declarations for data classes under the hood.
Answer
Kotlin destructuring relies on position-based componentN() methods generated by the compiler. The compiler maps destructuring variables to sequential component method calls on the source object. This position-based indexing can lead to bugs if the primary constructor properties are reordered.
Senior Follow-Up
How does the auto-generated hashCode() method handle nullable parameters inside data classes?
Common Candidate Mistake
Assuming that data class `copy()` performs a deep copy of nested mutable collections.

6. Sealed Classes & Closed Hierarchies OOP

Definition

Sealed classes represent restricted class hierarchies, restricting inheritance to subclasses declared within the same module. This allows the compiler to perform exhaustiveness checks in conditional blocks.

Syntax

sealed class UiState {
    object Loading : UiState()
    data class Success(val items: List) : UiState()
    data class Error(val msg: String) : UiState()
}

Internal Working

The compiler generates a private constructor on the sealed class, preventing instantiation from outside the package. It maintains a list of subclasses in the class file metadata, which is used during compilation to verify exhaustiveness in `when` expressions.

JVM Generated Code

Sealed classes compile to abstract classes with package-private constructors:

// Decompiled Java representation
public abstract class UiState {
   private UiState() {} // package-private constructor prevents external inheritance
}

Memory Implications

Declare stateless subclasses as object instead of class. This ensures the compiler creates a singleton instance, preventing redundant allocations on the heap.

Performance Considerations

Exhaustiveness checks are completed at compile time, eliminating the performance overhead of default else branches at runtime.

Real Example

fun render(state: UiState) = when (state) {
    is UiState.Loading -> showSpinner()
    is UiState.Success -> displayList(state.items)
    is UiState.Error -> showError(state.msg)
    // No else branch required because the list of subclasses is closed
}
Interview Question
How does the compiler enforce exhaustiveness for sealed classes in when expressions?
Answer
The compiler reads the subclass metadata stored in the sealed class declaration. If the when expression is used as a statement (which requires a return value), the compiler verifies that all subclasses are handled, throwing a compiler error if any subclass is missing.
Senior Follow-Up
What is the difference between a sealed class and a sealed interface in Kotlin 1.5+?
Common Candidate Mistake
Adding a redundant default `else` branch to when statements evaluating sealed classes, which silences compiler warnings when a new subclass is added.

7. Objects: Singletons and Expressions OOP

Definition

The object keyword defines a singleton class declaration. It can also be used as an expression (creating anonymous inner classes on the fly).

Syntax

object DatabaseConfig { val host = "localhost" }
val listener = object : View.OnClickListener {
    override fun onClick(v: View) { println("Clicked") }
}

Internal Working

Singleton objects use a static constructor initialization pattern. The instance is constructed when the class is first loaded by the JVM, guaranteeing thread safety during initialization.

JVM Generated Code

The compiler generates a static final field named INSTANCE inside the singleton class:

// Decompiled Java representation
public final class DatabaseConfig {
   public static final DatabaseConfig INSTANCE;
   private DatabaseConfig() {}
   static {
      INSTANCE = new DatabaseConfig();
   }
}

Memory Implications

A singleton object is kept in memory for the duration of the process lifecycle. Anonymous object expressions allocate a new object on the heap for each execution of the expression block.

Performance Considerations

Accessing properties on a singleton object involves a direct static field lookup, which is fast and does not require instantiation overhead.

Real Example

object UserSession {
    var userId: String? = null
    fun clear() { userId = null }
}
Interview Question
Explain the thread-safety guarantees of Kotlin singleton object initialization.
Answer
Kotlin singleton objects are thread-safe because they compile down to static blocks. The JVM guarantees that static initializers are executed sequentially and only once when the class is loaded, ensuring thread-safe instantiation.
Senior Follow-Up
What is the difference between object declarations and object expressions regarding initialization timing?
Common Candidate Mistake
Storing references to short-lived contexts (like Activities) inside a singleton object, causing memory leaks.

8. Companion Objects OOP

Definition

A companion object is an object nested inside a class. It can access the private members of the enclosing class, serving as a replacement for static methods in Java.

Syntax

class User private constructor(val name: String) {
    companion object {
        fun create(name: String) = User(name)
    }
}

Internal Working

The companion object compiles to a nested static class. The enclosing class contains a static reference to this inner class instance, which is resolved when the enclosing class loads.

JVM Generated Code

The compiler generates a static field named Companion on the outer class:

// Decompiled Java representation
public final class User {
   public static final User.Companion Companion = new User.Companion((DefaultConstructorMarker)null);
   
   public static final class Companion {
      public final User create(String name) { return new User(name); }
   }
}

Memory Implications

Companion objects are instantiated when the outer class is loaded, allocating space for the static inner class instance on the heap.

Performance Considerations

Using `@JvmStatic` annotations instructs the compiler to generate static helper methods directly on the outer class, avoiding companion method lookup overhead when called from Java.

Real Example

class BookingToken {
    companion object {
        private const val PREFIX = "EXP-"
        fun generate(id: String) = "$PREFIX$id"
    }
}
Interview Question
Do companion objects compile to static methods? How can you call them from Java code?
Answer
No. They compile to nested singleton objects. To call them from Java, you must reference the Companion instance: OuterClass.Companion.method(). To generate static helper methods on the outer class directly, annotate the companion method with @JvmStatic.
Senior Follow-Up
How does JvmStatic annotation affect bytecode size and method dispatching?
Common Candidate Mistake
Assuming that companion object methods are resolved as native Java static methods without using annotations.

9. Functions: Named Arguments, Defaults, and Infix Functions

Definition

Kotlin supports advanced function declarations, including named arguments, default parameter values, variable-length arguments (`varargs`), and custom `infix` notations.

Syntax

fun sendRequest(url: String, timeout: Long = 2000L) { ... }
infix fun String.onto(other: String) = Pair(this, other)
// Usage: "Expedia" onto "Developer"

Internal Working

Default parameters are compiled using a synthetic bitmask parameter. When the caller omits arguments, the method checks the bitmask and falls back to the default values. `infix` methods compile to standard virtual functions that take exactly one parameter.

JVM Generated Code

A method with default parameters generates a synthetic method to resolve arguments:

// Decompiled Java representation
public static void sendRequest$default(String url, long timeout, int mask, Object marker) {
   if ((mask & 2) != 0) {
      timeout = 2000L; // Default fallback
   }
   sendRequest(url, timeout);
}

Memory Implications

Using default parameters compiles additional synthetic helper methods, slightly increasing class metadata size on disk but avoiding local variable allocations at runtime.

Performance Considerations

Infix methods have the same performance characteristics as standard virtual function calls, offering clean syntax without adding runtime overhead.

Real Example

class QueryBuilder {
    private val filters = mutableMapOf()
    
    // Infix method allows for natural DSL declarations
    infix fun String.eq(value: String) {
        filters[this] = value
    }
}
Interview Question
How are default parameter values compiled to Java bytecode, and how does Java access them?
Answer
The compiler generates a synthetic method containing a bitmask parameter. When calling from Java, all arguments must be passed unless the Kotlin method is annotated with @JvmOverloads, which instructs the compiler to generate overloaded versions of the method.
Senior Follow-Up
What are the limitations of JvmOverloads when applying it to constructor declarations?
Common Candidate Mistake
Assuming that Java clients can call methods with default parameters without applying the `@JvmOverloads` annotation.

10. Higher-Order Functions Functions

Definition

A higher-order function is a function that takes one or more functions as parameters, or returns a function as its result.

Syntax

fun executeTask(action: () -> Unit) {
    println("Starting task")
    action()
}

Internal Working

Higher-order functions compile to JVM types using the FunctionN interface hierarchy (e.g. Function0, Function1). The lambda parameter is represented as an object instance of these interfaces at runtime.

JVM Generated Code

A lambda parameter is passed as a Function instance reference:

// Decompiled Java representation
public final void executeTask(Function0 action) {
   System.out.println("Starting task");
   action.invoke(); // Invokes lambda execution
}

Memory Implications

Passing non-inlined higher-order lambdas allocates a new `Function` object instance on the heap for each invocation, increasing garbage collection pressure.

Performance Considerations

Non-inlined higher-order functions introduce invocation and object allocation overhead. This can be optimized by applying the `inline` keyword to copy the lambda body directly to the call site.

Real Example

fun calculateDiscount(price: Double, formula: (Double) -> Double): Double {
    return formula(price)
}
Interview Question
What is the runtime representation of a Kotlin higher-order function parameter on the JVM?
Answer
They compile to instance implementations of the FunctionN interface hierarchy (such as Function0 or Function1). The lambda body is executed by calling the interface's invoke() method on the instance.
Senior Follow-Up
How does the compiler handle variables captured inside non-inlined lambda closures?
Common Candidate Mistake
Failing to inline higher-order functions that are executed frequently (such as inside collections loops), causing high garbage collection load.

11. Lambdas & Closures Functions

Definition

Lambdas are function literals that can be passed as arguments or stored in variables. A closure is a block of code that can access and modify variables declared in its outer scope.

Syntax

val printMessage = { msg: String -> println(msg) }
var callCount = 0
val counter = { callCount++ } // Closure capturing mutable outer variable

Internal Working

Lambdas that do not capture outer variables compile to static singleton instances to prevent redundant allocations. Lambdas that capture variables (closures) instantiate a new `Function` object for each execution, wrapping captured primitives in mutable wrapper objects.

JVM Generated Code

A captured mutable reference (like `var count = 0`) is wrapped in a helper object reference:

// Decompiled Java representation of captured var
final Ref.IntRef countRef = new Ref.IntRef();
countRef.element = 0;
Function0 lambda = new Function0() {
   public Object invoke() {
      countRef.element++; // Modifies wrapped element property
      return null;
   }
};

Memory Implications

Capturing variables in closures allocates wrapping reference classes (like `IntRef`), increasing heap utilization and memory pressure.

Performance Considerations

Capturing variables in closures creates wrapping objects, which can cause garbage collection overhead if executed frequently. Use stateless lambdas or inline functions where possible to avoid allocations.

Real Example

fun createCounter(): () -> Int {
    var localCounter = 0
    return { ++localCounter } // Closure captures and retains 'localCounter' reference
}
Interview Question
How does Kotlin implement variable capture in closures at the JVM level?
Answer
For immutable variables, the compiler passes their values as constructor arguments to the generated lambda class. For mutable variables, it wraps the variable in a helper reference object (such as Ref.ObjectRef or Ref.IntRef), allowing the lambda to read and write to the wrapped value.
Senior Follow-Up
Explain the memory lifecycle of variables captured inside a returned lambda closure.
Common Candidate Mistake
Modifying captured variables inside concurrent, non-thread-safe lambdas, leading to race conditions.

12. Inline Functions & Bytecode Injection Compiler

Definition

Marking a function as inline instructs the compiler to copy the function body and its lambda parameters directly to the call site, avoiding function call and object allocation overhead.

Syntax

inline fun  lock(body: () -> T): T {
    acquireLock()
    try { return body() } finally { releaseLock() }
}

Internal Working

During compilation, the compiler inserts the bytecode of the inline function and its lambda parameters directly into the call site. This inline replacement allows for non-local returns (exiting the calling function directly from the lambda).

JVM Generated Code

The call site compiles directly without generating `Function` objects or virtual call instructions:

// Decompiled Java representation at call site
acquireLock();
try {
   // Lambda contents are compiled directly inline here
} finally {
   releaseLock();
}

Memory Implications

Eliminates the allocation of intermediate `Function` instances on the heap, reducing garbage collection overhead during high-frequency execution.

Performance Considerations

Avoid inlining large functions that do not take lambda parameters, as copying the body to multiple call sites increases compiled bytecode size (code bloat).

Real Example

inline fun measureTime(block: () -> Unit) {
    val start = System.nanoTime()
    block()
    println("Time taken: ${System.nanoTime() - start} ns")
}
Interview Question
How does the 'inline' keyword affect performance and bytecode size?
Answer
Inlining eliminates lambda object allocations and virtual call overhead by copying the function body directly to the call site. However, inlining large functions with multiple call sites can significantly increase the compiled bytecode size (code bloat).
Senior Follow-Up
How do non-local returns work in inline functions compared to standard nested loops?
Common Candidate Mistake
Inlining large helper methods that do not take lambdas, causing unnecessary bytecode bloat.

13. Reified Type Parameters Generics

Definition

Marking a generic type parameter as reified preserves type information at runtime inside inline functions, bypassing JVM generic type erasure.

Syntax

inline fun  checkType(item: Any): Boolean {
    return item is T // Valid: T is reified
}

Internal Working

During function inlining, the compiler replaces the generic type placeholder with the concrete class reference (e.g. `String::class.java`) at the call site. This type parameter substitution is only possible because the function is inlined.

JVM Generated Code

The generic check compiles directly to an instanceof check at the call site:

// Kotlin call: checkType("Expedia")
// Decompiled Java representation at call site
boolean result = "Expedia" instanceof String;

Memory Implications

Preserves type checks without the memory overhead of passing reflection class instances (`Class`) manually.

Performance Considerations

Compiled type checks run at native execution speed, avoiding reflection overhead during runtime type checks.

Real Example

inline fun  List.filterType(): List {
    return this.filter { it is T }.map { it as T }
}
Interview Question
Why does 'reified' require the function to be marked 'inline'?
Answer
Because the JVM uses type erasure, which removes generic parameters at compile time. By inlining the function, the compiler copies the body to the call site and replaces the generic parameter placeholder with the concrete type class literal, bypassing type erasure. This replacement is only possible during function inlining.
Senior Follow-Up
Can you access reified type parameters from Java code? Why or why not?
Common Candidate Mistake
Attempting to call reified functions from Java code, which is not supported because Java does not support Kotlin's compile-time inlining.

14. Class Delegation OOP

Definition

Class delegation redirects interface methods to a specified delegate object using the by keyword, implementing the composition over inheritance pattern cleanly.

Syntax

interface Printer { fun print() }
class ConsolePrinter : Printer { override fun print() = println("Print") }
class Document(printer: Printer) : Printer by printer // Delegates to printer

Internal Working

The compiler generates stub implementations for all interface methods inside the delegating class. These stub methods forward calls directly to the backing delegate field initialized in the constructor.

JVM Generated Code

The compiler generates composition fields and stub methods:

// Decompiled Java representation of composition delegation
public final class Document implements Printer {
   private final Printer $$delegate_0;
   public Document(Printer printer) {
      this.$$delegate_0 = printer;
   }
   public void print() {
      this.$$delegate_0.print(); // Forwards call
   }
}

Memory Implications

The delegating class instance holds a reference to the delegate object, slightly increasing reference memory allocation on the heap.

Performance Considerations

Interface method forwarding introduces a minor redirection overhead, which is optimized by JIT method devirtualization at runtime.

Real Example

class CustomList(private val innerList: List) : List by innerList {
    // Override specific methods while delegating the rest to innerList
    override fun isEmpty(): Boolean = innerList.size == 0
}
Interview Question
Explain the internal working of class delegation (using the 'by' keyword) at the JVM level.
Answer
The compiler generates a private backing field to store the delegate instance. It then generates stub implementations for all interface methods that forward calls directly to the backing delegate instance, implementing composition automatically.
Senior Follow-Up
What happens if you override a delegated interface method in the delegating class?
Common Candidate Mistake
Assuming that class delegation creates a subtyping relationship between the delegating class and the delegate class.

15. Property Delegates: Custom, Lazy, Observable, Vetoable Delegation

Definition

Property delegation redirects property read/write access calls to a helper delegate class using by.

Syntax

val lazyValue: String by lazy { "Expedia" }
var observedPrice: Int by Delegates.observable(100) { _, old, new -> ... }

Internal Working

The compiler generates a private backing field to store the delegate class instance and references it inside the getter/setter methods. It also creates a static `KProperty` metadata reference to describe the delegated property.

JVM Generated Code

The compiler generates getter/setter redirections and a metadata field descriptor:

// Decompiled Java representation of property delegate
private final Lazy lazyValue$delegate = LazyKt.lazy(action);
public final String getLazyValue() {
   return (String)this.lazyValue$delegate.getValue();
}

Memory Implications

Delegating properties allocates the helper delegate objects (such as `SynchronizedLazyImpl`) on the heap. This can add up if used for multiple properties in large classes.

Performance Considerations

Lazy initialization thread safety (`SYNCHRONIZED`) uses locks, which can add overhead. Use `NONE` mode if initialization is guaranteed to run on a single thread (like the Android main thread).

Real Example

class HotelSearch {
    // Only initialized when first accessed
    val resultsCache by lazy(LazyThreadSafetyMode.NONE) {
        loadHeavyCache()
    }
}
Interview Question
Explain the double-checked locking implementation inside Kotlin's default SynchronizedLazyImpl.
Answer
The default by lazy mode (SYNCHRONIZED) uses double-checked locking on a synchronization lock object. It checks the value reference, synchronizes on the lock if uninitialized, checks the value again, executes the lambda, and stores the result, ensuring thread safety.
Senior Follow-Up
What are the differences in memory overhead between LazyThreadSafetyMode.SYNCHRONIZED and NONE?
Common Candidate Mistake
Using the default synchronized lazy delegate for local variables or UI-thread bound properties, adding unnecessary lock overhead.

16. Extension Functions and Properties OOP

Definition

Extensions define functions or properties on a class without modifying its source code. Extensions are resolved statically and do not modify the target class.

Syntax

fun String.removeSpaces() = this.replace(" ", "")
val Int.isEven: Boolean get() = this % 2 == 0

Internal Working

Extensions compile directly to Java static methods where the receiver class instance is passed as the first parameter. Because they compile to static methods, extensions are resolved statically at compile time, not polymorphically at runtime.

JVM Generated Code

An extension property compiles to a Java static method taking the receiver instance:

// Decompiled Java representation of extension property getter
public static final boolean isEven(int $receiver) {
   return $receiver % 2 == 0;
}

Memory Implications

Extensions compile to static methods and do not allocate additional heap memory on the receiver instances.

Performance Considerations

Resolves as a static method call, avoiding virtual dispatch dispatching table lookup overhead, which is fast.

Real Example

// Statically resolved extension
open class BaseItem
class HotelItem : BaseItem()

fun BaseItem.name() = "Base"
fun HotelItem.name() = "Hotel"

val item: BaseItem = HotelItem()
println(item.name()) // Prints "Base" because extensions are resolved statically
Interview Question
Why are Kotlin extension functions not polymorphic? Prove it using static resolution rules.
Answer
Because they compile to static JVM methods. If an extension function is defined for both a base class and a subclass, the version that executes is determined by the static compiler type of the variable reference, not the concrete runtime class of the receiver.
Senior Follow-Up
What happens if you define an extension function with the same signature as a member function of a class?
Common Candidate Mistake
Expecting an extension function to override a member function with the same signature (member functions always take precedence).

17. Generics: Parameterized Types and Bounds Generics

Definition

Generics define parameterized types, allowing classes and functions to operate on different types with type safety. Generic parameter types can be restricted using bounds (e.g. `T : Number`).

Syntax

class Box(val item: T)
fun  calculateSum(a: T, b: T): Double { ... }

Internal Working

The JVM uses type erasure, which removes generic parameters at compile time. At runtime, the type parameter `T` is replaced by its upper bound (e.g. `java.lang.Object` or `java.lang.Number`), and the compiler inserts type casts where necessary.

JVM Generated Code

The generic class compiles to standard non-generic JVM bytecode with type casts:

// Decompiled Java representation showing type erasure
public final class Box {
   private final Object item; // T is erased to Object
   public Box(Object item) { this.item = item; }
}

Memory Implications

Generic parameters compile to object references, forcing primitive types (such as `Int`) to be boxed into their wrapper objects, which increases memory footprint.

Performance Considerations

Boxing and unboxing primitive generic parameters introduces allocation and conversion overhead. Use arrays or specialized non-generic collections for performance-critical tasks.

Real Example

class Repository {
    private val store = mutableListOf()
    fun add(item: T) { store.add(item) }
}
open class Entity(val id: String)
Interview Question
Explain what type erasure is on the JVM and how it affects Kotlin Generics.
Answer
Type erasure is a compiler process that removes generic parameter types at compile time to maintain compatibility with older JVM versions. The type parameter is replaced by its upper bound (or Object), and the compiler inserts casts at call sites to guarantee type safety.
Senior Follow-Up
How does Kotlin use type bounds (e.g. out and in) to implement declaration-site variance?
Common Candidate Mistake
Attempting to perform type checks (like `is T`) inside a non-inline generic function, which fails because the type parameter is erased at runtime.

18. Generics Variance (out, in, * ) Generics

Definition

Variance describes how subtyping relationships of generic parameters translate to subtyping relationships of the parameterized types (covariance, contravariance, and invariance).

Syntax

interface Producer { fun produce(): T } // Covariant
interface Consumer { fun consume(item: T) } // Contravariant
val list: List<*> = listOf(1, "Text") // Star Projection

Internal Working

Kotlin variance is resolved at compile time. out restricts type parameters to read-only (producer) positions. in restricts type parameters to write-only (consumer) positions. The compiler generates wildcards (`? extends T` and `? super T`) in the compiled JVM class signatures.

JVM Generated Code

The compiler generates Java wildcard signatures to support variance interoperability:

// Decompiled signature equivalent
public interface Producer {
   T produce(); // Generates wildcard signatures in dependent classes
}

Memory Implications

Variance parameters are evaluated at compile time and do not add any additional memory overhead at runtime.

Performance Considerations

Allows for generic class reuse and correct subtyping, avoiding the need for manual type casting or data duplication at runtime.

Real Example

open class Vehicle
class Car : Vehicle()

fun copyVehicles(from: List, to: MutableList) {
    for (item in from) {
        to.add(item) // Valid: covariant source, contravariant destination
    }
}
Interview Question
Differentiate between out and in variance keywords, providing equivalent Java wildcard representations.
Answer
out T represents covariance (equivalent to Java's ? extends T), restricting type parameters to producer positions to preserve subtyping. in T represents contravariance (equivalent to Java's ? super T), restricting type parameters to consumer positions to invert subtyping.
Senior Follow-Up
Why does the Kotlin compiler restrict 'out' type parameters from being passed as arguments to class methods?
Common Candidate Mistake
Failing to apply variance to read-only collections, which prevents subclasses from being passed to functions that read data safely.

19. Coroutines Mechanics & Suspension Coroutines

Definition

Coroutines are light-weight cooperative threads. They allow for non-blocking asynchronous code execution by suspending and resuming tasks without blocking the underlying thread.

Syntax

suspend fun fetchPrice(): Int {
    delay(1000) // Suspension point
    return 100
}

Internal Working

At compile time, the compiler transforms suspend functions using Continuation-Passing Style (CPS), adding a `Continuation` parameter and altering the return type to `Any?`. Execution states are managed using an internal state machine (switch statement) inside the generated `SuspendLambda` class.

JVM Generated Code

The compiled class inherits from `SuspendLambda` and manages execution states via a label-driven switch statement:

// Decompiled representation of state machine execution
public final Object invokeSuspend(Object $result) {
   switch(this.label) {
      case 0:
         this.label = 1;
         if (delay(1000L, this) == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED;
      case 1:
         return 100; // Returns computed value
   }
}

Memory Implications

Suspension points allocate state machine instances on the heap to store local variables and state. This can add memory pressure if suspension occurs frequently in tight loops.

Performance Considerations

Coroutines are more lightweight than threads. You can run thousands of concurrent coroutines on a small set of threads without the context-switching overhead of physical threads.

Real Example

import kotlinx.coroutines.*

fun main() = runBlocking {
    launch(Dispatchers.Default) {
        val result = fetchPrice()
        println("Price: $result")
    }
}
Interview Question
How does the Kotlin compiler implement suspend functions under the hood?
Answer
It applies CPS (Continuation-Passing Style) transformation. The signature is altered to add a Continuation parameter, and the return type is changed to Any?. The compiler strips the suspend keyword and generates an internal state machine to manage execution states across suspension points.
Senior Follow-Up
What is the role of the COROUTINE_SUSPENDED sentinel value returned by suspend functions?
Common Candidate Mistake
Assuming that suspend functions block thread execution like Java's `Thread.sleep()`.

20. Flows: Cold Asynchronous Streams Coroutines

Definition

Kotlin `Flow` represents a cold asynchronous stream of data that emits values sequentially and completes normally or with an exception.

Syntax

fun emitRates(): Flow = flow {
    emit(10)
    emit(20)
}

Internal Working

A Flow is a cold stream, meaning the producer block runs on-demand for each collector. Under the hood, collecting a flow is equivalent to executing the suspension block, passing a `FlowCollector` instance to receive emitted values.

JVM Generated Code

The flow block compiles down to a `SafeCollector` class that intercepts and validates calls to the `emit` method:

// Decompiled representation of Flow collection execution
public final Object collect(FlowCollector collector, Continuation completion) {
   // Executes producer block, forwarding emissions to collector.emit()
}

Memory Implications

Flow operators (like `map` or `filter`) wrap original flows in new flow definitions, which allocates small helper instances. This memory overhead is optimized by compiler inlining.

Performance Considerations

Use `flowOn` to configure execution threads for upstream operators. Collecting flows on the main thread is safe because upstream operations can run on background dispatchers.

Real Example

import kotlinx.coroutines.flow.*

suspend fun processFlow() {
    emitRates()
        .map { it * 2 } // Upstream operation
        .flowOn(Dispatchers.Default) // Configures upstream to run on background dispatcher
        .collect { println(it) } // Collector runs on the calling thread context
}
Interview Question
What does it mean for a Flow to be cold, and how does flowOn work?
Answer
A Flow is cold because the producer block runs on-demand for each collector. The flowOn operator changes the coroutine context for *upstream* operators only, spawning a separate coroutine to emit values while the collector runs on the caller thread context.
Senior Follow-Up
How does SafeCollector guarantee execution thread safety in custom flow builders?
Common Candidate Mistake
Changing the dispatcher inside a flow block using `withContext`, which violates Flow context preservation rules and causes runtime crashes (use `flowOn` instead).

21. Channels Coroutines

Definition

Channels provide a hot point-to-point communication channel to transfer streams of values between coroutines, behaving like a blocking queue.

Syntax

val channel = Channel(capacity = 10)
channel.send(1) // Suspends if the channel buffer is full
val value = channel.receive() // Suspends if the channel is empty

Internal Working

Channels manage a queue of elements along with lists of suspended send and receive continuations. If the queue is empty, the receiver suspends and its continuation is added to the receiver queue. When a new element is sent, the channel resumes the suspended receiver with the value.

JVM Generated Code

Channel operations compile to interface calls that access synchronized lock variables inside the channel execution implementation:

// Conceptual decompiled JVM execution snippet
public Object send(Object element, Continuation completion) {
   // Acquires channel queue lock, inserts element, or resumes suspended receiver
}

Memory Implications

Channels hold references to suspended continuations and buffered elements. Make sure to close channels when they are no longer needed to prevent memory leaks.

Performance Considerations

Choosing a rendezvous channel (capacity = 0) forces synchronization between sender and receiver, whereas buffered channels allow senders to continue executing without suspending immediately.

Real Example

import kotlinx.coroutines.channels.Channel
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun testChannel() = runBlocking {
    val channel = Channel()
    launch {
        channel.send("Ping")
    }
    println(channel.receive()) // Receives and prints "Ping"
}
Interview Question
What is the difference between a Channel and a Flow in Kotlin?
Answer
A Flow is a cold stream that runs its producer block on-demand for each collector. A Channel is a hot queue that runs independent of consumers. Senders and receivers transfer values concurrently, and elements are consumed only once.
Senior Follow-Up
Under what conditions does a Channel send operation suspend?
Common Candidate Mistake
Using a standard Channel to broadcast events to multiple active observers, which causes events to be split among them randomly.

22. DSLs (Domain Specific Languages) Advanced

Definition

Kotlin DSLs allow you to build custom, type-safe builders using language features like extension literals and function types with receivers.

Syntax

class HTML { fun body() { ... } }
fun html(init: HTML.() -> Unit): HTML {
    val html = HTML()
    html.init() // Executes with HTML as the receiver
    return html
}
// Usage: html { body() }

Internal Working

Function types with receivers (`HTML.() -> Unit`) compile to standard higher-order function signatures (`Function1`). The compiler passes the receiver instance as the first argument, allowing properties to be accessed using the implicit `this` keyword inside the block.

JVM Generated Code

The compiler translates the receiver function type to a standard function type:

// Decompiled Java representation of receiver block execution
public static final HTML html(Function1 init) {
   HTML html = new HTML();
   init.invoke(html); // Passes html as the receiver argument
   return html;
}

Memory Implications

DSLs nesting can create short-lived builder objects, which increases garbage collection churn if executed frequently. Use `@DslMarker` annotations to prevent outer receiver access leaks.

Performance Considerations

Mark DSL helper methods as `inline` to eliminate lambda allocations and direct virtual call overhead during builder execution.

Real Example

class HotelQuery {
    var city: String = ""
    var guests: Int = 1
}

fun query(init: HotelQuery.() -> Unit): HotelQuery {
    return HotelQuery().apply(init)
}

// Type-safe query building
val myQuery = query {
    city = "Seattle"
    guests = 2
}
Interview Question
How does Kotlin implement type-safe builders, and what is the role of function types with receivers?
Answer
Kotlin type-safe builders use function types with receivers (e.g. Receiver.() -> Unit). This compiles to a function signature that takes the receiver instance as the first parameter, allowing the block to access receiver properties implicitly using the this keyword.
Senior Follow-Up
What is the purpose of the @DslMarker annotation in nested builder scopes?
Common Candidate Mistake
Failing to apply `@DslMarker` to nested DSL classes, allowing inner builders to implicitly access properties of outer builders.

23. Reflection in Kotlin Advanced

Definition

Reflection is a set of library APIs that allows you to inspect class structures, properties, functions, and annotations at runtime.

Syntax

val kClass: KClass = Hotel::class
val members = kClass.members // Lists properties and functions
val constructor = kClass.constructors.first()

Internal Working

Kotlin reflection utilizes backing metadata annotations (`@Metadata`) generated by the compiler. It reads these annotations at runtime to reconstruct Kotlin-specific structures (like nullability or data class properties) that are otherwise erased in JVM class files.

JVM Generated Code

Class files include a `@Metadata` annotation containing encoded binary data representing Kotlin types, nullability properties, and method parameters:

// Decompiled class annotation header
@Metadata(
   mv = {1, 9, 0},
   d1 = {"...encoded Kotlin types data..."},
   d2 = {"id", "price", "Hotel", ...}
)
public final class Hotel { ... }

Memory Implications

Using Kotlin reflection loading metadata libraries (such as `kotlin-reflect.jar`) adds around 2MB-3MB to the application package size. It also allocates multiple metadata wrappers at runtime, increasing memory footprint.

Performance Considerations

Reflection method lookups and property updates are slow and bypass compiler optimization checks. For performance-critical code paths, use direct method calls or code-generation libraries (like KSP) instead.

Real Example

import kotlin.reflect.full.*

fun printClassMembers(instance: Any) {
    val kClass = instance::class
    println("Class: ${kClass.simpleName}")
    for (prop in kClass.memberProperties) {
        println("Property: ${prop.name} = ${prop.call(instance)}")
    }
}
Interview Question
How does Kotlin reflection read Kotlin-specific types (like nullability) that are not natively supported by Java reflection?
Answer
Kotlin reflection reads the binary metadata stored in the @Metadata annotation generated by the compiler. This annotation contains encoded strings representing Kotlin-specific properties, nullability flags, and parameter names that are otherwise erased in JVM class files.
Senior Follow-Up
What is the performance overhead difference between using Java reflection and Kotlin reflection APIs on Android?
Common Candidate Mistake
Using Kotlin reflection APIs frequently in performance-critical code paths (like JSON parsers or view binders), causing layout lag.

24. Annotations Advanced

Definition

Annotations associate metadata with code declarations (classes, functions, parameters). They can be processed at compile time or read at runtime using reflection APIs.

Syntax

@Target(AnnotationTarget.CLASS, AnnotationTarget.FUNCTION)
@Retention(AnnotationRetention.RUNTIME)
annotation class ExpediaApi(val endpoint: String)

Internal Working

Annotations compile down to Java annotation interfaces. The compiler embeds annotation values in the class file attributes. Depending on the retention policy, the JVM class loader loads this metadata, making it accessible to reflection APIs at runtime.

JVM Generated Code

The annotation compiles to a public interface extending `java.lang.annotation.Annotation`:

// Decompiled Java representation of annotation
public @interface ExpediaApi {
   String endpoint();
}

Memory Implications

Runtime-retained annotations add attributes data to class files in memory. Compile-time annotations (`SOURCE` or `BINARY` retention) are discarded after compilation, leaving no memory footprint at runtime.

Performance Considerations

Compile-time annotations processed using KSP or annotation processors generate code before compilation, avoiding reflection overhead and improving runtime performance.

Real Example

@ExpediaApi("/v2/hotels")
class HotelFetcher {
    @Target(AnnotationTarget.VALUE_PARAMETER)
    annotation class QueryParam(val name: String)
    
    fun getHotels(@QueryParam("city") cityCode: String) { ... }
}
Interview Question
Explain the differences between SOURCE, BINARY, and RUNTIME annotation retention policies in Kotlin.
Answer
SOURCE annotations are discarded by the compiler. BINARY annotations are stored in the class files but are not loaded by the JVM class loader. RUNTIME annotations are loaded by the JVM and are accessible via reflection APIs at runtime.
Senior Follow-Up
How do compile-time annotations processed via KSP differ from runtime annotations?
Common Candidate Mistake
Setting the default retention policy to RUNTIME when compiling annotation-based processors that only need to process metadata at compile time.
