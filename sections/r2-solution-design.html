Problem 1: Offline-First Architecture (Hotel Search) Offline-First

Requirements

  • Functional: Render cached results instantly. Fetch background updates. Invalidate cache after 30 minutes.
  • Non-Functional: Seamless UI updates without screen flickers. Low battery overhead.

Architecture Diagram

[Search UI] <-- (Collect Flow) -- [Repository] --> [Local Cache (Room DB)]
                                       |
                                       +-- (Fetch API) --> [Retrofit Network Client]

Trade-offs & Scalability

DB-as-SSOT: Writing network updates to the local database before emitting to the UI ensures data consistency across sessions, at the cost of slight disk write latency. Index queries on fields like price and ratings to optimize database scans as data scales.

How do you structure the repository to support offline-first updates?
Answer
Return a Flow that emits cached values from the local database immediately. Launch a background task to fetch fresh data via the API, write it to the database on success, and handle network errors gracefully by emitting an error state or showing a warning indicator in the UI.
Common Mistake
Emitting network results directly to the UI without writing them to the database first, violating the single source of truth principle.

Problem 2: Real-Time Price Updates Real-Time

Requirements

  • Functional: Update hotel prices in real-time. Disconnect connection when in background.
  • Non-Functional: Reconnect within 1 second of returning to screen. Exponential backoff retry.

Architecture Diagram

[UI View] <-- (State Flow) -- [ViewModel] <-- (WebSocket) -- [WebSocket Manager (OkHttp)]

Trade-offs & Scalability

WebSockets vs SSE: WebSockets enable bi-directional communication, ideal for client heartbeats. Server-Sent Events (SSE) are unidirectional and consume fewer resources, but do not support client-to-server pings. Exponential backoff prevents overloading the server during reconnection storms.

Design a reconnection strategy for real-time WebSockets.
Answer
Use an exponential backoff strategy (e.g. retry after 1s, 2s, 4s, up to 32s). Monitor network connectivity changes using ConnectivityManager to trigger an immediate reconnection attempt when the device goes back online. Disconnect the socket when the app enters the background to save battery.
Common Mistake
Keeping the WebSocket connection active when the app is in the background, violating battery optimization guidelines.

Problem 3: Pagination & List Caching Pagination

Requirements

  • Functional: Scroll through large lists smoothly. Cache loaded pages in SQLite.
  • Non-Functional: Prevent duplicate network requests. Smooth scroll at 60fps.

Architecture Diagram

[RecyclerView UI] <-- [Pager (Paging 3)] -- [RemoteMediator] --> [Room Database cache]
                                                    |
                                                    +-- (Fetch API) --> [Retrofit]

Trade-offs & Scalability

Using Jetpack Paging 3 with RemoteMediator simplifies caching and loading states but introduces architecture complexity. Store pagination keys explicitly in a mapping table to avoid duplicate items when items are updated on the server.

Explain the role of RemoteMediator in Paging 3.
Answer
RemoteMediator acts as a boundary callback. When the database runs out of cached items, it fetches the next page from the network and saves it to the database, which then updates the UI.
Common Mistake
Using raw database offsets for paging, which causes duplicate items if items are deleted or added during pagination.

Problem 4: Analytics Event Pipeline Pipeline

Requirements

  • Functional: Queue events locally. Upload in batches (10 events or 30 seconds).
  • Non-Functional: Guaranteed delivery for high-priority events. Limit database size to 50MB.

Architecture Diagram

[UI click] --> [AnalyticsManager] --> [Room Database Queue] --> [WorkManager Upload]

Trade-offs & Scalability

Writing events to a database avoids data loss if the app crashes but adds minor disk write overhead. Batching saves battery by reducing cellular radio activation cycles. Prune the database by deleting the oldest low-priority events if it exceeds size limits.

Why is a local database queue preferred over in-memory buffers?
Answer
In-memory queues lose all logged events if the application crashes, runs out of memory, or is killed by the OS. Writing events to a local Room database queue ensures data persistence across sessions, allowing the data to be uploaded once the app is restarted and connection is restored.
Common Mistake
Uploading events individually as they occur, which drains the device's battery by keeping the network radio active continuously.

Problem 5: Push Notification Routing System

Requirements

  • Functional: Receive FCM messages. Route to target screen. Handle silent payload updates.
  • Non-Functional: Comply with Android notification channel requirements. Low latency.

Architecture Diagram

[FCM Service] --> [NotificationRouter] --> [DeepLinkRegistry] --> [Target Activity]
                         | (Silent data sync)
                         +--> [WorkManager SyncTask]

Trade-offs & Scalability

Decouple notification handling from specific activities by routing deep link URIs dynamically. Use WorkManager to process data updates in background sync notifications to respect the OS service execution limits.

How do you handle background data sync notifications?
Answer
Do not perform heavy synchronization tasks directly inside the Firebase service, as this can trigger service timeouts. Instead, parse the payload, extract synchronization tags, and schedule a background task via WorkManager to run the sync task safely.
Common Mistake
Attempting to launch activities directly from background notification services on newer Android versions, which is blocked by the OS.

Problem 6: Real-Time Chat System Chat

Requirements

  • Functional: Bi-directional messaging. Message delivery states (sent, delivered, read). Offline history synchronization.
  • Non-Functional: Connection latency below 2 seconds. Messages must be delivered in order.

Architecture Diagram

[Chat UI] <-- (Collect) -- [MessageRepository] --> [Local Cache (Room DB)]
                                   |
                                   +-- (WebSocket) --> [OkHttp WebSocket Manager]

Trade-offs & Scalability

Using WebSockets enables real-time messaging but consumes more battery than HTTP polling. Encrypt messages in the local Room database using SQLCipher to secure personal chat logs.

How do you handle chat message synchronization after offline periods?
Answer
When connection is re-established, request delta synchronization from the server, passing the last synchronized message ID. Store incoming messages in the Room database, resolving duplicate payloads by deduplicating against the unique message ID.
Common Mistake
Displaying incoming messages in the UI without persisting them to the database first, leading to lost history if the app is closed.

Problem 7: BLE Device Synchronization System

Requirements

  • Functional: Scan for adjacent BLE devices. Read and write GATT characteristics.
  • Non-Functional: Handle connection drops. Duty-cycle scan execution to save battery.

Architecture Diagram

[BleDeviceManager] --> [GATT Queue handler] --> [BluetoothAdapter GATT Thread]

Trade-offs & Scalability

Scanning continuously returns immediate device updates but drains battery. Wrap the BLE scanning and GATT connection inside a Foreground Service to prevent the OS from terminating the synchronization task when the app is in the background.

Why are BLE operations thread-hostile, and how do you resolve it?
Answer
Android's BLE GATT operations are asynchronous and thread-hostile. Initiating overlapping reads or writes triggers state errors. To handle this, serialize all GATT commands in a sequential queue, executing them one at a time on a dedicated background HandlerThread or using Coroutine Mutex locks.
Common Mistake
Executing GATT callbacks directly on the Main Thread, leading to frame drops during heavy data transfers.

Problem 8: Image Loading & Memory Caching Engine Caching

Requirements

  • Functional: Fetch and decode images. Support placeholders and error assets.
  • Non-Functional: Scroll smoothly at 60fps/120fps. Prevent OOM errors.

Architecture Diagram

[UI ImageView] --> [ImageLoader] --> [Memory LruCache (Bitmaps)] --> [DiskLruCache]
                                                                        |
                                                                        +--> [Network]

Trade-offs & Scalability

Caching full-resolution bitmaps in memory avoids decoding overhead but consumes significant RAM. Downsample bitmaps using inSampleSize based on target view dimensions before loading them into memory. Use Bitmap pooling to reuse memory, reducing GC churn.

How does bitmap downsampling prevent OutOfMemory errors?
Answer
Loading a high-resolution image into memory consumes RAM based on dimensions (Width * Height * 4 bytes for ARGB_8888), not file size. Downsampling reads the image dimensions first using inJustDecodeBounds = true, calculates the ratio (sample size) matching the target view dimensions, and loads the resized image into memory, reducing RAM footprint by up to 90%.
Common Mistake
Loading full-resolution images into small thumbnail views, causing rapid memory growth and OOM crashes.

Problem 9: Background Data Synchronization Engine Sync

Requirements

  • Functional: Sync local database with server changes. Retry on connection failure.
  • Non-Functional: Respect device constraints (Wi-Fi, charging). Backoff retry policy.

Architecture Diagram

[WorkManager] -- (Check constraints) --> [SyncWorker] --> [SyncRepository] --> [Server API]

Trade-offs & Scalability

Running sync tasks inside a Foreground Service ensures immediate execution but is resource-intensive and restricted by the OS. WorkManager is deferrable but guarantees execution under device constraints, saving battery.

How do you implement transaction integrity during sync updates?
Answer
Perform database writes inside a Room Transaction block. If the synchronization task is cancelled (e.g. constraints are lost), the database transaction is rolled back, preventing partial or corrupted local state updates.
Common Mistake
Failing to handle task cancellation inside the worker's execution block, which keeps the background task running even after constraints are lost.
