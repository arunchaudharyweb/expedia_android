Problem 1: Offline-First Architecture (Hotel Search) Offline-First

Requirements

Functional: Render cached hotel search results instantly. Fetch background network updates. Cache specific booking pages for offline view.
Non-Functional: Seamless UI updates during background sync. Low battery impact. Cache expiration threshold at 30 minutes.

Architecture Diagram

+-------------+      Collects       +--------------------+
|  Search UI  | <------------------ |  HotelRepository   |
+-------------+                     +--------------------+
                                               |
                     +-------------------------+-------------------------+
                     | (Emit cached db first)                            | (Fetch network update)
                     v                                                   v
          +--------------------+                               +--------------------+
          | Room Database (DB) | <------- Writes Success ----- | Retrofit API Client|
          +--------------------+                               +--------------------+

Trade-offs

Writing network updates to the database before displaying them in the UI introduces minor write-latency. However, this enforces a single source of truth and prevents UI inconsistency.

Scalability Considerations

Query indexes must be placed on frequently filtered database fields (like price and rating) to keep search queries fast as the cached dataset grows.

Testing Strategy

Write integration tests using in-memory Room database instances to verify that calling the repository triggers both cache emission and network refresh.

Interview Question
How do you implement an offline-first cache invalidation policy?
Answer
Store a timestamp metadata field along with the cached data. When querying, check if the current system time exceeds the cached timestamp by the expiration threshold. If expired, emit the cached data with a "stale" flag while launching an immediate background network fetch to refresh the data.
Senior-Level Follow-up
How do you handle schema migrations for local offline databases without clearing user data?
Common Mistake
Directly updating UI state from network results without persisting them to the database, breaking offline support for subsequent sessions.

Problem 2: Pagination & List Caching Pagination

Requirements

Functional: Support incremental scrolling for search results. Cache pages locally in order. Reset pagination state on search query change.
Non-Functional: Maintain a 60fps/120fps scroll rate. Prevent duplicate network page requests under rapid scrolls.

Architecture Diagram

+------------------+     collect()      +--------------------+
|  RecyclerView UI | <----------------- |  Pager (Paging 3)  |
+------------------+                    +--------------------+
                                                  |
                         +------------------------+------------------------+
                         v RemoteMediator                                  v PagingSource
              +--------------------+                             +--------------------+
              | Room Database cache| <----- Network Fetch ------ | Retrofit API Client|
              +--------------------+                             +--------------------+

Trade-offs

Using Jetpack Paging 3 simplifies scroll state management but introduces architecture complexity (requires RemoteMediator). The alternative (manual scroll listeners) is simpler but prone to duplicate page load triggers.

Scalability Considerations

Store page indices and item ordering keys explicitly in a mapping table in Room. This avoids duplicate list items when items are updated or reordered on the server.

Testing Strategy

Test the RemoteMediator boundary class using mock API responses to verify that database insert actions are executed sequentially.

Explain the RemoteMediator interface in Jetpack Paging 3.
Answer
RemoteMediator handles loading data from the network when the database runs out of cached items. It receives pagination keys, requests the next page from the network, and writes it directly to the database. The database then updates the UI paging stream.
Senior-Level Follow-up
How do you implement jump-to-position mechanics in database-backed paging lists?
Common Mistake
Using raw database offsets for paging, which causes duplicate items if items are deleted or added during pagination.

Problem 3: Memory & Disk Caching Engine Caching

Requirements

Functional: Cache serialized HTTP responses and domain models. Implement a Least Recently Used (LRU) eviction strategy.
Non-Functional: Read latency below 5ms. Disk size limits must be respected to prevent app bloat.

Architecture Diagram

+------------------+                   +--------------------+
|  Repository Call | ----------------> |  CacheController   |
+------------------+                   +--------------------+
                                                 |
                       +-------------------------+-------------------------+
                       | (Fast path)                                       | (Slow path)
                       v                                                   v
            +--------------------+                               +--------------------+
            |  LruCache (RAM)    |                               |  DiskLruCache      |
            +--------------------+                               +--------------------+

Trade-offs

Memory-only caches are extremely fast but cleared when the process is killed. Adding a disk cache provides persistence across app launches at the cost of slight disk I/O overhead.

Scalability Considerations

Use a multi-level cache model: check RAM first, then Disk. Encrypt sensitive cached items (using EncryptedSharedPreferences or encrypted SQLite) to protect user credentials.

Testing Strategy

Write unit tests that verify eviction policies by adding items that exceed the cache size threshold and asserting that the oldest items are removed.

What is the difference between LruCache and DiskLruCache?
Answer
LruCache resides entirely in the application process heap memory, providing nanosecond-level access. DiskLruCache writes files to the device's internal storage, which is persistent but requires disk I/O, resulting in millisecond-level access times.
Senior-Level Follow-up
How do you configure cache-control headers in OkHttpClient to automate response caching?
Common Mistake
Reading from the disk cache on the main thread, causing frame drops and potential ANRs.

Problem 4: Push Notification Delivery System Notifications

Requirements

Functional: Receive Firebase Cloud Messages (FCM). Route notifications to target screens. Handle custom payloads.
Non-Functional: Decouple notification parsing from the UI. Support notification channels post Android 8.

Architecture Diagram

+------------------+                   +--------------------+
|  FCM Service     | ----------------> | NotificationRouter |
+------------------+                   +--------------------+
                                                 |
                       +-------------------------+-------------------------+
                       | (Silent / Data sync)                              | (Visual Alert)
                       v                                                   v
            +--------------------+                               +--------------------+
            | WorkManager task   |                               | NotificationManager|
            +--------------------+                               +--------------------+

Trade-offs

Processing payloads immediately inside FirebaseMessagingService is fast, but complex actions must be offloaded to WorkManager to comply with the OS's 10-second service execution limit.

Scalability Considerations

Use a centralized routing registry to resolve deep links dynamically, decoupling notification handlers from hardcoded activity destinations.

Testing Strategy

Mock the RemoteMessage class to verify that the router correctly parses payloads and creates the expected notification channels.

What are silent push notifications, and what are their execution limits?
Answer
Silent notifications contain only data payloads without displaying a user alert. They are designed to trigger background data sync. Under Android guidelines, they should not execute heavy tasks directly inside FCM services. Instead, enqueue a background task via WorkManager to perform the synchronization safely.
Senior-Level Follow-up
How do you build a secure PendingIntent for a notification action button post Android 12?
Common Mistake
Attempting to launch activities directly from background notification services on newer Android versions, which is blocked by the OS.

Problem 5: Chat & Messaging Application Chat

Requirements

Functional: Maintain a bi-directional messaging connection. Display message delivery states. Synchronize chat history offline.
Non-Functional: Reconnection latency below 2 seconds. Low battery overhead. Messages must be delivered in order.

Architecture Diagram

+------------------+                   +--------------------+
|     Chat UI      | <---------------- |  MessageRepository |
+------------------+   Flow State      +--------------------+
                                                 |
                       +-------------------------+-------------------------+
                       | (Persistent Cache)                                | (Bi-directional)
                       v                                                   v
            +--------------------+                               +--------------------+
            | Room Chat Database | <----- Sync writes ---------- |  WebSocket Manager |
            +--------------------+                               +--------------------+

Trade-offs

Using WebSockets enables real-time messaging but uses more battery. The alternative (long polling) is slow and increases server connection overhead.

Scalability Considerations

Encrypt messages in the local Room database using SQLCipher to prevent unauthorized access to personal chat logs.

Testing Strategy

Use a mock WebSocket server to verify reconnection logic, error handling, and message order synchronization during network drops.

How do you implement an exponential backoff reconnection strategy for chat WebSockets?
Answer
When the connection drops, schedule a retry starting with a 1-second delay. Double the delay on every subsequent failure (2s, 4s, 8s, up to a maximum of 32s). Listen to network connectivity changes to trigger an immediate retry when the device goes back online.
Senior-Level Follow-up
Explain how message deduplication is managed on the client when receiving duplicate socket payloads.
Common Mistake
Updating the chat UI directly from network events without persisting messages to the local database first.

Problem 6: BLE Device Synchronization System

Requirements

Functional: Scan for adjacent BLE devices. Establish connection and read characteristics. Persist sensor packets.
Non-Functional: Handle connection drops gracefully. Keep scanning overhead low to prevent battery drain.

Architecture Diagram

+------------------+                   +--------------------+
|  DeviceSync UI   | <---------------- |  BleDeviceManager  |
+------------------+   Status Flow     +--------------------+
                                                 |
                       +-------------------------+-------------------------+
                       | (Scan registry)                                   | (Hardware)
                       v                                                   v
            +--------------------+                               +--------------------+
            | Room Device DB     | <----- Write metrics -------- | Android Bluetooth   |
            +--------------------+                               | Adapter / GATT      |
                                                                 +--------------------+

Trade-offs

Scanning continuously returns immediate device updates but drains battery. The alternative (periodic duty-cycle scanning) saves battery but delays device detection.

Scalability Considerations

Wrap the BLE scanning and GATT connection inside a Foreground Service to prevent the OS from terminating the synchronization task when the app is in the background.

Testing Strategy

Decouple GATT interactions behind an interface. Inject fake GATT objects in unit tests to verify state transitions and packet parsing logic.

Why are BLE operations notoriously difficult to manage on Android, and how do you handle threading?
Answer
BLE GATT operations are asynchronous and thread-hostile. Initiating overlapping reads or writes triggers state errors. To handle this, serialize all GATT commands (discover services, read, write) in a sequential queue, executing them one at a time on a dedicated background HandlerThread or using Coroutine Mutex locks.
Senior-Level Follow-up
How does Android's BLE MTU size negotiation affect data transmission speed?
Common Mistake
Executing GATT callbacks directly on the Main Thread, leading to frame drops during heavy data transfers.

Problem 7: Image Loading & Caching Engine Image Loading

Requirements

Functional: Fetch and decode remote images. Display placeholders and error assets. Support image downsampling.
Non-Functional: Scroll smoothly at 60fps/120fps. Prevent memory leaks from view reference recycling.

Architecture Diagram

+------------------+                   +--------------------+
|  ImageView / UI  | ----------------> | ImageLoaderEngine  |
+------------------+                   +--------------------+
                                                 |
                       +-------------------------+-------------------------+
                       | (Fast Cache)                                      | (Slow Cache / Network)
                       v                                                   v
            +--------------------+                               +--------------------+
            | LruCache (Bitmaps) | <----- Fetch network -------- | DiskCache / OkHttp |
            +--------------------+                               +--------------------+

Trade-offs

Caching full-resolution bitmaps avoids decoding overhead but consumes significant memory. Caching compressed image bytes saves memory but requires on-the-fly decoding, increasing CPU utilization.

Scalability Considerations

Use Bitmap pooling (reusing Bitmap memory via inBitmap) to reduce garbage collector overhead and prevent GC-induced frame drops during fast scrolls.

Testing Strategy

Test the cache resolution logic using mock web responses, verifying that requests for cached images bypass network calls.

Explain how bitmap downsampling prevents OutOfMemory errors.
Answer
Loading a high-resolution image into memory consumes RAM based on dimensions (Width * Height * 4 bytes for ARGB_8888), not file size. Downsampling reads the image dimensions first using inJustDecodeBounds = true, calculates the ratio (sample size) matching the target view dimensions, and loads the resized image into memory, reducing RAM footprint by up to 90%.
Senior-Level Follow-up
Explain how modern libraries (like Coil or Glide) track view recycling to cancel active image requests.
Common Mistake
Loading full-resolution images into small thumbnail views, causing rapid memory growth and OOM crashes.

Problem 8: Background Data Sync Engine Sync

Requirements

Functional: Sync local database with the server in the background. Respect charging and network constraints.
Non-Functional: Support retry backoff. Run task even if the app or device restarts.

Architecture Diagram

+------------------+                   +--------------------+
|  SyncTrigger     | ----------------> |    WorkManager     |
+------------------+                   +--------------------+
                                                 |
                       +-------------------------+-------------------------+
                       | (Check Constraints)                               | (Job Execution)
                       v                                                   v
            +--------------------+                               +--------------------+
            | SQLite DB state    | <----- Read offline state --- |    SyncWorker      |
            +--------------------+                               +--------------------+

Trade-offs

Running sync tasks inside a Foreground Service ensures immediate execution but is resource-intensive and restricted by the OS. WorkManager is deferrable but guarantees execution under device constraints, saving battery.

Scalability Considerations

Implement incremental synchronization (delta sync) using change tracking tokens to upload only modified data instead of the entire database.

Testing Strategy

Use WorkManager's WorkManagerTestInitHelper to simulate constraint changes (such as losing connection) and verify task rescheduling behavior.

How does WorkManager handle task cancellation when constraints are lost during execution?
Answer
When constraints are lost, WorkManager triggers cooperative cancellation. The worker's coroutine scope is cancelled, throwing a CancellationException. The task is then stopped and rescheduled to run again once constraints are met.
Senior-Level Follow-up
How do you implement transaction integrity when writing synced data to the Room database?
Common Mistake
Failing to handle task cancellation inside the worker's execution block, which keeps the background task running even after constraints are lost.

Problem 9: Analytics Event Pipeline Pipeline

Requirements

Functional: Queue UI interaction events (clicks, impressions) locally. Upload events in batches. Respect priority levels.
Non-Functional: Limit local database size. Exponential backoff retry policy.

Architecture Diagram

+------------------+                   +--------------------+
|    Tracker UI    | ----------------> |  AnalyticsManager  |
+------------------+                   +--------------------+
                                                 |
                                       +--------------------+
                                       | Room Event Queue   |
                                       +--------------------+
                                                 |
                                       +--------------------+
                                       | WorkManager Upload |
                                       +--------------------+

Trade-offs

Queueing events in a database prevents data loss but adds disk write latency. An in-memory queue is faster but loses events if the app crashes before upload.

Scalability Considerations

Limit the maximum database size (e.g. 50MB). If exceeded, prune the database by deleting the oldest low-priority events to prevent storage bloat.

Testing Strategy

Write unit tests to verify batch compilation, priority-based sorting, and database pruning logic.

Why is an offline database queue preferred over in-memory buffers for analytics events?
Answer
In-memory queues lose all logged events if the application crashes, runs out of memory, or is killed by the OS. Writing events to a local Room database queue ensures data persistence across sessions, allowing the data to be uploaded once the app is restarted and connection is restored.
Senior-Level Follow-up
How do you implement batch compression to reduce network bandwidth during event uploads?
Common Mistake
Uploading events individually as they occur, which drains the device's battery by keeping the network radio active continuously.
