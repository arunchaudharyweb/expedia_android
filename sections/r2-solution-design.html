Solution Design (Round 2 Architecture)

⏱ 25 min read ✓ Round 2 Core

Architecture designs for key travel application features. Focuses on offline-first synchronization, real-time streams, caching strategies, and data pipelines.

Problem 1: Offline-First Hotel Search Offline-First

Clarifying Questions:

  • Should cached search results be filterable while offline? (Yes, support basic text/price filtering).
  • How fresh should the cache be? (Invalidate or warn if cache is older than 30 minutes).

Functional Requirements:

  • Display cached hotel listings immediately upon opening the search screen.
  • Fetch fresh listings in the background and update the UI gracefully.
  • Allow users to view details of cached hotels while offline.

Non-Functional Requirements:

  • UI rendering must be responsive, avoiding frame drops during cache updates.
  • Optimize database writes to minimize battery impact.

Component Diagram:

+------------------+     Queries     +--------------------+
|    Search UI     | <------------>  |  HotelRepository   |
+------------------+                 +--------------------+
                                               |
                       +-----------------------+-----------------------+
                       |                                               |
                       v                                               v
            +--------------------+                           +--------------------+
            | Room Database (DB) |                           | Retrofit API Client|
            +--------------------+                           +--------------------+

Data Flow Walkthrough:

  1. The Search UI subscribes to the HotelRepository data stream (a Flow of lists of hotels).
  2. The repository fetches cached data from the Room database, emitting it immediately to update the UI.
  3. At the same time, the repository triggers a network request via Retrofit to fetch fresh listings.
  4. When the network request succeeds, the repository writes the new data to Room, which automatically triggers a new emission to update the UI.

Key Technology Choices:

  • Room Database: Serves as the single source of truth, supporting reactive queries via Coroutine Flows.
  • Retrofit: Standard library for network requests, integrating with Coroutine suspending calls.

Trade-offs:

  • UI Flickering: Updating the database immediately after a network request can cause the UI to flicker. Mitigated by using diffing utilities (like DiffUtil) in the UI adapters.

Failure Handling:

  • If the network request fails, show a snackbar warning that the UI is displaying offline cached data.

Testing Strategy:

  • Use in-memory Room databases for repository integration tests to verify cache-then-network emission sequences.

Expedia-Relevance Note:

Travelers often search for hotels while in transit or with weak airport Wi-Fi. An offline-first search ensures they can access pricing and booking details reliably under poor network conditions.

Interview Question
How do you structure a repository to emit cached data first, fetch fresh data, and handle network errors?
Answer
Return a Flow that emits cached values from the local database immediately. Launch a background task to fetch fresh data via the API, write it to the database on success, and handle network errors gracefully by emitting an error state or showing a warning indicator in the UI.
Senior Follow-Up
How do you handle conflict resolution when background sync conflicts with user modifications?
Common Candidate Mistake
Emitting network results directly to the UI without writing them to the database first, violating the single source of truth principle.

Problem 2: Real-Time Price Updates Real-Time

Clarifying Questions:

  • How frequently do prices change? (Prices update every few seconds during active booking sessions).
  • What is the background behavior requirement? (Disconnect connection when the app goes to the background to save battery).

Functional Requirements:

  • Update hotel prices in real-time on the details screen.
  • Display a visual indicator (such as a green flash) when a price changes.

Non-Functional Requirements:

  • Minimize battery usage and data consumption when the app is in the background.
  • Establish connection within 1 second of entering the screen.

Component Diagram:

+------------------+                 +--------------------+
|    Details UI    |                 |   PriceRepository  |
+------------------+                 +--------------------+
        ^                                      ^
        | State Flow                           |
+------------------+      Updates      +--------------------+
|    ViewModel     | <---------------- | WebSocket Manager  |
+------------------+                   +--------------------+

Data Flow Walkthrough:

  1. The Details UI opens, subscribing to the ViewModel's price state flow.
  2. The ViewModel requests price updates from the repository, which opens a WebSocket connection.
  3. The server broadcasts price updates over the WebSocket channel.
  4. The repository receives updates, parses the payloads, and updates the ViewModel state to refresh the UI.

Key Technology Choices:

  • WebSockets (OkHttp): Provides a persistent, low-overhead bi-directional communication channel for frequent updates.
  • Server-Sent Events (SSE): Alternative for unidirectional server-to-client updates. We choose WebSockets because it is widely supported and allows for client-side heartbeat signals.

Trade-offs:

  • Connection Overhead: Maintaining persistent WebSocket connections uses more battery than simple HTTP polling, but is more efficient than rapid polling when prices change frequently.

Failure Handling:

  • Implement an exponential backoff reconnection strategy. If the connection fails, fall back to occasional HTTP polling to ensure the user still sees updated prices.

Testing Strategy:

  • Use mock WebSocket servers to test connection lifecycles, reconnection logic, and behavior during network drops.

Expedia-Relevance Note:

Flight and hotel prices fluctuate frequently during booking. Real-time price updates prevent users from encountering price mismatches at checkout, increasing booking success rates.

Interview Question
Design a reconnection strategy for a WebSocket connection on a price tracking screen.
Answer
Use an exponential backoff strategy (e.g. retry after 1s, 2s, 4s, up to 32s). Monitor network connectivity changes using ConnectivityManager to trigger an immediate reconnection attempt when the device goes back online.
Senior Follow-Up
How do you handle stale pricing data in the UI if the connection drops?
Common Candidate Mistake
Keeping the WebSocket connection active when the app is in the background, violating battery optimization guidelines.

Problem 3: Pagination with Database Caching Pagination

Clarifying Questions:

  • Should search filter adjustments reset the pagination state? (Yes, new searches should invalidate the current pagination state).
  • How is end-of-list reached? (The API returns a null token or empty page).

Functional Requirements:

  • Support smooth scrolling through large lists of hotels.
  • Cache pages in a local database so users can scroll through loaded pages offline.
  • Display loading indicators at the end of the list during page fetches.

Non-Functional Requirements:

  • Prevent duplicate network requests for the same page.
  • Scroll smoothly at 60fps.

Component Diagram:

+------------------+                   +--------------------+
|  RecyclerView UI |                   |      PagingData    |
+------------------+                   +--------------------+
                                                 ^
+------------------+  RemoteMediator   +--------------------+
|   Room Cache     | <---------------- |    Pager Config    |
+------------------+                   +--------------------+
                                                 | API Call
                                       +--------------------+
                                       | Retrofit Service   |
                                       +--------------------+

Data Flow Walkthrough:

  1. The Paging library's Pager coordinates data loading using a RemoteMediator.
  2. When the database runs out of cached items, RemoteMediator fetches the next page from the network API.
  3. The fetched page data is written to the Room database.
  4. The database update triggers an emission to refresh the UI list.

Key Technology Choices:

  • Jetpack Paging 3: Coordinates data requests, handles caching, and simplifies page loading states.
  • Room + RemoteMediator: Implements a local database cache boundary for the Paging library.

Trade-offs:

  • Complexity: Setting up Paging 3 with Room is more complex than simple offset paging, but provides smoother scrolling and offline support.

Failure Handling:

  • If a page fetch fails, show a retry button at the end of the list, allowing the user to retry the request without losing their current scroll position.

Testing Strategy:

  • Write integration tests for the RemoteMediator to verify that page tokens are parsed correctly and saved to the database.

Expedia-Relevance Note:

Expedia hotel search results can return hundreds of listings. Efficient pagination reduces memory usage on mobile devices and limits server load.

Interview Question
Explain the role of RemoteMediator in Paging 3 and how it differs from a PagingSource.
Answer
PagingSource loads data directly from a single source (either network or database) for the UI. RemoteMediator acts as a boundary callback. When the database runs out of cached items, it fetches the next page from the network and saves it to the database, which then updates the UI.
Senior Follow-Up
How do you manage pagination keys in Room to prevent out-of-order page inserts?
Common Candidate Mistake
Using raw offset/limit pagination for dynamic search results. This can cause duplicate items to appear in the list if items are added or reordered during search.

Problem 4: Analytics Event Pipeline Pipeline

Clarifying Questions:

  • What is the batching threshold? (Upload events in batches of 10, or every 30 seconds).
  • How are critical events handled? (Mark events with priority levels; critical events are uploaded immediately).

Functional Requirements:

  • Queue user interaction events (clicks, impressions) in a local cache.
  • Upload events in batches to minimize network connection overhead.
  • Retry failed uploads using an exponential backoff policy.

Non-Functional Requirements:

  • Guaranteed delivery for high-priority event types.
  • Limit local cache size to prevent the app from consuming excessive storage.

Component Diagram:

+------------------+                   +--------------------+
|    Tracker UI    | ----------------> |  AnalyticsManager  |
+------------------+                   +--------------------+
                                                 |
                                       +--------------------+
                                       | Room Event Queue   |
                                       +--------------------+
                                                 |
                                       +--------------------+
                                       | WorkManager Upload |
                                       +--------------------+

Data Flow Walkthrough:

  1. The UI calls AnalyticsManager to log user events.
  2. The manager writes the events immediately to a Room database event queue.
  3. A background task is scheduled via WorkManager to process and upload the event queue.
  4. When the queue reaches the batch threshold or time limit, the worker uploads the events to the server and deletes them from the local database on success.

Key Technology Choices:

  • Room Database: Replaces fragile in-memory arrays to prevent data loss if the app crashes before events are uploaded.
  • WorkManager: Handles uploading events in the background, managing retries and network constraints automatically.

Trade-offs:

  • Real-time Latency vs Battery Life: Batching events saves battery but delays delivery. We mitigate this by supporting immediate uploads for high-priority events.

Failure Handling:

  • If the upload fails, retry using WorkManager's backoff policy. If the database exceeds its maximum size limit (e.g. 50MB), drop the oldest low-priority events to prevent storage issues.

Testing Strategy:

  • Write unit tests to verify batch splitting, priority filtering, and database pruning logic.

Expedia-Relevance Note:

Expedia uses analytics data to optimize booking funnels. A reliable event pipeline ensures business metrics are captured accurately, even under spotty network conditions.

Interview Question
Design a crash-resilient analytics upload pipeline that limits network calls.
Answer
Write events to a local Room database queue first. Use WorkManager to batch upload events when the device is connected to Wi-Fi or charging, retrying failed uploads using an exponential backoff policy to save battery.
Senior Follow-Up
Under what conditions would you drop events instead of retrying them indefinitely?
Common Candidate Mistake
Storing queued events in an in-memory list, causing events to be lost if the app crashes or is killed by the system.
