Section 0: Android Fundamentals

⏱ 25 min read ✓ Part 1 Core

Comprehensive guide covering essential and advanced fundamentals of Kotlin, Coroutines, Reactive paradigms, Jetpack Compose, Android Platform, and Hilt.

1. Kotlin Language Fundamentals Kotlin

Quick Cheat Sheet

  • Variables & Backing Fields: val (read-only, generates getter) vs var (mutable, generates getter and setter). Under the hood, a backing field (field) is generated to store the value unless the property has custom getters/setters that don't reference it.
  • Data Types & Null Safety: Primitive types (like Int, Boolean) are represented as JVM primitives unless boxed (nullable types like Int?, generics). Null safety is enforced at compile time using nullable types (T?), safe calls (?.), Elvis operator (?:), and non-null assertions (!!). Platform types (T!) originate from Java code without nullability annotations and bypass compile-time null checks, risking NullPointerException at runtime.
  • Class Hierarchies & Objects:
    • class: Final by default. Use open to allow subclassing.
    • data class: Automatically generates equals(), hashCode(), toString(), copy(), and componentN() functions. Requires at least one primary constructor parameter.
    • sealed class: Represents closed, restricted class hierarchies. All subclasses must be known at compile time, enabling exhaustive when checks.
    • object: Declares a thread-safe singleton. companion object defines static-like members tied to the enclosing class.
  • Collections Internals:
    • ArrayList: Resizable-array implementation. Backed by a JVM array that copies and grows by a factor of 1.5x when capacity is exceeded.
    • HashMap: Array of buckets (linked lists). When a bucket exceeds 8 nodes and total size is >= 64, the list converts to a balanced Red-Black Tree for O(log N) lookup instead of O(N).
    • LinkedHashMap: Extends HashMap but maintains a doubly-linked list running through all of its entries, preserving insertion or access order.
    • Mutable vs Immutable: Kotlin interfaces (like List vs MutableList) enforce read-only access at compile time, but at runtime, they compile to Java standard collections (e.g. java.util.ArrayList) which are mutable.
  • Functions & Lambdas:
    • Extension Functions: Static resolution. Resolved at compile-time as standard static methods receiving the receiver object as the first parameter.
    • Higher-Order Functions: Receive functions as parameters or return them, creating function instances (closures) that can allocate memory.
    • Inline: Tells the compiler to copy the function body and lambda argument bytecode directly into the call site, eliminating method call and closure allocation overhead.
    • Noinline / Crossinline: Use noinline to prevent inlining of a specific lambda parameter. Use crossinline to restrict the lambda from using non-local returns (exiting the enclosing function).
    • Reified Types: Access generic type parameters at runtime inside inline functions using inline fun , bypassing JVM type erasure.
  • Delegation & Scope Functions:
    • Delegation: Class delegation forwards interface implementations to an object. Property delegates (by lazy, by Delegates.observable) intercept reads/writes. lazy supports 3 thread-safety modes: SYNCHRONIZED (locked, default), PUBLICATION (multiple threads can execute, first returned wins), and NONE (no safety checks).
    • Generics: out T (Covariance) permits producing values (safe for read, subtypes assignable to supertypes). in T (Contravariance) permits consuming values (safe for write, supertypes assignable to subtypes).
    • Destructuring: Deconstructs an object into multiple variables using generated componentN() methods.
    • Scope Functions: let (returns result, refers as it), run (returns result, refers as this), with (returns result, takes parameter, refers as this), apply (returns context object, refers as this), also (returns context object, refers as it).
Why are extension functions resolved statically?
Answer
Kotlin resolves extension functions statically based on the declared receiver type at compile-time, not the runtime type. The compiler translates fun String.print() into a Java static method public static final void print(String $receiver). Because it does not modify the receiver class's actual class definition or inject methods into its virtual method table, polymorphism does not apply.
Senior-Level Follow-up
How does the by lazy SYNCHRONIZED mode avoid locking overhead after initialization? (It uses double-checked locking, checking if the value is initialized before acquiring the monitor lock).
Common Mistake
Assuming read-only Kotlin collections (like `List`) are thread-safe and immutable. They are compile-time read-only wrappers; if the underlying reference points to a mutable list, concurrent modifications can crash the app.

2. Coroutines & Concurrency Async

Quick Cheat Sheet

  • Processes vs Threads: A process is an isolated execution space containing its own memory Heap. A thread is the smallest scheduling unit within a process, sharing the heap but having its own stack.
  • Concurrency vs Parallelism: Concurrency is about dealing with multiple tasks at once (interleaving execution on a single core). Parallelism is executing multiple tasks simultaneously (on multiple CPU cores).
  • Coroutines Core: Coroutines are cooperative light-weight threads. They execute inside a CoroutineScope which tracks and manages their lifecycle via a CoroutineContext (comprising a Job, Dispatcher, and ExceptionHandler).
  • Builders & Scope: launch is fire-and-forget (returns a Job). async is for tasks that return a result (returns a Deferred). Job is hierarchical; cancelling a parent cancels all child jobs. SupervisorJob isolates children so that a failing child does not cancel the parent or siblings.
  • Under the Hood (Continuation CPS): Suspending functions do not block threads. The Kotlin compiler translates suspending functions into Continuation-Passing Style (CPS). The runtime passes an implicit Continuation callback parameter, compiling the code into a state machine. When execution suspends, the function saves its state variables, exits the call stack, and returns a special COROUTINE_SUSPENDED token, allowing the thread to do other work. Once the async task completes, the continuation calls resumeWith() to restore the state machine and continue execution.
  • Dispatchers:
    • Dispatchers.Main: Bound to the UI main loop thread (via Handler/Looper).
    • Dispatchers.IO: Elastic pool of threads (up to 64 or CPU cores) optimized for blocking network/disk operations.
    • Dispatchers.Default: Fixed thread pool (size matching CPU core count) optimized for CPU-intensive calculations.
    • Dispatchers.Unconfined: Starts executing the coroutine in the current thread context, suspending and resuming on whatever thread the suspension trigger executes.
  • Structured Concurrency & Errors: Guarantees that children complete before their parent exits. Cancellations propagate downwards. Uncaught exceptions propagate upwards to cancel the parent, unless using `SupervisorJob` or a scoped `supervisorScope`. Use a `CoroutineExceptionHandler` to catch unhandled exceptions at the root of a coroutine tree.
  • Flows, StateFlow, SharedFlow, Channels:
    • Flow: Cold reactive stream. Emits values only when collected.
    • StateFlow: Hot state-holder flow. Retains and replays the current value (requires initial value), ignoring duplicates.
    • SharedFlow: Hot event-broadcaster flow. Broadcasts events to all active collectors, useful for one-off commands (like navigation).
    • Channel: Hot queue-like communication pipeline. Transfer values between coroutines using `send` and `receive` semantics.
Explain how the Continuation state machine handles suspension without allocating a new thread.
Answer
The compiler transforms every suspending function into a class implementing Continuation. Inside, a state variable (usually an int label) tracks the execution progress. Each suspension point is a label case in a switch statement. When suspending, the local variables are saved to the continuation instance, the function returns COROUTINE_SUSPENDED, and the calling thread is freed. Upon resumption, the framework invokes resumeWith(), jumps back into the switch using the saved label, and restores the variables.
Senior-Level Follow-up
What is the difference between coroutine cancellation and thread interruption? (Cancellation is cooperative; the coroutine must check `isActive` or call a suspending function to cancel. Interruption is an OS/JVM signal).
Common Mistake
Swallowing exceptions inside `launch` or failing to provide a `CoroutineExceptionHandler` at the root scope, causing the application to crash.

3. Reactive Programming Paradigms Reactive

Quick Cheat Sheet

  • Cold vs Hot Streams:
    • Cold Streams (Flow): Independent streams that execute their producer code only when collection starts. Multiple collectors trigger separate independent executions.
    • Hot Streams (StateFlow/SharedFlow): Broadcast streams that emit values regardless of whether any collectors are active. They share a single execution source.
  • StateFlow vs SharedFlow: StateFlow holds a single state value and replays it to new collectors (acting as a state-holder). SharedFlow does not store a state (unless configured with a replay cache > 0) and broadcasts transit events (acting as an event emitter).
  • Operators:
    • map / filter: Standard transform and filter actions.
    • flatMapLatest: Cancels previous active flows when a new item is emitted, ideal for search queries.
    • flowOn: Configures the dispatcher context of the **upstream** stream source, leaving downstream collector Dispatchers unaffected.
  • Backpressure Concepts: Occurs when the producer emits items faster than the consumer can process them.
    • Suspension: Flow's natural backpressure mechanism. Emitters suspend until collectors finish.
    • buffer(): Buffers emitted items in a channel, allowing the producer to emit ahead of slow collectors.
    • conflate(): Discards intermediate values, sending only the latest state to a slow collector.
  • Flow vs LiveData: LiveData is Main-thread bound and aware of Android Lifecycles, stopping execution in background states. Flow is lifecycle-agnostic, pure Kotlin, supports extensive operators, and requires helpers like repeatOnLifecycle to behave safely in UI layers.
Why is flowOn only affecting upstream operators?
Answer
Kotlin Flow uses context preservation. The collection is executed in the context of the collector. To execute upstream operations in a different context, flowOn intercepts the stream, wraps the upstream in a channel, and runs it on a separate coroutine using the specified dispatcher. Downstream operators continue to execute in the collector's context.
Senior-Level Follow-up
What is the default replay cache configuration of StateFlow? (Replay size of 1, conflating duplicate state emissions).
Common Mistake
Using a SharedFlow with a replay count of 1 for one-time events, which leads to re-triggering actions (like navigation or toasts) on screen rotation.

4. Jetpack Compose & UI Architecture Compose

Quick Cheat Sheet

  • Composable Functions & Slot Table: Composable functions are annotated with @Composable. The Compose compiler transforms these into functions that receive a composer object and write UI structure parameters into a layout database called the Slot Table.
  • State & Remember:
    • State: An observable wrapper that registers recomposition scopes on read.
    • remember: Retains value across recomposition phases within the Slot Table.
    • rememberSaveable: Automatically saves the state inside the Bundle system to survive configuration changes and process death.
    • State Hoisting: Pattern of moving state up to make a composable stateless, promoting reusability and testability.
  • Side Effects:
    • LaunchedEffect: Runs a suspend block tied to a key. Re-launches the coroutine if the key changes, and cancels it when leaving composition.
    • DisposableEffect: Side effect that requires cleanup. Runs its block and executes `onDispose` when the key changes or the composable exits.
    • produceState: Converts non-Compose state sources (like a flow or callback) into a Compose state.
    • derivedStateOf: Calculates a state based on other Compose states, caching the calculation and only triggering updates when the computed output changes, preventing redundant recompositions.
  • Snapshot System: The database engine of Compose state. It tracks values at a specific point in time, allowing isolated reads and writes across threads (Multi-Version Concurrency Control). Commit actions schedule recomposition.
  • Performance: Stability & Skippability:
    • Stable / Immutable: Types where values are unchanging, or whose change notifications are visible to Compose. Compose skips rendering composables if their arguments are Stable and unchanged (Skippability).
    • Unstable Types: Classes (like standard Lists, or third-party classes) whose state changes cannot be tracked. Composables receiving unstable arguments cannot be skipped and must re-evaluate on every recomposition. Annotate with `@Stable` or `@Immutable`, or use Kotlin immutable collections to solve.
    • Deferring Reads: Defer reading state values by passing them inside lambdas (e.g. `Modifier.offset { IntOffset(...) }`) to bypass the composition phase, routing updates directly to the Layout/Draw phases.
What is the difference between remember(key) and derivedStateOf?
Answer
remember(key) recalculates its block every time the key changes. derivedStateOf listens to state changes inside its block, but only triggers recomposition if the *computed result* actually changes. Use derivedStateOf when a state changes frequently (like scroll positions) but you only want to recompose when a specific threshold condition is met (e.g., scroll offset > 100).
Senior-Level Follow-up
How does passing unstable collections (like List) in a composable parameter cause redundant recomposition, and how do you fix it? (Wrap them in stable models, use `@Stable` annotations, or use kotlinx immutable collections).
Common Mistake
Reading a rapidly changing scroll state directly inside the composition scope without wrapping it in a derivedStateOf or using a lambda modifier, leading to 60fps recompositions.

5. Android Platform & Framework Core Platform

Quick Cheat Sheet

  • Lifecycles:
    • Activity: onCreate -> onStart -> onResume -> onPause -> onStop -> onDestroy. Retains state bundle across process death via onSaveInstanceState().
    • Fragment: Has separate view and fragment lifecycles. View is destroyed when placed on backstack; always use viewLifecycleOwner for observers to prevent memory leaks.
    • Process & Application: Process termination is decided by the OS out-of-memory score (OOM ADJ). App lifecycles are initialized in a custom Application class.
  • App Startup Library: Provides a straightforward, performant way to initialize components at application startup. Instead of declaring separate ContentProviders (which add process startup overhead), it shares a single provider.
  • Handler, Looper, and MessageQueue: The main UI thread runs a loop managed by a Looper. It blocks on Linux epoll when empty. Tasks are enqueued into a MessageQueue. A Handler wraps message queue interactions to post runnables and schedule messages across threads.
  • Runtime & GC: Android Runtime (ART) compile tiers use Interpreter, JIT (run-time compile), and AOT (compiling during idle cycles using Baseline Profiles). GC uses concurrent copying collectors to reduce pause times.
  • IPC & Security: Binder IPC uses memory-mapped buffer mappings to copy data across processes. Transactions have a strict 1MB limit. PendingIntents grant security privileges and must declare FLAG_IMMUTABLE (API 31+) to avoid modification vulnerability exploits.
  • Background Work:
    • WorkManager: Standard for deferrable, guaranteed work. Persists tasks in a local SQLite database, scheduling execution via JobScheduler when constraints (network, charging) are met.
    • Services: Base framework components. Run on the main thread by default.
    • Foreground Services: Must declare specific types (API 34+) and display a persistent notification to prevent the OS from killing the process.
Why are ContentProviders considered expensive for library initialization?
Answer
Each ContentProvider declared in an app's manifest is instantiated on the main thread during app startup, before Application.onCreate() is invoked. This adds binding overhead and slows down launch times. The App Startup library resolves this by declaring a single ContentProvider that coordinates initialization order dynamically.
Senior-Level Follow-up
Explain the role of the epoll system call in the Android Main Looper loop. (It sleeps the thread on an event file descriptor, consuming zero CPU cycles until a new message is posted).
Common Mistake
Assuming that running code inside a standard Service moves execution off the main UI thread. (Services block the main thread by default).

6. Dependency Injection with Hilt DI

Quick Cheat Sheet

  • Hilt Core: Build-time dependency injection library built on top of Dagger. It automatically generates standard boilerplate code for scopes, lifecycles, and validation checking at compile-time.
  • Hilt Components & Lifecycles:
    • SingletonComponent: Tied to the Application class (retains `@Singleton` instances).
    • ActivityRetainedComponent: Survives Activity configuration changes.
    • ActivityComponent: Tied to the Activity lifecycle (retains `@ActivityScoped` instances).
    • ViewModelComponent: Tied to ViewModel scope (retains `@ViewModelScoped` instances).
  • Injection Types:
    • Constructor Injection: Preferred type. Annotating constructor with `@Inject` makes dependencies explicit and testable.
    • Field Injection: Injecting public fields annotated with `@Inject` in system-instantiated components (Activities, Fragments).
  • Modules: Hilt modules annotated with @Module and @InstallIn(...) bind interfaces or provide third-party objects.
    • @Provides: Written as static methods to create instances of classes you do not own (like Retrofit or Room databases).
    • @Binds: Abstract methods used to bind implementation classes to interfaces, reducing compiler boilerplate.
What is the difference between @Provides and @Binds in a Hilt module?
Answer
@Provides is a function that contains initialization code to build and return an object. It is used when building third-party objects or when instantiation logic is required. @Binds is an abstract function that simply maps an implementation type to an interface. @Binds is preferred because it generates no bytecode representation at compile-time, improving build speed.
Senior-Level Follow-up
How does the @InstallIn annotation ensure type-safety across different Dagger components?
Common Mistake
Injecting short-lived scoped dependencies (like `@ActivityScoped` objects) inside long-lived components (like `@Singleton` objects), causing memory leaks or scope crashes.
