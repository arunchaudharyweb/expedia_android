1. Variables, Data Types & Null Safety Kotlin

Concepts & Explanations

  • Variables (val vs var): val declares a read-only variable (generating only a getter under the hood); var declares a mutable variable (generating both getter and setter).
  • Data Types: Under the hood, standard types like Int, Double, and Boolean compile to JVM primitive types (like int, double) to avoid boxing overhead, unless they are nullable or used in generics.
  • Null Safety: Eliminates NullPointerException at compile-time. Kotlin differentiates between non-nullable types (cannot be null) and nullable types (declared with ?). Safely access members using safe-calls (?.), fallback default values using Elvis (?:), or force execution with the non-null assertion operator (!!).

Code Example

// val vs var & Data Types
val readOnlyName: String = "Expedia" // Read-only: compile-time constant reference
var mutableCounter: Int = 100 // Mutable: compiles to primitive 'int'

// Null Safety implementation
fun printUserLength(username: String?, defaultVal: String = "Guest") {
    // Safe Call Operator (?.)
    val length: Int? = username?.length
    
    // Elvis Operator (?:)
    val safeLength: Int = username?.length ?: defaultVal.length
    
    println("Length: $safeLength")
    
    // Non-null assertion (!!) - Throws NullPointerException if username is null
    if (username != null) {
        val forcedLength = username!!.length
    }
}
Does val guarantee immutability in Kotlin?
Answer
No. val only guarantees that the reference itself is read-only (you cannot reassign the variable). However, if the object referred to is mutable (such as a MutableList), its internal state can still be modified. For true immutability, use read-only collections and immutable properties.

2. Class Hierarchies, Objects & Interfaces OOP

Concepts & Explanations

  • Classes & Inheritance: Kotlin classes are closed (final) by default. You must explicitly mark a class and its open methods as open to support inheritance.
  • Data Classes: Automatically generate boilerplate methods: equals(), hashCode(), toString(), copy(), and componentN() for destructuring.
  • Sealed Classes: Define restricted class hierarchies. Unlike standard classes, all subclasses of a sealed class must be declared in the same file/package, ensuring exhaustive when expressions.
  • Objects & Companions: Declaring object creates a thread-safe Singleton instance dynamically. companion object defines static-like scopes within classes.
  • Interfaces: Can contain abstract declarations as well as concrete default method implementations (unlike Java 7 and below).

Code Example

// Open class for inheritance & Interfaces
interface Loggable {
    fun logInfo() = println("Logging interface default impl")
}

open class Vehicle(val wheels: Int) : Loggable

class Car(wheels: Int, val model: String) : Vehicle(wheels) {
    override fun logInfo() = println("Car model: $model, wheels: $wheels")
}

// Data Class
data class HotelRoom(val id: String, val price: Double)

// Sealed Class
sealed class NetworkResult {
    data class Success(val data: String) : NetworkResult()
    data class Failure(val error: Throwable) : NetworkResult()
    object Loading : NetworkResult()
}

// Singleton and Companion Objects
object DatabaseManager {
    fun connect() = println("Connected to Database")
}

class ApiService {
    companion object {
        const val BASE_URL = "https://api.expedia.com"
        fun create(): ApiService = ApiService()
    }
}
Why are sealed classes preferred over enums for representing state?
Answer
Enums are limited because every enum constant is restricted to the same constructor parameters and must be a singleton instance. Subclasses of a sealed class can have their own custom constructor parameters, hold distinct state data (e.g. one holds a loading boolean, another holds a payload object), and exist as multiple separate instances.

3. Collections & ArrayList/HashMap/LinkedHashMap Internals Collections

Concepts & Explanations

  • ArrayList Internals: Backed by a standard dynamic JVM array. When elements are added and capacity is exceeded, ArrayList creates a new array 1.5x larger and copies the old array items over.
  • HashMap Internals: Utilizes an array of nodes (buckets). Maps keys to indices using hashCode() and resolving index conflicts via linked lists. If conflicts in a single bucket grow beyond 8 nodes and total size is >= 64, the list converts into a balanced Red-Black tree (reducing worst-case search complexity from O(N) to O(log N)).
  • LinkedHashMap: Maintains a doubly-linked list running through all of its nodes. This maintains a predictable iteration order matching insertion or access order.
  • Mutable vs Immutable Collections: Kotlin divides interfaces into read-only (List) and mutable (MutableList). However, at runtime on the JVM, both compile to standard Java collections (e.g. java.util.ArrayList) which are fully mutable. Kotlin only enforces read-only access at compile-time.

Code Example

// Mutable vs Immutable references
val readOnlyList: List = listOf("Delta", "United") // Read-only interface
val mutableList: MutableList = mutableListOf("Delta", "United") // Mutable interface

// Modifying under the hood
fun modifyCollections() {
    mutableList.add("AirAsia") // Okay
    
    // Casting to access mutability (Java interoperability gap)
    if (readOnlyList is java.util.ArrayList) {
        (readOnlyList as java.util.ArrayList).add("Crashed!") // Compiles and modifies!
    }
}

// HashMap & LinkedHashMap
val hashStore = HashMap<String, Int>() // Unordered entries
val insertionOrderedStore = LinkedHashMap<String, Int>() // Preserves insertion order
What happens when two keys hash to the same bucket in a HashMap?
Answer
It creates a collision. HashMap stores collision entries in the same array slot as a singly-linked list of nodes. During search, it walks the list checking equals(). If the list size exceeds 8 (and total HashMap size is at least 64), the linked list is treeified into a balanced Red-Black Tree, optimizing lookup to O(log N).

4. Functions, Lambdas & Inline/Crossinline/Noinline/Reified Types Functional

Concepts & Explanations

  • Extension Functions: Declared outside class definitions. They are resolved statically at compile-time as static utility calls, receiving the receiver class object as the first argument.
  • Higher-Order Functions & Lambdas: Functions that accept other functions as parameters or return them. Lambdas allocate an instance of the FunctionN interface on the JVM heap, creating memory pressure if called inside loops.
  • Inline Functions: Tells the compiler to copy the function body and lambda code directly into the calling site, bypassing method invocation and class allocation overhead.
  • Noinline / Crossinline: Use noinline on parameter lambdas to keep them as objects (e.g. to store them in variables). Use crossinline to prevent lambdas from executing non-local returns (exiting the calling scope).
  • Reified Types: Access generic type parameters at runtime inside inline functions using inline fun , bypassing Java's type erasure.

Code Example

// Extension Function
fun String.shout(): String = this.uppercase() + "!"

// Inline, Noinline, and Crossinline
inline fun executeTasks(
    crossinline onComplete: () -> Unit,
    noinline storeAction: () -> Unit
) {
    println("Beginning task")
    val runnable = Runnable { 
        // crossinline allows this call inside nested local scopes/runnables
        onComplete() 
    }
    runnable.run()
    
    // noinline allows this lambda to be treated as an object and stored
    val stored = storeAction 
    stored()
}

// Reified Type Parameters
inline fun <reified T> checkType(value: Any): Boolean {
    // Reified parameter allows 'is' check directly at runtime
    return value is T 
}
Why is reified only available inside inline functions?
Answer
Normally, the JVM performs type erasure, deleting generic type parameters (like <T>) during compilation. Because inline functions copy-paste the function body directly to the call site, the compiler knows the exact class type passed at that specific location. It replaces the generic checks directly with concrete class bytecode checks (e.g., instanceof String).

5. Class/Property Delegation & Generics (Variance) Advanced

Concepts & Explanations

  • Delegation: Delegates implementation to another object. Class delegation (class Car : Vehicle by electricEngine) maps interface methods dynamically. Property delegation (var name by delegate) intercepts getter and setter accesses.
  • lazy delegate: Thread-safe modes:
    • SYNCHRONIZED: Monitor locks the block to ensure single-thread execution (default).
    • PUBLICATION: Multiple threads can execute simultaneously, but only the first returned value is stored.
    • NONE: No lock check. Fast, but unsafe in multi-threaded environments.
  • Generics & Variance:
    • out T (Covariance): Producer context. Safely reads supertype references; blocks write operations (subtypes assignable to supertypes).
    • in T (Contravariance): Consumer context. Safely writes subtype references; blocks read operations (supertypes assignable to subtypes).

Code Example

// Class Delegation
interface Engine { fun start() }
class ElectricEngine : Engine { override fun start() = println("Electric Hum") }
class ElectricCar(engine: Engine) : Engine by engine // Delegates 'start()' to engine

// Property Delegation & Lazy
val dbConnection: String by lazy(LazyThreadSafetyMode.SYNCHRONIZED) {
    println("Initializing Heavy DB...")
    "MySQLConnection"
}

// Generics Variance
class Producer<out T>(private val value: T) {
    fun get(): T = value // Allowed: producing T
    // fun set(item: T) {} // Error: Covariant type T cannot be in 'in' position
}

class Consumer<in T> {
    fun consume(item: T) = println("Consumed: $item") // Allowed: consuming T
    // fun retrieve(): T = TODO() // Error: Contravariant type T cannot be in 'out' position
}
Explain why 'out T' cannot be used as an input parameter in class methods.
Answer
If a class is covariant (out T), it allows assigning subtype instances to supertype variables (e.g. assigning Producer<Dog> to Producer<Animal>). If Producer<Animal> allowed input parameters of type Animal, you could pass a Cat into a class that internally expects a Dog, causing a crash. restircting inputs ensures type safety.

6. Destructuring & Scope Functions (let, run, apply, also, with) Syntactic Sugar

Concepts & Explanations

  • Destructuring: Unpacks classes into variables. The compiler maps destructuring variables directly to the compiled component1(), component2() methods generated in data classes.
  • Scope Functions Reference Matrix:
    Function Object Reference Return Value
    let it Lambda result
    run this Lambda result
    with this Lambda result
    apply this Context object
    also it Context object

Code Example

// Destructuring Example
data class User(val name: String, val age: Int)

fun testDestructuring() {
    val (username, userAge) = User("Arun", 28) // unpacks variables
    println("$username is $userAge years old")
}

// Scope Functions usages
class Hotel {
    var name: String = ""
    var rating: Int = 0
}

fun testScopeFunctions() {
    val hotel = Hotel().apply {
        this.name = "Expedia Ritz" // 'this' reference: configure parameters
        rating = 5
    } // returns configured 'Hotel' object

    val info: String = hotel.run {
        "Hotel: $name, Rating: $rating" // 'this' reference: computes and returns String
    }

    hotel.let {
        println("Hotel name length: ${it.name.length}") // 'it' reference: runs block
    }

    val loggedHotel = hotel.also {
        println("Logging hotel configuration: ${it.name}") // 'it' reference: returns original hotel
    }
}
When should you use let vs run?
Answer
Use let when performing operations on a nullable object safely (e.g. nullableObj?.let { ... }) and accessing it via the it parameter, preserving outside scopes. Use run when configuring properties of an object and computing a result using its implicit this context.
